# Проектирование баз данных

## 1. Введение в СУБД и файловые системы

### 1. Перечислите и опишите основные проблемы хранения данных в плоских файлах (flat files) – подход на основе файлов. #flat-files #basics
- [x] Избыточность и дублирование данных в разных файлах, приводящие к неэффективному использованию дисковой памяти.
- [x] Зависимость данных от прикладных программ (структура файлов жестко зафиксирована в коде, и любое её изменение требует перекомпиляции приложений).
- [ ] Ограничение на максимальный размер файла в рамках файловой системы, из-за которого невозможно хранить более 1000 записей.
- [ ] Несовместимость кодировок символов при передаче текстовых файлов по сети Интернет между различными ОС.
- [ ] Отсутствие поддержки графических интерфейсов для просмотра текстовых и бинарных файлов без специализированного ПО.
- [ ] Необходимость постоянного дефрагментирования дискового пространства силами системного администратора.
- [ ] Фиксированное ограничение на количество колонок в плоском файле, которое не может превышать 256.
- [ ] Высокая уязвимость файлов к воздействию вредоносного программного обеспечения из-за отсутствия встроенных антивирусных модулей.
- [ ] Невозможность архивации или резервного копирования данных без полной остановки работы всей операционной системы.
- [ ] Сложность интеграции плоских файлов с веб-страницами из-за отсутствия встроенной поддержки протокола HTTP.

* **Краткий ответ**: Основные проблемы файлового подхода: избыточность (дублирование) данных, несогласованность (неконсистентность), жесткая зависимость структуры данных от кода программ, плохие возможности совместного доступа и слабая безопасность.
* **Пояснение**: При файловом подходе к хранению информации (плоские файлы, flat files) данные распределены по файлам различных отделов организации. Это приводит к следующим фундаментальным проблемам:
  1. *Избыточность данных (Data Redundancy)*: Одно и то же поле (например, адрес клиента) хранится во многих файлах, что тратит память и усложняет обновление.
  2. *Несогласованность данных (Data Inconsistency)*: Из-за избыточности при изменении адреса в одном месте он может остаться старым в другом.
  3. *Зависимость от программ (Program-Data Dependence)*: Логическая и физическая структура данных зашита в код приложений. Если изменить длину поля в файле, придется переписать и перекомпилировать все программы, работающие с ним.
  4. *Несовместимость форматов*: Разные программы могут писать данные в разных форматах, что затрудняет их объединение.
  5. *Сложность совместного доступа*: Файловые системы обычно блокируют файл целиком для одного пользователя, делая невозможным одновременное редактирование записей разными сотрудниками.
  6. *Слабая защита и разграничение прав*: Сложно настроить доступ к конкретным полям внутри файла для разных ролей пользователей.

## 2. Классификация СУБД и моделей данных

### 2. Системы классификации современных БД (реляционные БД, документированные БД, колоночные БД, распределенные БД, хранилища данных). Опишите плюсы и минусы реляционных БД. #db-classification #relational
- [x] Достоинством реляционных БД является строгое обеспечение транзакционных свойств ACID и высокая согласованность данных при частых обновлениях.
- [x] Недостатком реляционных БД является плохая горизонтальная масштабируемость «из коробки» и падение производительности при работе со слабоструктурированными данными.
- [ ] Реляционные БД не поддерживают использование индексов, из-за чего поиск по текстовым полям всегда выполняется полным сканированием таблицы.
- [ ] Документированные БД хранят данные исключительно в виде плоских XML-файлов и не поддерживают сериализацию в формат JSON.
- [ ] Колоночные БД оптимизированы для обработки транзакций в реальном времени (OLTP) и не подходят для аналитических запросов (OLAP).
- [ ] Распределенные БД требуют физического размещения всех серверов в одном дата-центре для работы протоколов консенсуса.
- [ ] Хранилища данных (DWH) предназначены исключительно для хранения неструктурированных медиафайлов (картинок и видео).
- [ ] Одним из главных минусов реляционных БД является полное отсутствие поддержки SQL как стандарта для выполнения запросов.
- [ ] Реляционные базы данных не позволяют связывать таблицы между собой, что требует дублирования всех связанных сущностей в каждой строке.
- [ ] Документированные базы данных хранят данные в виде двумерных таблиц с обязательным объявлением схемы данных (schema-on-write).

* **Краткий ответ**: Современные БД делятся на реляционные (SQL) и нереляционные (NoSQL: документированные, колоночные, графовые, key-value), распределенные БД и хранилища данных (DWH). Плюсы реляционных БД: ACID, стандартизация (SQL), целостность данных. Минусы: плохая горизонтальная масштабируемость, жесткая схема (schema-on-write).
* **Пояснение**: Классификация баз данных по модели представления и назначению:
  - *Реляционные БД* (PostgreSQL, MySQL, Oracle): Данные хранятся в связанных таблицах. Строгая схема.
  - *Документированные (Document-oriented) БД* (MongoDB, CouchDB): Данные хранятся в виде гибких документов (JSON/BSON). Schema-less.
  - *Колоночные (Columnar) БД* (ClickHouse, Cassandra): Хранят данные по колонкам, а не по строкам. Оптимальны для чтения больших объемов аналитических данных (OLAP).
  - *Распределенные БД*: Системы, физически размещенные на нескольких узлах сети, обеспечивающие отказоустойчивость и масштабируемость.
  - *Хранилища данных (DWH)*: Специализированные БД для интеграции исторических данных из разных источников и аналитики.
  
  **Плюсы реляционных БД**:
  1. *Надежность (ACID)*: Атомарность, Согласованность, Изолированность, Долговечность транзакций.
  2. *Стандарт SQL*: Единый мощный язык декларативных запросов.
  3. *Целостность данных*: Поддержка ограничений (NOT NULL, UNIQUE, FOREIGN KEY).
  
  **Минусы реляционных БД**:
  1. *Сложность масштабирования*: Горизонтальное масштабирование (шардирование) реляционных БД затруднено из-за необходимости поддерживать связи и транзакции между серверами.
  2. *Негибкость схемы*: Изменение структуры таблицы на терабайтных базах — сложная операция.
  3. *Избыточные накладные расходы* на соединения (JOIN) при сложных иерархических структурах данных.

### 3. Данные и модели данных. Классификация моделей данных. Опишите модели данных. #data-models #classification
- [x] Логические (на основе записей) модели данных включают иерархическую, сетевую и реляционную модели.
- [x] Объектно-ориентированная модель данных интегрирует концепции объектного программирования (классы, наследование) непосредственно в структуру базы данных.
- [ ] Сетевая модель данных допускает наличие у дочернего узла только одного родительского узла, формируя строгое древовидное представление.
- [ ] Реляционная модель данных представляет информацию в виде ориентированного графа, где дуги обозначают физические указатели в памяти.
- [ ] Концептуальная модель данных описывает физическую структуру файлов данных, индексов и распределение страниц на жестком диске.
- [ ] Иерархическая модель данных является наиболее современной и полностью вытеснила реляционные базы данных из коммерческого сектора.
- [ ] Модели данных классифицируются исключительно по количеству одновременно подключающихся пользователей к серверу СУБД.
- [ ] В реляционной модели данных основным элементом является дерево отношений, ветви которого описывают связи между сущностями.
- [ ] Модель "сущность-связь" (ER-модель) используется только на стадии физической реализации базы данных в СУБД.
- [ ] Документная модель данных полностью исключает возможность хранения текстовой информации, допуская только числовые массивы.

* **Краткий ответ**: Модель данных — это интегрированная коллекция понятий для описания данных, связей между ними и ограничений. Классифицируются на объектные (ER-модель, объектно-ориентированная), логические на основе записей (иерархическая, сетевая, реляционная) и физические модели.
* **Пояснение**: Модели данных служат для абстракции реального мира. Они делятся на три уровня абстракции:
  1. *Концептуальные модели* (например, ER-модель «Сущность-Связь»): Описывают предметную область на языке понятий пользователей, без привязки к конкретной СУБД.
  2. *Логические модели (на основе записей)*:
     - *Иерархическая*: Данные организованы в виде дерева (родитель-потомок). У потомка ровно один предок (отношение 1:M). Пример: файловая система, XML.
     - *Сетевая*: Расширение иерархической модели, где потомок может иметь несколько предков (отношение M:M). Представляется в виде ориентированного графа.
     - *Реляционная*: Данные организованы в виде двумерных таблиц (отношений). Нет явных физических связей-указателей, связи строятся на равенстве значений полей.
     - *Объектно-ориентированная*: Данные представляются в виде объектов с их методами и свойствами.
  3. *Физические модели*: Описывают способы размещения данных в памяти компьютера (индексы, методы доступа, организация файлов).

## 3. Архитектура СУБД

### 4. Перечислите и опишите основные архитектурные компоненты (модули) СУБД. #dbms-architecture #components
- [x] Процессор запросов (включая парсер, оптимизатор и генератор кода), отвечающий за трансляцию и оптимизацию декларативных SQL-инструкций.
- [x] Менеджер транзакций и менеджер буферов, управляющие параллельным доступом, журналированием изменений и кэшированием данных в ОЗУ.
- [ ] Драйвер видеокарты, обеспечивающий аппаратное ускорение рендеринга таблиц и графиков на мониторе пользователя.
- [ ] Модуль проверки орфографии текстовых полей, предотвращающий грамматические ошибки при вставке строковых значений.
- [ ] Планировщик задач операционной системы, отвечающий за запуск СУБД при старте компьютера.
- [ ] Компилятор C++, встроенный в СУБД для преобразования SQL-запросов в исполняемые `.exe` файлы.
- [ ] Модуль интеграции с почтовым сервером для автоматической рассылки спама при изменении записей.
- [ ] Менеджер периферийных устройств, распределяющий права доступа к принтерам и сканерам организации.
- [ ] Сетевой шлюз безопасности, блокирующий доступ к базе данных по протоколам FTP и SMTP.
- [ ] Модуль архивации ZIP, сжимающий все неактивные таблицы СУБД в фоновом режиме работы.

* **Краткий ответ**: Архитектурные компоненты СУБД: процессор запросов (парсер, оптимизатор, исполнитель), менеджер данных на диске, менеджер буферов, менеджер транзакций, менеджер блокировок (параллельного доступа) и менеджер журналов (WAL).
* **Пояснение**: СУБД — сложная программная система. Ее архитектура состоит из следующих ключевых модулей:
  1. *Компоненты обработки запросов (Query Processor)*:
     - **Парсер (Анализатор)**: Проверяет синтаксис SQL-запроса и переводит его во внутреннее представление (дерево разбора).
     - **Оптимизатор (Optimizer)**: Анализирует статистику данных и строит наиболее эффективный план выполнения запроса (выбор индексов, порядка соединений таблиц).
     - **Исполнитель (Executor)**: Выполняет шаги выбранного плана, обращаясь к уровню хранения.
  2. *Компоненты хранения и буферизации*:
     - **Менеджер буферного пула (Buffer Manager)**: Управляет оперативной памятью СУБД, считывает страницы данных с диска в ОЗУ и вытесняет их обратно.
     - **Менеджер файлов данных (Storage Manager)**: Отвечает за выделение места на диске и чтение/запись низкоуровневых блоков данных.
  3. *Компоненты надежности и конкурентности*:
     - **Менеджер транзакций (Transaction Manager)**: Гарантирует выполнение принципа ACID.
     - **Менеджер блокировок (Lock Manager)**: Синхронизирует параллельные транзакции, предотвращая конфликты при изменении одних и тех же строк.
     - **Менеджер журнала (Recovery/Log Manager)**: Пишет логи транзакций (Write-Ahead Log - WAL) для восстановления БД после сбоев.

## 4. Реляционная модель данных и её основы

### 5. Дайте определение реляционной модели данных. Раскройте центральное понятие этой модели — «отношение» — и перечислите его ключевые свойства. #relational-model #relations
- [x] Реляционная модель данных — это модель, представляющая данные в виде совокупности взаимосвязанных двумерных таблиц, называемых отношениями.
- [x] Отношение представляет собой подмножество декартова произведения множества доменов, обладающее свойством отсутствия дубликатов кортежей.
- [ ] Отношение — это древовидный объект, где узлы содержат физические адреса связанных файлов на жестком диске.
- [ ] Одним из ключевых свойств отношений является строгий физический порядок строк, определяющий скорость работы СУБД.
- [ ] Отношения реляционной модели данных допускают наличие полностью идентичных (дублирующихся) строк без ограничений.
- [ ] Отношение состоит из неограниченного числа колонок, типы данных которых могут динамически меняться от строки к строке.
- [ ] Реляционная модель данных основывается исключительно на теории графов и сетевых топологиях.
- [ ] Под отношением понимается связь "один-ко-многим", реализованная на уровне концептуального проектирования.
- [ ] Свойства отношений требуют, чтобы имена столбцов обязательно совпадали во всех таблицах базы данных.
- [ ] В отношении допускается хранение вложенных таблиц и массивов в качестве значений атрибутов в любой ячейке.

* **Краткий ответ**: Реляционная модель данных представляет информацию в виде набора отношений (таблиц). Отношение — это математическое подмножество декартова произведения доменов. Его свойства: уникальность кортежей, отсутствие порядка кортежей и атрибутов, атомарность значений атрибутов.
* **Пояснение**: Реляционная модель была предложена Эдгаром Ф. Коддом в 1970 году и базируется на теории множеств и логике предикатов.
  
  **Центральное понятие: Отношение (Relation)** — это двумерная таблица, состоящая из заголовка (схемы) и тела (множества кортежей). Формально, если заданы домены $D_1, D_2, ..., D_n$, то отношение $R$ есть подмножество их декартова произведения: $R \subseteq D_1 \times D_2 \times ... \times D_n$.
  
  **Ключевые свойства отношений**:
  1. *Отсутствие кортежей-дубликатов*: Поскольку тело отношения является математическим множеством, в нем не может быть двух одинаковых элементов (кортежей).
  2. *Неупорядоченность кортежей (строк)*: Порядок строк в отношении не имеет значения.
  3. *Неупорядоченность атрибутов (столбцов)*: Порядок столбцов не важен, так как каждый столбец идентифицируется уникальным в рамках отношения именем.
  4. *Атомарность значений атрибутов*: Все значения на пересечении строк и столбцов должны быть простыми (атомарными). В ячейке не может содержаться список, массив или другое отношение (это требование 1-й нормальной формы).

## 5. Реляционная алгебра и исчисление

### 6. Реляционная алгебра. Перечислить и описать предложенные Э.Ф. Коддом 8 операций над отношениями. #relational-algebra #codd-operations
- [x] Традиционные теоретико-множественные операции включают объединение, пересечение, разность отношений и декартово произведение.
- [x] Специальные реляционные операции включают выборку (фильтрацию строк), проекцию (выбор колонок), соединение и деление отношений.
- [ ] Восемь операций Кодда включают шифрование, архивацию, сортировку, группировку, индексацию, бэкап, трансляцию и сжатие.
- [ ] Операция проекции оставляет только уникальные строки, удаляя все непересекающиеся столбцы в одной выбранной таблице.
- [ ] Декартово произведение объединяет только строки с совпадающими значениями внешних ключей в двух исходных таблицах.
- [ ] Операция деления реляционной алгебры делит числовые значения одной колонки на значения другой колонки.
- [ ] Объединение возможно для любых двух отношений, независимо от их структуры, схемы и типов данных.
- [ ] Специальные операции реляционной алгебры выполняются исключительно на физическом уровне хранения данных в ОЗУ.
- [ ] Операции реляционной алгебры изменяют схемы исходных отношений непосредственно на жестком диске.
- [ ] Соединение (JOIN) является базовой теоретико-множественной операцией и не может быть выражено через другие операции алгебры.

* **Краткий ответ**: Э. Кодд предложил 8 операций, разделенных на две группы: теоретико-множественные (объединение, пересечение, разность, декартово произведение) и специальные реляционные (выборка/ограничение, проекция, соединение, деление).
* **Пояснение**: Реляционная алгебра — это процедурный язык запросов. Операции принимают одно или два отношения на вход и возвращают новое отношение на выходе.
  
  **1. Традиционные теоретико-множественные операции**:
  - *Объединение (Union, $\cup$)*: Строит отношение, содержащее все кортежи, которые есть хотя бы в одном из исходных отношений. Требует совпадения схем.
  - *Пересечение (Intersection, $\cap$)*: Возвращает кортежи, присутствующие одновременно в обоих отношениях.
  - *Разность (Difference, $-$)*: Возвращает кортежи, которые есть в первом отношении, но отсутствуют во втором.
  - *Декартово произведение (Cartesian Product, $\times$)*: Объединяет каждый кортеж первого отношения с каждым кортежем второго отношения.
  
  **2. Специальные реляционные операции**:
  - *Выборка / Ограничение (Selection, $\sigma$)*: Выбирает строки из отношения, удовлетворяющие заданному условию.
  - *Проекция (Projection, $\pi$)*: Выбирает указанные столбцы из отношения с удалением дубликатов строк.
  - *Соединение (Join, $\bowtie$)**: Связывает кортежи двух отношений по общему условию (равенству общих атрибутов).
  - *Деление (Division, $\div$)**: Возвращает кортежи отношения $A(X,Y)$, которые связаны со всеми кортежами отношения $B(Y)$.

## 6. Язык SQL и типы данных

### 7. Опишите операторы манипулирования данными и операторы языка управления данными. #sql #dml #dcl
- [x] К операторам манипулирования данными (DML) относятся SELECT, INSERT, UPDATE и DELETE, используемые для работы с содержимым таблиц.
- [x] К операторам управления данными (DCL) относятся GRANT и REVOKE, управляющие правами доступа пользователей к объектам базы данных.

- [ ] Оператор INSERT позволяет изменять структуру столбцов таблицы, добавляя новые свойства сущностей на лету.
- [ ] К операторам управления данными (DCL) относятся CREATE USER и DROP USER для полного удаления учетных записей.
- [ ] Оператор DELETE используется для изменения типов данных существующих столбцов в физической схеме базы.
- [ ] Оператор REVOKE используется для отката изменений незавершенных транзакций в рамках текущей сессии.
- [ ] Все операторы DML автоматически фиксируют транзакции, не допуская возможности отката с помощью ROLLBACK.

* **Краткий ответ**: Операторы манипулирования данными (DML) — `SELECT`, `INSERT`, `UPDATE`, `DELETE` — служат для работы с данными. Операторы управления данными (DCL) — `GRANT`, `REVOKE` — служат для назначения и отзыва прав пользователей на объекты БД.
* **Пояснение**: Язык SQL функционально разделен на несколько подязыков:
  
  **1. DML (Data Manipulation Language) — Язык манипулирования данными**:
  Предназначен для работы с кортежами (записями) в таблицах.
  - `SELECT`: Извлечение (выборка) данных из одной или нескольких таблиц.
  - `INSERT`: Вставка новых записей в таблицу.
  - `UPDATE`: Модификация существующих записей по заданному критерию.
  - `DELETE`: Удаление записей из таблицы.
  
  **2. DCL (Data Control Language) — Язык управления данными**:
  Используется для управления привилегиями доступа к таблицам, представлениям и другим объектам.
  - `GRANT`: Предоставление пользователю или роли определенных прав (например, права на выполнение `SELECT` или `UPDATE` для конкретной таблицы).
  - `REVOKE`: Отзыв ранее предоставленных прав доступа.

### 8. Опишите операторы языка определения данных. #sql #ddl
- [x] К операторам определения данных (DDL) относятся CREATE, ALTER и DROP, служащие для создания, изменения структуры и удаления объектов БД.
- [x] Оператор TRUNCATE относится к категории DDL, так как быстро очищает таблицу путем пересоздания её внутренней структуры на диске.
- [ ] Оператор UPDATE является ключевым элементом DDL, так как обновляет логическое определение данных в схеме.
- [ ] К операторам DDL относится оператор GRANT, описывающий структуру привилегий пользователей в системном каталоге.
- [ ] Операторы DDL используются для непосредственной вставки новых кортежей в физические таблицы базы данных.
- [ ] Оператор DROP используется для временного отключения индексов без их физического удаления с жесткого диска.
- [ ] Любой оператор DDL требует обязательного вызова COMMIT для подтверждения создания таблиц во всех СУБД.
- [ ] Оператор ALTER позволяет модифицировать значения отдельных ячеек в строках таблицы по заданному первичному ключу.
- [ ] Операторы DDL выполняются исключительно при проектировании концептуальной модели данных в виде ER-диаграммы.
- [ ] Оператор CREATE TABLE автоматически заполняет создаваемую таблицу тестовыми демонстрационными данными.

* **Краткий ответ**: DDL (Data Definition Language) — это семейство операторов SQL, используемых для определения, изменения и удаления структуры объектов базы данных (таблиц, индексов, представлений, триггеров). К ним относятся `CREATE`, `ALTER`, `DROP` и `TRUNCATE`.
* **Пояснение**: DDL-операторы оперируют метаданными в системном каталоге СУБД:
  - `CREATE`: Создание новых объектов базы данных (например, `CREATE TABLE`, `CREATE INDEX`).
  - `ALTER`: Изменение структуры существующего объекта (добавление/удаление столбцов, изменение типов, добавление ограничений).
  - `DROP`: Полное физическое удаление объекта и его структуры из базы данных (например, `DROP TABLE Students`).
  - `TRUNCATE`: Быстрое удаление всех данных из таблицы без удаления самой структуры. Освобождает физические страницы данных на уровне файловой системы, минуя построчное удаление и логирование каждой строки.

## 7. Проектирование баз данных: методология

### 9. Метод проектирования базы данных «сверху-вниз». #db-design #top-down
- [x] Метод проектирования «сверху-вниз» (top-down) начинается с концептуального моделирования предметной области и выделения сущностей и связей.
- [x] Данный подход ориентирован на глубокий анализ бизнес-процессов организации и требований пользователей перед переходом к конкретным таблицам.

* **Пояснение**: Проектирование «сверху-вниз» — это классический дедуктивный метод, идущий от общего к частному:
  1. *Анализ требований и концептуальное проектирование*: Архитектор общается с заказчиком, выявляет сущности (Entity), их атрибуты и связи (Relationship) между ними. Строится ER-диаграмма.
  2. *Логическое проектирование*: Концептуальная модель транслируется в реляционную модель. Сущности становятся таблицами, атрибуты — столбцами, связи реализуются через первичные и внешние ключи. Выполняется нормализация.
  3. *Физическое проектирование*: Логическая модель переносится в конкретную СУБД (например, PostgreSQL), где выбираются типы данных, настраиваются индексы, партиционирование и параметры хранения.

### 10. Опишите характерную особенность сущности сильного и слабого типов. Приведите примеры таких сущностей. #er-model #entities #strong-weak
- [x] Сильная сущность существует независимо от других сущностей и обладает собственным первичным ключом для уникальной идентификации кортежей.
- [x] Слабая (зависимая) сущность не может существовать без родительской сущности, и ее первичный ключ частично или полностью формируется из внешнего ключа родителя.
- [ ] Сильная сущность хранит только числовые типы данных, а слабая сущность может хранить исключительно текстовые значения.
- [ ] Примером слабой сущности является таблица "Сотрудники", которая полностью контролирует жизненный цикл таблицы "Департаменты".
- [ ] Слабая сущность всегда имеет больший объем физической памяти на диске, чем сильная сущность.
- [ ] Сильная сущность не может иметь связей типа "один-ко-многим" с другими таблицами в схеме базы данных.
- [ ] Для идентификации строк слабой сущности никогда не используются атрибуты других сущностей.
- [ ] Связь между сильной и слабой сущностями в нотации Чена отображается в виде двойного круга с пунктирной линией.
- [ ] Слабая сущность может существовать в базе данных даже после удаления записи о соответствующей ей сильной сущности.
- [ ] Первичный ключ сильной сущности всегда состоит не менее чем из трех составных атрибутов.

* **Краткий ответ**: Сильная (независимая) сущность существует самостоятельно и имеет собственный уникальный первичный ключ. Слабая (зависимая) сущность не может существовать без связанной с ней родительской (сильной) сущности; ее ключ является составным и включает в себя первичный ключ родителя.
* **Пояснение**:
  - **Сильная сущность (Strong Entity)**:
    * *Особенность*: Существование сущности не зависит от существования каких-либо других сущностей в БД. Каждый экземпляр можно уникально идентифицировать с помощью собственного набора атрибутов (первичного ключа).
    * *Пример*: Сущность `Покупатель` (Customer) с первичным ключом `ID_Покупателя`.
  - **Слабая сущность (Weak Entity)**:
    * *Особенность*: Экземпляр слабой сущности не может существовать без соответствующего экземпляра родительской сущности. При удалении родительской записи все зависимые записи слабой сущности также должны быть удалены (каскадное удаление).
    * *Пример*: Сущность `Родственники_Сотрудника` (Dependents) зависит от сущности `Сотрудник` (Employee). У нее нет уникального глобального ключа. Ее первичный ключ является составным: (`ID_Сотрудника`, `Имя_Родственника`), где `ID_Сотрудника` — это внешний ключ, ссылающийся на родителя.

### 11. Опишите последовательность шагов концептуального и логического проектирования базы данных. #db-design #conceptual #logical
- [x] Концептуальное проектирование включает определение сущностей, их связей, атрибутов и доменов на основе анализа требований пользователей.
- [x] Логическое проектирование включает преобразование концептуальной модели в реляционную схему, ее нормализацию и валидацию ограничений целостности.

- [ ] Концептуальное проектирование полностью игнорирует требования пользователей, основываясь на технических ограничениях оборудования.

- [ ] Во время концептуального проектирования все связи "многие-ко-многим" сразу заменяются промежуточными таблицами в БД.

* **Краткий ответ**: 
  1. *Концептуальное проектирование*: Сбор требований -> Определение сущностей -> Определение связей -> Определение атрибутов и их доменов -> Построение концептуальной ER-модели.
  2. *Логическое проектирование*: Преобразование ER-модели в реляционные таблицы -> Устранение связей M:M -> Нормализация отношений (до 3NF/BCNF) -> Проверка ограничений целостности -> Валидация логической модели по требованиям пользователей.
* **Пояснение**: Процесс проектирования базы данных разделен на четкие методологические шаги:
  **Шаги концептуального проектирования (не зависит от СУБД)**:
  1. *Выявление сущностей*: Определение основных объектов предметной области (например, *Товар*, *Склад*).
  2. *Определение связей*: Установление ассоциаций между сущностями (*Товар* хранится на *Складе*).
  3. *Связывание атрибутов*: Определение характеристик сущностей (*Название*, *Артикул*).
  4. *Определение доменов атрибутов*: Описание допустимых значений (например, цена — положительное вещественное число).
  5. *Построение ER-диаграммы*: Визуализация концептуальной схемы.
  **Шаги логического проектирования (ориентировано на реляционную модель, но независимо от конкретной СУБД)**:
  1. *Преобразование сущностей в таблицы*: Каждая сущность становится схемой отношения.
  2. *Разрешение связей*: Перенос первичных ключей в качестве внешних для связей 1:M; создание промежуточных таблиц для связей M:M.
  3. *Нормализация*: Проверка отношений на соответствие нормальным формам (1NF, 2NF, 3NF) для удаления избыточности.
  4. *Определение ограничений*: Задание правил уникальности, обязательности (NOT NULL) и ссылочной целостности.

### 12. Опишите последовательность шагов логического и физического проектирования базы данных. #db-design #logical #physical

  - **Физическое проектирование** переводит логическую модель в конкретную техническую среду. Шаги:
    1. *Проектирование таблиц и типов данных*: Перевод абстрактных логических типов (строка, число) в типы данных выбранной СУБД (например, `VARCHAR`, `NUMERIC`, `UUID` в PostgreSQL).
    2. *Определение физической организации*: Настройка табличных пространств (tablespaces), определение стратегии секционирования (партиционирования) для очень больших таблиц.
    3. *Проектирование индексов*: Создание индексов (`B-Tree`, `Hash`, `GIN`, `GiST`) для ускорения частых запросов.
    4. *Реализация бизнес-правил*: Написание ограничений (`CHECK`), триггеров, хранимых процедур и функций на диалектах SQL (PL/pgSQL, PL/SQL).
    5. *Безопасность*: Настройка учетных записей, ролей, прав доступа и правил шифрования данных.

## 8. Качество схемы БД: избыточность и аномалии

### 13. Избыточность данных. Объясните термин, приведите пример. #redundancy #database-design
- [x] Избыточность данных — это многократное дублирование одной и той же информации в рамках базы данных, приводящее к непроизводительному расходу дисковой памяти.
- [x] Примером избыточности является хранение ФИО и телефона сотрудника в каждой строке таблицы «Продажи» вместо ссылки на таблицу «Сотрудники».
- [ ] Избыточность данных — это ситуация, когда размер базы данных превышает физический объем оперативной памяти сервера.
- [ ] Избыточность является главным требованием третьей нормальной формы для обеспечения быстродействия транзакций.
- [ ] Хранение уникальных идентификаторов первичных ключей в таблице является примером вредной избыточности данных.
- [ ] Избыточность данных полностью исключает возможность появления аномалий обновления и удаления в таблицах.
- [ ] Примером избыточности является использование индексов для ускорения поиска записей в базе данных.
- [ ] Избыточность данных возникает только при использовании СУБД PostgreSQL и отсутствует в коммерческих базах Oracle.
- [ ] Избыточностью называют хранение истории изменений данных в специальном журнале транзакций (WAL).
- [ ] Избыточность данных предотвращает сбои жестких дисков благодаря дублированию секторов на аппаратном уровне.

* **Краткий ответ**: Избыточность данных (Data Redundancy) — это повторное (дублирующее) хранение одних и тех же данных в базе. *Пример*: Хранение названия и адреса филиала компании в каждой строке таблицы «Сотрудники» (если 100 сотрудников работают в одном филиале, адрес филиала продублирован 100 раз).
* **Пояснение**: Избыточность возникает из-за плохого проектирования схемы БД (когда сущности с разным жизненным циклом объединены в одну таблицу).
  **Проблемы, вызываемые избыточностью**:
  1. *Нерациональное использование памяти*: Хранение длинных повторяющихся строк тратит ресурсы диска и оперативной памяти СУБД.
  2. *Аномалии обновления*: Если адрес филиала меняется, его нужно обновить у всех 100 сотрудников. Если обновить у 99, возникнет рассогласование данных.
  3. *Аномалии вставки*: Нельзя внести новый филиал в базу данных, пока туда не устроен хотя бы один сотрудник.
  4. *Аномалии удаления*: При увольнении последнего сотрудника из филиала сотрется и информация о самом филиале.
  *Решение*: Разнесение сущностей по разным таблицам (нормализация) и связывание их через внешние ключи (`FOREIGN KEY`).

### 14. Аномалия обновления. Объясните термин, приведите пример. #anomalies #redundancy
- [x] Аномалия обновления — это логическая противоречивость данных, возникающая при изменении дублирующегося значения не во всех местах его хранения.
- [x] Примером является ситуация, когда при смене телефона клиента запись обновляется в таблице «Договоры», но остается старой в таблице «Активные заказы».
- [ ] Аномалией обновления называют ошибку СУБД, при которой отключается питание сервера в процессе выполнения транзакции.
- [ ] Аномалия обновления возникает исключительно при попытке вставки пустых значений NULL в столбцы с ограничением NOT NULL.
- [ ] Данный термин описывает ситуацию, когда несколько пользователей одновременно пытаются прочитать одну и ту же запись из таблицы.
- [ ] Аномалия обновления полностью устраняется путем отключения внешних ключей и уникальных индексов в БД.
- [ ] Примером аномалии обновления является автоматическое резервное копирование, запущенное в рабочее время СУБД.
- [ ] Аномалия обновления возникает из-за использования транзакций с уровнем изолированности Serializable.
- [ ] Это сбой физических секторов жесткого диска, приводящий к искажению бинарных файлов базы данных.
- [ ] Аномалия обновления описывает невозможность изменить тип данных колонки с INTEGER на VARCHAR в пустой таблице.

* **Краткий ответ**: Аномалия обновления (Update Anomaly) — это несогласованность данных, которая возникает, когда избыточные данные изменяются не во всех записях таблицы, где они продублированы. *Пример*: При смене адреса поставщика адрес обновляется только в части строк поставок товара, из-за чего один и тот же поставщик в базе имеет разные адреса.
* **Пояснение**: Аномалия обновления — прямое следствие избыточности данных.
  Рассмотрим ненормализованную таблицу `Сотрудники_Проекты (Таб_Номер, ФИО, Проект, Руководитель_Проекта)`.
  Если у проекта «Альфа» меняется руководитель, СУБД должна обновить поле `Руководитель_Проекта` во всех строках сотрудников, работающих на этом проекте. Если транзакция обновит только часть строк, то в базе данных один и тот же проект одновременно будет иметь двух разных руководителей.
  *Решение*: Приведение таблицы к третьей нормальной форме (3NF). В данном случае проект и его руководитель должны быть вынесены в отдельную таблицу `Проекты (Название_Проекта, Руководитель_Проекта)`, а в таблице `Сотрудники` останется только внешний ключ на таблицу проектов. При смене руководителя проекта изменение произойдет ровно в одной строке одной таблицы, что физически исключает аномалию обновления.

## 9. Элементы реляционной модели

### 15. Опишите элементы реляционной модели БД: кортеж, атрибут, домен. Перечислите свойства отношений. #relational-model #elements
- [x] Кортеж соответствует строке отношения, атрибут — именованному столбцу, а домен определяет множество допустимых значений для данного атрибута.
- [x] Свойства отношений включают уникальность кортежей, отсутствие их упорядоченности и атомарность значений всех атрибутов.
- [ ] Домен — это имя сервера базы данных в локальной компьютерной сети организации.
- [ ] Кортеж — это специальный тип индекса, используемый для ускорения поиска по числовым столбцам отношений.
- [ ] Атрибут в реляционной модели данных является синонимом хранимой процедуры, выполняемой при обновлении строк.
- [ ] Свойства отношений допускают существование кортежей с одинаковыми значениями всех полей, если у них разные системные адреса на диске.
- [ ] Домен представляет собой логическое объединение нескольких таблиц, связанных внешними ключами.
- [ ] Атрибуты отношения обязательно должны иметь сложную древовидную структуру для поддержки вложенных списков.
- [ ] Отношение всегда должно иметь упорядоченный список кортежей, отсортированный по времени их создания.
- [ ] Кортежи реляционной модели данных могут содержать неограниченное число дополнительных динамических столбцов.

* **Краткий ответ**:
  - **Кортеж (Tuple)** — строка таблицы, содержащая конкретный экземпляр данных сущности.
  - **Атрибут (Attribute)** — столбец таблицы, представляющий собой именованное свойство сущности.
  - **Домен (Domain)** — подмножество значений определенного типа, из которого атрибут может принимать свои значения.
  - **Свойства отношений**: Уникальность строк, неупорядоченность строк, неупорядоченность столбцов, атомарность значений атрибутов.
* **Пояснение**:
  - *Домен* задает не только базовый тип данных (например, строка или число), но и семантический смысл и диапазон значений. Например, домены «ФИО» и «Адрес» оба могут иметь тип `VARCHAR(200)`, но это разные домены, и операции сравнения между ними логически бессмысленны.
  - *Атрибут* является применением домена в контексте конкретного отношения. Например, в отношении `Сотрудники` атрибут `Домашний_Адрес` берет значения из домена «Адрес».
  - *Кортеж* — это упорядоченное множество пар вида `<Имя_Атрибута, Значение_Атрибута>`. Для схемы таблицы из $n$ столбцов кортеж является $n$-кой значений.
  - *Свойства отношений* гарантируют математическую строгость модели. Поскольку тело отношения — это множество кортежей, здесь работают законы теории множеств (в множестве нет одинаковых элементов, и порядок элементов не важен).

## 10. Теория нормализации

### 16. Опишите процесс нормализации. Ненормализованная форма. 1NF, 2NF, 3NF. Правила нормализации. #normalization #1nf #2nf #3nf
- [x] Процесс нормализации — это пошаговая декомпозиция таблиц с целью устранения избыточности данных и исключения аномалий модификации.
- [x] Отношение находится в 2NF, если оно находится в 1NF и каждый непервичный атрибут полностью функционально зависит от первичного ключа.
- [ ] Ненормализованная форма допускает хранение только уникальных числовых идентификаторов без текстовых полей.
- [ ] Для приведения к 1NF необходимо, чтобы все непервичные атрибуты транзитивно зависели от первичного ключа таблицы.
- [ ] 3NF требует обязательного наличия хотя бы одного внешнего ключа, ссылающегося на иерархическую структуру данных.
- [ ] Правила нормализации запрещают использование первичных ключей, состоящих более чем из одного атрибута.
- [ ] Отношение находится в 1NF, если в ячейках таблиц содержатся вложенные массивы и списки объектов.
- [ ] Процесс нормализации всегда приводит к увеличению производительности запросов SELECT, требующих соединения многих таблиц.
- [ ] Приведение ко второй нормальной форме (2NF) выполняется только для таблиц с простыми (одноатрибутными) первичными ключами.
- [ ] Нормализация требует удаления всех индексов и триггеров для упрощения логической структуры базы данных.

* **Краткий ответ**: Нормализация — метод проектирования БД, заключающийся в декомпозиции таблиц для минимизации избыточности и аномалий.
  - **1NF**: Все значения атрибутов атомарны (нет повторяющихся групп или массивов в ячейках).
  - **2NF**: Находится в 1NF + каждый неключевой атрибут функционально полно зависит от первичного ключа (нет зависимости от части ключа).
  - **3NF**: Находится в 2NF + отсутствуют транзитивные зависимости неключевых атрибутов от первичного ключа.
* **Пояснение**:
  - *Ненормализованная форма (UNF)*: Таблица, содержащая многозначные атрибуты или повторяющиеся группы. Например, поле `Телефоны` хранит строку `"123-45, 678-90"`.
  - *Переход к 1NF*: Разбиение многозначных атрибутов на отдельные строки или вынос их в дочернюю таблицу. Каждая ячейка теперь содержит ровно одно атомарное значение.
  - *Переход к 2NF*: Устранение частичных функциональных зависимостей. Если первичный ключ составной `(ID_Студента, ID_Курса)`, а поле `Название_Курса` зависит только от `ID_Курса` (части ключа), то таблицу декомпозируют на `Студенты_Курсы (ID_Студента, ID_Курса)` и `Курсы (ID_Курса, Название_Курса)`.
  - *Переход к 3NF*: Устранение транзитивных зависимостей. Если в таблице `Сотрудники (ID, ФИО, ID_Отдела, Название_Отдела)` поле `Название_Отдела` зависит от `ID_Отдела`, которое, в свою очередь, зависит от `ID` (первичного ключа), то зависимость `ID` -> `Название_Отдела` транзитивна. Таблицу декомпозируют на `Сотрудники (ID, ФИО, ID_Отдела)` и `Отделы (ID_Отдела, Название_Отдела)`.

## 11. Проектирование: детальные аспекты методологий

### 17. Охарактеризуйте метод проектирования «сверху-вниз». Укажите, в каких ситуациях этот метод является предпочтительным. #db-design #top-down #methodology
- [x] Метод проектирования «сверху-вниз» предпочтителен при создании крупных информационных систем с нуля, когда требования к бизнес-логике понятны, но структура исходных данных еще не определена.
- [x] Этот метод обеспечивает высокую концептуальную целостность базы данных и минимизирует дублирование сущностей благодаря предварительному ER-моделированию.
- [ ] Данный подход предпочтителен, когда необходимо быстро объединить множество разрозненных файлов данных и таблиц Excel в единую базу.
- [ ] Метод «сверху-вниз» используется только тогда, когда СУБД не поддерживает создание ограничений ссылочной целостности.
- [ ] Этот метод является предпочтительным при жестких ограничениях на объем оперативной памяти сервера базы данных.
- [ ] Проектирование «сверху-вниз» не требует привлечения бизнес-аналитиков и выполняется силами системного администратора.
- [ ] Данный подход применяется только для баз данных, не использующих реляционную модель и язык запросов SQL.
- [ ] При методе «сверху-вниз» проектирование начинается с детальной настройки физических параметров жестких дисков.
- [ ] Этот метод является предпочтительным, если логическая схема базы данных уже полностью реализована в сторонней СУБД.
- [ ] Метод запрещает проведение нормализации отношений на этапе логического проектирования базы данных.

* **Краткий ответ**: Метод «сверху-вниз» идет от общей концептуальной модели (бизнес-сущностей и требований) к логическим таблицам и физическим файлам. Он предпочтителен при создании новых крупных систем (greenfield-разработка) с четко определенной бизнес-логикой, так как обеспечивает системную целостность и исключает ранние ошибки структурирования.
* **Пояснение**: Метод «сверху-вниз» (top-down) — это архитектурный подход к проектированию. Он начинается с макроуровня и опускается до микродеталей.
  **Когда этот метод предпочтителен**:
  1. *Разработка систем с нуля (Greenfield)*: Когда нет унаследованных (legacy) баз данных или разрозненных файлов, структуру которых нужно обязательно сохранить.
  2. *Крупные корпоративные проекты*: Где сложная бизнес-логика и множество пользовательских сценариев требуют строгого соответствия концептуальной модели реальным процессам организации.
  3. *Проекты с сильной аналитической поддержкой*: Где есть возможность провести полноценный этап сбора и спецификации требований перед началом кодирования.
  *Ключевые преимущества*: Высокая логическая строгость схемы, соответствие бизнес-требованиям, минимизация риска создания дублирующихся или изолированных таблиц, упрощение будущей поддержки и доработки системы.

## 12. Шардирование и партиционирование

### 18. Шардирование и партицирование. Перечислите и опишите способы шардирования и партицирования #scalability #sharding #partitioning
- [x] Партиционирование (секционирование) — это разделение таблицы на части внутри одной СУБД, а шардирование — распределение частей таблицы по разным физическим серверам.
- [x] Способы разделения включают горизонтальное партиционирование (по строкам, например, по диапазонам дат) и вертикальное партиционирование (по колонкам).
- [ ] Шардирование выполняется исключительно на стороне жесткого диска с помощью настроек RAID-контроллера.
- [ ] Вертикальное шардирование распределяет четные строки таблицы на один сервер, а нечетные — на другой сервер.
- [ ] При горизонтальном партиционировании данные каждой колонки сохраняются в отдельном файле базы данных.
- [ ] Метод Range-партиционирования распределяет строки на основе криптографического хэша от значения первичного ключа.
- [ ] Шардирование баз данных полностью исключает необходимость использования индексов для ускорения поиска записей.
- [ ] Способ List-партиционирования разделяет данные на основе строго заданных непересекающихся диапазонов числовых значений.
- [ ] Шардирование по принципу "многие-ко-многим" требует обязательного дублирования всех серверов базы данных.
- [ ] Партиционирование таблиц поддерживается только в нереляционных базах данных типа MongoDB.

* **Краткий ответ**:
  - **Партиционирование (Partitioning)**: Разделение одной таблицы на несколько физических секций (партиций) в пределах одного сервера СУБД.
  - **Шардирование (Sharding)**: Горизонтальное масштабирование, при котором данные одной логической таблицы распределяются между независимыми физическими серверами (шардами).
  - **Способы**: Горизонтальное (по строкам) и вертикальное (по столбцам). Стратегии разделения: по диапазонам (Range), спискам (List), по хэшу (Hash/Key).
* **Пояснение**:
  - *Вертикальное партиционирование*: Разделение таблицы по колонкам. Редко используемые или очень длинные текстовые поля (типа `text`, `blob`) выносятся в отдельную таблицу с той же связью 1:1, чтобы ускорить сканирование основных полей.
  - *Горизонтальное партиционирование*: Разделение таблицы по строкам. СУБД распределяет строки по партициям автоматически на основе ключа партиционирования.
    * **Range (По диапазонам)**: Например, по датам. Записи за каждый месяц пишутся в свою партицию: `orders_2026_01`, `orders_2026_02`.
    * **List (По спискам)**: Разделение по конкретным значениям. Например, по странам: `RU`, `US`, `CN`.
    * **Hash (По хэшу)**: СУБД берет хэш-функцию от ключа (например, `ID_Покупателя`) и делит результат на число партиций (по остатку от деления).
  - *Шардирование* применяется, когда один сервер перестает справляться с нагрузкой на чтение/запись. При этом СУБД или приложение использует алгоритм маршрутизации (например, хэш-функцию от первичного ключа), чтобы определить, на какой именно шард записать или откуда прочитать строку.

## 13. Реляционная алгебра: подробный разбор операций

### 19. Опишите операции реляционной алгебры: выборка, проекция, соединение и деление отношений. Приведите примеры. #relational-algebra #operations
- [x] Выборка (фильтрация строк по условию) обозначается символом $\sigma$, а проекция (выбор конкретных столбцов с удалением дубликатов строк) — символом $\pi$.
- [x] Соединение ($\bowtie$) сопоставляет строки двух отношений на основе равенства общих атрибутов, а деление ($\div$) используется для поиска кортежей, связанных со всеми кортежами делителя.
- [ ] Операция выборки полностью удаляет из исходного отношения столбцы, не участвующие в условии фильтрации.
- [ ] Пример проекции: выборка сотрудников, у которых заработная плата превышает значение 50 000 рублей.
- [ ] Соединение отношений возможно только в том случае, если они имеют абсолютно идентичный набор всех атрибутов.
- [ ] Деление отношений в реляционной алгебре является базовой операцией и не может быть выражено через выборку и проекцию.
- [ ] Примером операции деления является деление числового значения цены товара на курс доллара в SQL-запросе.
- [ ] Проекция позволяет объединять строки из двух разных таблиц, имеющих одинаковые типы первичных ключей.
- [ ] Выборка обозначается символом $\bowtie$ и возвращает только общие строки для двух таблиц.
- [ ] Операции реляционной алгебры всегда физически удаляют данные с диска при выполнении операции разности.

* **Краткий ответ**:
  - **Выборка ($\sigma_{условие}(R)$)**: Возвращает кортежи отношения $R$, удовлетворяющие условию.
  - **Проекция ($\pi_{А_1, ..., А_k}(R)$)**: Возвращает отношение, содержащее только указанные столбцы $A_1, ..., A_k$ с исключением дубликатов строк.
  - **Соединение ($R \bowtie S$)**: Объединяет кортежи $R$ и $S$ на основе совпадения значений общих атрибутов.
  - **Деление ($R \div S$)**: Возвращает кортежи из $R(X,Y)$ по атрибутам $X$, которые в отношении $R$ связаны со всеми значениями атрибутов $Y$ из отношения $S(Y)$.
* **Пояснение**:
  - *Выборка (Selection)*: Выполняет горизонтальное подмножество. В SQL соответствует секции `WHERE`.
  - *Проекция (Projection)*: Выполняет вертикальное подмножество. В SQL соответствует секции `SELECT DISTINCT`.
  - *Соединение (Natural Join)*: Находится путем вычисления декартова произведения двух отношений с последующей фильтрацией по равенству одноименных атрибутов и последующей проекцией для удаления дублирующихся колонок. В SQL соответствует `NATURAL JOIN` или `JOIN ... ON`.
  - *Деление (Division)*: Сложная операция. Если $R$ имеет схему $(A, B)$, а $S$ имеет схему $(B)$, то результатом $R \div S$ является отношение со схемой $(A)$, содержащее такие значения $a$, что для любого $b$ из $S$ кортеж $(a, b)$ присутствует в $R$.

  - *Выборка*: `SELECT name FROM Users WHERE id = 1;`
* **Пояснение**:
  SQL (Structured Query Language) позволяет описывать требуемый результат, а СУБД сама решает, как его получить.
  **1. Создание таблицы (`CREATE TABLE`)**: Задает структуру отношения на физическом уровне.


## 16. Три-значная логика и NULL в SQL

### 23. Чем NULL отличается от пустой строки или нуля в SQL?. #sql #null #basics
- [x] NULL означает полное отсутствие значения или «неизвестное значение», в то время как ноль — это число, а пустая строка — строка нулевой длины.
- [x] Любые арифметические операции и сравнения с NULL (например, `NULL = NULL` или `NULL + 5`) возвращают значение NULL (Unknown) в трехзначной логике SQL.
- [ ] NULL физически равен пустой строке во всех реляционных СУБД, включая PostgreSQL и MS SQL Server.
- [ ] Использование NULL позволяет существенно сэкономить дисковую память, так как NULL вообще не занимает байт на диске.
- [ ] Ограничение UNIQUE в SQL не допускает вставку более чем одного значения NULL в индексируемую колонку в СУБД PostgreSQL.
- [ ] Для проверки равенства столбца значению NULL в SQL используется классический оператор равенства `= NULL`.
- [ ] NULL является синонимом логического значения FALSE и всегда приводит к ложности любых выражений в конструкции WHERE.
- [ ] СУБД автоматически заменяет все значения NULL на нули при расчете агрегатной функции SUM по выбранной колонке.
- [ ] NULL отличается от пустой строки только тем, что может храниться исключительно в числовых полях таблиц.
- [ ] При сравнении `'текст' != NULL` результатом выполнения выражения в SQL будет логическая истина (TRUE).

* **Краткий ответ**: NULL — это специальный маркер, обозначающий отсутствие значения или неизвестность. Ноль — это конкретное числовое значение, а пустая строка (`''`) — это строка нулевой длины. Сравнения с NULL возвращают Unknown, поэтому проверяются через `IS NULL` / `IS NOT NULL`.
* **Пояснение**:
  В реляционных базах данных важно отличать отсутствие информации от ее наличия в виде пустого значения.
  - **Ноль (0)**: Это числовое значение. Мы знаем, что количество товаров равно 0.
  - **Пустая строка (`''`)**: Это текстовое значение. Мы знаем, что отчество отсутствует (длина строки равна 0).
  - **NULL**: Мы не знаем значение. Например, телефон клиента неизвестен. Он может существовать, а может и нет.
  
  **Особенности трехзначной логики (3VL) в SQL**:
  В SQL логические выражения могут принимать три значения: `True` (Истина), `False` (Ложь) и `Unknown` (Неизвестно).
  Любое сравнение с NULL дает `Unknown`. Например, `WHERE Salary = NULL` не вернет ни одной строки, даже те, где Salary действительно NULL. Для этого нужно писать `WHERE Salary IS NULL`.
  То же самое касается операций сравнения: `NULL = NULL` вернет `Unknown`, а не `True`.

## 17. Архитектура ANSI/SPARC

### 24. Системы баз данных. Архитектура ANSI/SPARC (в т.ч. схема). Описать все уровни архитектуры. #dbms-architecture #ansi-sparc
- [x] Трехуровневая архитектура ANSI/SPARC состоит из внешнего (пользовательские представления), концептуального (общая логическая схема) и внутреннего (физическое хранение) уровней.
- [x] Данная архитектура направлена на обеспечение независимости данных, позволяя изменять один уровень без необходимости модификации соседних.
- [ ] Архитектура ANSI/SPARC описывает схему передачи сетевых пакетов между клиентом и сервером СУБД по протоколу TCP/IP.
- [ ] Внешний уровень архитектуры описывает структуру размещения файлов данных и индексов непосредственно на жестких дисках.
- [ ] Концептуальный уровень содержит описание только физических типов данных конкретной СУБД (например, double precision или bytea).
- [ ] Схема ANSI/SPARC требует обязательного использования промежуточного веб-сервера для трансляции SQL-запросов.
- [ ] Внутренний уровень архитектуры напрямую взаимодействует с пользователем через графический интерфейс (GUI).
- [ ] Архитектура ANSI/SPARC полностью отменена в связи с переходом на нереляционные документо-ориентированные базы данных.
- [ ] Согласно этой схеме, база данных должна физически реплицироваться на три разных сервера для каждого из уровней.
- [ ] Внешний уровень архитектуры является единым для всех пользователей и не допускает создания индивидуальных представлений.

* **Краткий ответ**: Архитектура ANSI/SPARC предлагает три уровня абстракции данных:
  1. *Внешний уровень (External Level)*: Индивидуальные пользовательские представления (представления, вьюшки).
  2. *Концептуальный уровень (Conceptual Level)*: Логическая структура всей базы данных (сущности, связи, ограничения).
  3. *Внутренний уровень (Internal Level)*: Физическая структура хранения данных (файлы, индексы, сжатие, страницы на диске).
  Схема: `[Пользователи] -> Внешний уровень (Представления) -> Концептуальный уровень (Логическая схема) -> Внутренний уровень (Физическое хранение) -> [Физическая БД на диске]`.
* **Пояснение**:
  Разработанная в 1975 году комитетом ANSI/X3/SPARC, эта архитектура решает проблему зависимости программ от данных.
  - **Внешний уровень**: Уровень пользователей и приложений. Каждый отдел компании видит только нужную ему часть базы данных (например, бухгалтерия видит зарплаты, а отдел кадров — стаж, оформленные через `VIEW`).
  - **Концептуальный уровень**: Описывает, какие данные хранятся в БД и какие связи существуют между ними. Это логический проект всей базы. Он не зависит от того, как данные хранятся физически, и от того, какие программы к ним обращаются.
  - **Внутренний уровень**: Физическая модель СУБД. Описывает, как именно записи размещаются в секторах диска, какие индексы созданы, как выделяется память под страницы.

### 25. Опишите принцип "Независимость от данных". #ansi-sparc #data-independence
- [x] Логическая независимость позволяет изменять концептуальную схему базы данных без необходимости изменения внешних представлений или прикладных программ.
- [x] Физическая независимость позволяет изменять структуру хранения данных на диске (например, переходить на новые индексы) без изменения концептуальной схемы.
- [ ] Принцип означает, что СУБД может работать без подключения к физическим накопителям (жестким дискам и SSD).
- [ ] Независимость от данных гарантирует, что при крахе операционной системы данные в таблицах не будут повреждены.
- [ ] Логическая независимость позволяет менять код прикладных программ без обновления версии компилятора.
- [ ] Физическая независимость означает независимость сервера СУБД от источника бесперебойного питания (ИБП).

* **Краткий ответ**: Независимость от данных — это возможность изменять схему данных на одном уровне архитектуры ANSI/SPARC без изменения схем на более высоких уровнях. Выделяют логическую независимость (изменение концептуального уровня не ломает внешний уровень) и физическую независимость (изменение внутреннего уровня не ломает концептуальный).
* **Пояснение**:
  - **Физическая независимость от данных**: Концептуальная схема остается неизменной при изменении физической структуры хранения.
    * *Пример*: Мы решили перенести файлы БД с одного диска на другой или создать индекс B-Tree для столбца. Это изменяет внутренний уровень, но логика SQL-запросов в приложениях и структура таблиц на концептуальном уровне остаются прежними.
  - **Логическая независимость от данных**: Внешние представления (пользовательские программы) остаются неизменными при изменении концептуальной схемы.
    * *Пример*: Мы разбили одну таблицу на две (декомпозиция при нормализации). Чтобы старые программы не сломались, мы создаем представление (`VIEW`) с прежним именем и структурой, выполняющее соединение (`JOIN`) новых таблиц. Программы продолжают работать так, будто ничего не изменилось.

Три условия (или аспекта) реляционной модели: структурный (данные в виде отношений), манипуляционный (операции реляционной алгебры/исчисления) и целостный (ограничения целостности сущностей и ссылок).
* **Пояснение**:
  Эдгар Кодд создал реляционную модель, чтобы преодолеть недостатки иерархических и сетевых БД, где для поиска данных требовалось писать сложные алгоритмы обхода графов ("ручная навигация" с помощью указателей).
  **Три аспекта реляционной модели**:
  1. *Структурная часть*: Все данные в базе представляются в виде отношений (двумерных таблиц) и ничего более.
  2. *Манипуляционная часть*: Набор декларативных операций для работы с данными. Мы описываем, *что* хотим получить (например, с помощью реляционной алгебры), а не *как* обойти дерево дисковых блоков.
  3. *Целостная часть*: Правила поддержания корректности данных (целостность сущностей — первичный ключ не NULL, ссылочная целостность — внешний ключ должен ссылаться на существующий первичный ключ).

### 28. Фундаментальные свойства отношений. Перечислите и опишите. #relational-model #properties
- [x] Фундаментальные свойства включают уникальность кортежей (отсутствие дубликатов строк) и неупорядоченность кортежей по вертикали.
- [x] Атрибуты отношения не упорядочены по горизонтали, их имена уникальны в рамках отношения, а все значения в ячейках атомарны.
- [ ] Свойства отношений требуют, чтобы количество строк в таблице всегда было равно количеству столбцов.
- [ ] В отношении допускается наличие полностью пустых строк, не содержащих даже первичного ключа.
- [ ] Порядок столбцов имеет критическое значение для корректного выполнения арифметических операций в СУБД.
- [ ] Свойства отношений допускают хранение сложных XML-деревьев внутри ячеек таблиц без нормализации.
- [ ] Уникальность кортежей гарантируется автоматическим шифрованием данных при записи на жесткий диск.
- [ ] Каждый атрибут отношения может принимать значения из любых доменов одновременно.

* **Краткий ответ**: Фундаментальные свойства отношений:
  1. Каждый кортеж уникален (нет дубликатов строк).
  2. Кортежи не упорядочены (порядок строк не важен).
  3. Атрибуты не упорядочены (порядок столбцов не важен).
  4. Имена атрибутов уникальны в рамках одного отношения.
  5. Значения атрибутов атомарны (простые типы данных, нет списков или вложенных таблиц).
* **Пояснение**:
  Эти свойства следуют из математического определения отношения как подмножества декартова произведения множеств.
  - *Уникальность строк*: В математическом множестве не может быть двух одинаковых элементов. В реляционной БД это свойство поддерживается обязательным наличием первичного ключа (`PRIMARY KEY`).
  - *Неупорядоченность*: Операции теории множеств не зависят от порядка перечисления элементов. В SQL порядок вывода строк гарантируется только при явном указании секции `ORDER BY`.
  - *Атомарность*: Это требование первой нормальной формы (1NF). В реляционной ячейке хранится только одно значение (например, число или строка), а не коллекция.

### 29. Основные понятия реляционной модели данных. Ключи. Ссылочная целостность. Покажите на примерах. #relational-model #keys #integrity
- [x] Первичный ключ (Primary Key) уникально идентифицирует строку в таблице, а внешний ключ (Foreign Key) ссылается на первичный ключ другой таблицы.
- [x] Ссылочная целостность требует, чтобы для каждого значения внешнего ключа существовала строка с аналогичным первичным ключом в связанной таблице.
- [ ] Потенциальный ключ — это ключ, который может быть использован в качестве внешнего ключа только после удаления первичного ключа.
- [ ] Ссылочная целостность запрещает использование значений NULL в любых столбцах реляционной таблицы.
- [ ] Пример ссылочной целостности: автоматическое удаление всей базы данных при попытке вставить некорректный символ.
- [ ] Внешний ключ обязательно должен иметь имя, полностью совпадающее с именем соответствующего ему первичного ключа.
- [ ] Ссылочная целостность проверяется только вручную администратором базы данных с помощью периодических запросов SELECT.
- [ ] Первичный ключ таблицы не может состоять более чем из одного столбца (атрибута).
- [ ] Значение внешнего ключа ни при каких условиях не может принимать пустое значение NULL.
- [ ] Ссылочная целостность гарантирует, что первичный ключ дочерней таблицы всегда численно больше первичного ключа родительской таблицы.

* **Краткий ответ**:
  - **Суперключ (Superkey)**: Набор атрибутов, уникально идентифицирующий кортеж.
  - **Потенциальный ключ (Candidate Key)**: Минимальный суперключ (нельзя удалить ни один атрибут без потери уникальности).
  - **Первичный ключ (Primary Key)**: Выбранный потенциальный ключ для идентификации строк.
  - **Внешний ключ (Foreign Key)**: Набор атрибутов одной таблицы, ссылающийся на первичный ключ другой таблицы.
  - **Ссылочная целостность (Referential Integrity)**: Правило, по которому внешний ключ должен либо быть равен какому-то значению первичного ключа в родительской таблице, либо быть NULL.
  *Пример*: Таблица `Сотрудники (id, name, dep_id)`. `id` — первичный ключ. `dep_id` — внешний ключ, ссылающийся на `id` в таблице `Отделы (id, name)`. Мы не можем вставить сотрудника с `dep_id = 99`, если в `Отделы` нет отдела с `id = 99`.
* **Пояснение**:
  Целостность данных — это их корректность и непротиворечивость. Реляционная модель поддерживает ее аппаратно:
  1. *Целостность сущностей (Entity Integrity)*: Ни один атрибут первичного ключа не может принимать значение NULL. Строку с неопределенным идентификатором невозможно адресовать.
  2. *Ссылочная целостность*: Ограничивает операции вставки, удаления и обновления.
     - При удалении родительской записи (например, отдела) СУБД может отреагировать согласно правилу внешнего ключа:
       * `ON DELETE RESTRICT / NO ACTION`: Запретить удаление отдела, пока в нем есть сотрудники.
       * `ON DELETE CASCADE`: Автоматически удалить всех сотрудников этого отдела.
       * `ON DELETE SET NULL`: Записать сотрудникам `dep_id = NULL`.

## 20. Реляционное исчисление

### 30. Реляционное исчисление. Пример. #relational-calculus #logic
- [x] Реляционное исчисление — это декларативный язык запросов, основанный на математической логике предикатов первого порядка.
- [x] Выделяют реляционное исчисление кортежей (где переменные представляют кортежи) и реляционное исчисление доменов (где переменные представляют значения доменов).
- [ ] Реляционное исчисление используется для выполнения сложных тригонометрических расчетов на стороне сервера СУБД.
- [ ] Пример исчисления кортежей в SQL: использование конструкции `HAVING SUM(Salary) > 10000` для фильтрации сгруппированных строк.
- [ ] Исчисление доменов полностью исключает возможность применения кванторов существования ($\exists$) и всеобщности ($\forall$).
- [ ] В реляционном исчислении запросы описывают пошаговый процедурный алгоритм извлечения данных из файлов.
- [ ] Реляционное исчисление разработано специально для работы с неструктурированными файловыми базами данных NoSQL.
- [ ] Формула реляционного исчисления не может возвращать логическое значение TRUE или FALSE при проверке кортежей.
- [ ] В исчислении кортежей переменные принимают значения исключительно из диапазона вещественных чисел.
- [ ] Реляционное исчисление является физическим представлением индексов B-Tree в оперативной памяти.

* **Краткий ответ**: Реляционное исчисление — декларативный язык запросов, основанный на исчислении предикатов первого порядка. В отличие от процедурной реляционной алгебры (где мы пишем формулу из операций), здесь мы описываем свойства данных, которые хотим получить. Бывает двух типов: исчисление кортежей (TRC) и исчисление доменов (DRC).
  *Пример (исчисление кортежей)*: Найти ФИО сотрудников с зарплатой > 50000.
  $\{ t.Name \mid Employee(t) \wedge t.Salary > 50000 \}$ (выбрать имя для кортежей $t$, принадлежащих отношению Employee, у которых атрибут Salary больше 50000).
* **Пояснение**:
  Реляционная алгебра и реляционное исчисление эквивалентны по своей выразительной силе (теорема Кодда). Любой запрос реляционной алгебры можно выразить в исчислении, и наоборот.
  Язык SQL базируется именно на реляционном исчислении (точнее, на исчислении кортежей). Мы пишем:
  ```sql
  SELECT t.Name FROM Employee t WHERE t.Salary > 50000;
  ```
  Это прямое отражение формулы TRC.
  - В исчислении кортежей переменная ($t$) обозначает строку таблицы.
  - В исчислении доменов переменные ($x_1, x_2$) обозначают отдельные ячейки (значения доменов). Формула DRC для того же запроса:
    $\{ x_{name} \mid \exists x_{id}, x_{salary} (Employee(x_{id}, x_{name}, x_{salary}) \wedge x_{salary} > 50000) \}$.

- [ ] Отношение формально определяется как ориентированный граф, ребра которого описывают функциональные зависимости.
- [ ] Схема отношения может содержать столбцы с одинаковыми именами, если они имеют разные домены допустимых значений.
- [ ] Экземпляр отношения — это логическое представление результатов соединения трех и более таблиц.
- [ ] Схема отношения задает только перечень первичных ключей, полностью игнорируя остальные атрибуты.
- [ ] Отношение состоит из заголовка отношений, тела отношений и системного реестра операционной системы.

* **Краткий ответ**:
  - **Формальная структура отношения**: Состоит из схемы (заголовка, Heading) и экземпляра (тела, Body).
  - **Схема отношения (Relation Schema)**: Задается как $R(A_1:D_1, A_2:D_2, ..., A_n:D_n)$, где $R$ — имя отношения, $A_i$ — атрибуты, а $D_i$ — их домены. Это статическая структура метаданных.
  - **Экземпляр отношения (Relation Instance)**: Множество из $r$ кортежей вида $(v_1, v_2, ..., v_n)$, где каждое $v_i \in D_i$. Это динамическое наполнение таблицы данными в конкретный момент времени.
* **Пояснение**:
  В терминах программирования:
  - Схема отношения подобна объявлению класса или структуры данных (например, `struct Student { int id; string name; }`). Она описывает форму данных, имена полей и их типы. Схема меняется редко (только через DDL-команды `ALTER TABLE`).
  - Экземпляр отношения подобен объектам этого класса, находящимся в памяти в процессе работы программы. Он постоянно меняется при добавлении, изменении или удалении строк (через DML-команды `INSERT`, `UPDATE`, `DELETE`).

### 32. Фазы проектирования базы данных. Первичный ключ отношений. #db-design #keys #phases
- [x] Фазы проектирования включают концептуальное (ER-модель), логическое (реляционные таблицы) и физическое (реализация в конкретной СУБД) проектирование.
- [x] Первичный ключ — это минимальный набор атрибутов, уникально идентифицирующий каждый кортеж отношения и не содержащий значений NULL.
- [ ] Первичный ключ обязательно должен состоять из автоинкрементного числового поля типа SERIAL.
- [ ] Фаза физического проектирования базы данных выполняется бизнес-аналитиками без привлечения технических специалистов.
- [ ] Первичный ключ может принимать дублирующиеся значения, если в таблице настроен уникальный индекс по другому полю.
- [ ] Логическое проектирование начинается с выбора физического сервера и установки на него операционной системы.
- [ ] Первичный ключ отношений предназначен исключительно для шифрования персональных данных пользователей на диске.
- [ ] В процессе концептуального проектирования создается база данных в виде готового SQL-кода для целевой СУБД.
- [ ] Первичный ключ может быть удален из таблицы, содержащей связанные строки в других таблицах с ограничением RESTRICT.
- [ ] Фазы проектирования выполняются последовательно и полностью исключают возможность возврата на предыдущий шаг при обнаружении ошибок.

* **Краткий ответ**:
  - **Фазы проектирования**: 1) Анализ требований; 2) Концептуальное проектирование (ER-диаграмма); 3) Логическое проектирование (реляционная схема, нормализация); 4) Физическое проектирование (типы данных СУБД, индексы, файлы).
  - **Первичный ключ (Primary Key)**: Минимальный потенциальный ключ, выбранный разработчиком для уникальной идентификации строк отношения. Он уникален (`UNIQUE`) и обязателен (`NOT NULL`).
* **Пояснение**:
  Процесс проектирования БД итеративен. Ошибки, допущенные на ранних фазах (например, неверно понятые бизнес-требования на концептуальном этапе), обходятся очень дорого, если обнаруживаются на этапе физической реализации или эксплуатации.
  
  **Первичный ключ**:
  - Может быть *простым* (один атрибут, например, `ID_Сотрудника`) или *составным* (несколько атрибутов, например, `ID_Рейса` + `Дата_Вылета` + `Номер_Места`).
  - Выбор первичного ключа важен для производительности: СУБД автоматически создает по нему уникальный индекс (обычно B-Tree) и часто использует его для физического упорядочивания данных на диске (кластеризованный индекс).

## 21. Жизненный цикл БД и сбор требований

### 33. Сбор и анализ требований. Спецификация требований пользователей для одного из пользовательских представлений. Опишите цикл работы СУБД. #db-design #requirements #dbms-cycle
- [x] Сбор требований включает интервьюирование пользователей, анализ документооборота предприятия и определение границ будущей информационной системы.
- [x] Спецификация требований описывает структуры данных, транзакции и ограничения целостности для конкретной роли пользователя (пользовательского представления).
- [ ] Цикл работы СУБД начинается с компиляции исходного кода всей операционной системы при каждом выполнении запроса SELECT.
- [ ] Пользовательское представление является физическим файлом на диске, содержащим персональные настройки шрифтов интерфейса.
- [ ] Анализ требований выполняется разработчиками баз данных без участия конечных пользователей и заказчиков.
- [ ] Спецификация требований пользователей содержит только перечень системных требований к объему оперативной памяти сервера.
- [ ] Цикл работы СУБД исключает этап обработки транзакций, допуская запись данных только в пакетном фоновом режиме.
- [ ] На этапе сбора требований сразу пишется код триггеров на языке PL/pgSQL для оптимизации будущей СУБД.
- [ ] Пользовательское представление (представление данных) всегда содержит полную физическую копию всех таблиц базы данных.
- [ ] Цикл работы СУБД завершается принудительным форматированием жестких дисков сервера раз в месяц.

* **Краткий ответ**:
  - **Сбор и анализ требований**: Первый этап ЖЦ БД. Включает определение целей системы, сбор описаний бизнес-процессов, интервью с сотрудниками, изучение бумажных форм и отчетов.
  - **Спецификация требований**: Документ, содержащий описание данных (сущности, атрибуты), запросы и транзакции, частоту использования системы и требования безопасности для конкретного пользовательского представления (User View — взгляд на БД отдельной группы пользователей, например, Отдела Продаж).
  - **Цикл работы СУБД**: Прием SQL-запроса -> Парсинг и синтаксический анализ -> Оптимизация (построение плана выполнения) -> Выполнение (чтение блоков с диска или из буфера, применение блокировок) -> Запись в журнал транзакций (WAL) -> Возврат результата пользователю.
* **Пояснение**:
  *Спецификация пользовательского представления* помогает декомпозировать сложную предметную область. Вместо проектирования всей БД сразу, аналитик проектирует локальные представления для каждого отдела, а затем объединяет их в единую концептуальную схему (интеграция представлений), разрешая конфликты синонимов (разные названия одного объекта) и омонимов (одно название для разных объектов).

## 22. Моделирование сущностей и связей (ER)

### 34. Модель данных «сущность-связь». Основные концепции ER-модели. #er-model #conceptual
- [x] ER-модель (Entity-Relationship) — это концептуальная модель данных, представляющая предметную область в виде сущностей, их атрибутов и связей.
- [x] Сущность представляет собой различимый объект реального мира, атрибут описывает свойство сущности, а связь отражает ассоциацию между ними.
- [ ] ER-модель является логическим представлением данных в реляционной СУБД и напрямую транслируется в SQL-код.
- [ ] Основной концепцией ER-модели является физическое размещение индексов B-Tree в оперативной памяти сервера.
- [ ] Сущность в ER-модели может содержать только один атрибут, который обязательно является числовым идентификатором.
- [ ] Связь между сущностями в ER-модели всегда должна иметь тип "многие-ко-многим", другие типы связей не допускаются.
- [ ] ER-модель разработана Эдгаром Коддом как дополнение к правилам нормализации реляционных таблиц.
- [ ] Атрибуты сущностей в концептуальной модели данных должны содержать физические пути к файлам на сервере.
- [ ] Модель «сущность-связь» используется исключительно для проектирования нереляционных графовых баз данных.
- [ ] Концепции ER-модели требуют обязательного удаления всех сущностей перед созданием связей между ними.

* **Краткий ответ**: ER-модель (Entity-Relationship model, модель сущность-связь) — семантическая модель предметной области, используемая для концептуального проектирования. Ключевые понятия: Сущность (Entity), Атрибут (Attribute), Связь (Relationship).
* **Пояснение**:
  Разработана Питером Ченом в 1976 году. ER-модель позволяет абстрагироваться от деталей реализации в конкретных СУБД и представить данные на уровне логики бизнеса.
  - **Сущность**: Класс однотипных объектов предметной области (например, `Сотрудник`, `Товар`).
  - **Атрибут**: Свойство сущности (например, `Имя`, `Цена`). Атрибуты могут быть простыми, составными (например, адрес: страна, город, улица), однозначными, многозначными (например, номера телефонов) или производными (вычисляемыми на основе других, например, возраст на основе даты рождения).
  - **Связь**: Ассоциация между двумя или более сущностями (например, связь «Работает» между `Сотрудником` и `Отделом`).

### 35. Опишите основные элементы ER-модели. #er-model #elements
- [x] Элементами ER-модели являются сущности (сильные и слабые), связи (различных степеней и кратности) и атрибуты (простые, составные, ключевые, многозначные).
- [x] Связи классифицируются по мощности (1:1, 1:M, M:N) и по степени участия сущностей (полное или частичное участие).
- [ ] Главным элементом ER-модели является системный триггер, выполняющий автоматическое резервное копирование данных.
- [ ] Атрибуты в ER-модели могут быть только простыми числовыми значениями без поддержки текстовых строк.
- [ ] Элементы ER-модели включают физические файлы табличных пространств СУБД и файлы конфигурации подключения.
- [ ] Слабая связь в ER-модели создается только между сущностями, находящимися на разных серверах базы данных.
- [ ] Первичный индекс таблицы является графическим элементом ER-диаграммы в нотации Чена.
- [ ] Степень связи в ER-модели определяет количество атрибутов, входящих в состав первичного ключа сущности.
- [ ] Многозначный атрибут в ER-модели — это атрибут, который может принимать значения из разных СУБД одновременно.
- [ ] Элементы ER-модели создаются разработчиком после физического импорта данных в базу данных PostgreSQL.

* **Краткий ответ**: Элементы ER-модели:
  1. *Сущности*: Сильные (независимые) и слабые (зависимые).
  2. *Атрибуты*: Простые, составные, ключевые, многозначные, производные.
  3. *Связи*: Бинарные (между 2 сущностями), рекурсивные (сущность связана сама с собой), n-арные (степень связи > 2).
  4. *Ограничения связей*: Кардинальность (кратность) и степень участия (обязательное/необязательное).
* **Пояснение**:
  - *Составной атрибут*: Состоит из нескольких простых атрибутов (например, `ФИО` -> `Фамилия`, `Имя`, `Отчество`).
  - *Многозначный атрибут*: Экземпляр сущности может иметь несколько значений для этого поля (у сущности `Книга` может быть несколько значений в атрибуте `Автор`).
  - *Ключевой атрибут*: Уникально идентифицирует экземпляр сущности (будущий первичный ключ).
  - *Степень связи*: Количество сущностей, участвующих в связи. Чаще всего связи *бинарные* (степень 2), но бывают *тернарные* (степень 3) и выше.

### 36. Атрибуты связей. Структурные ограничения. Кратность связей. Кратность сложных связей. Ограничения кардинальности и степени участия. #er-model #constraints #relationships
- [x] Связи в ER-модели могут обладать собственными атрибутами, которые при переходе к реляционной схеме переносятся в дочернюю или промежуточную таблицу.
- [x] Ограничение кардинальности определяет максимальное число экземпляров связи, а степень участия (обязательное или необязательное) задает минимальное число.
- [ ] Кратность сложных связей всегда строго равна отношению "один-к-одному" для предотвращения аномалий избыточности.
- [ ] Структурные ограничения связей настраиваются исключительно на физическом уровне хранения с помощью индексов B-Tree.
- [ ] Ограничение кардинальности определяет максимальное количество столбцов в связанной реляционной таблице.
- [ ] Атрибуты связей могут создаваться только для связей типа "один-к-одному", их использование в связях M:N строго запрещено.
- [ ] Полное участие сущности в связи означает, что первичный ключ этой сущности обязательно является составным.
- [ ] Кратность связей в ER-модели измеряется в мегабайтах дискового пространства, выделенного под хранение связей.
- [ ] Степень участия сущности в связи может принимать значения только 0 или 1, другие значения не допускаются.
- [ ] Атрибуты связей автоматически удаляются СУБД при первой операции очистки базы данных (VACUUM).

* **Краткий ответ**:
  - **Атрибуты связей**: Свойства, характеризующие саму связь, а не сущности. *Пример*: Связь «Обучается» между `Студентом` и `Курсом` имеет атрибут `Оценка`.
  - **Кратность (кардинальность) связи**: Максимальное количество экземпляров сущности $A$, которые могут быть связаны с одним экземпляром сущности $B$ (типы: 1:1, 1:M, M:N).
  - **Степень участия (Participation Constraint)**: Минимальное количество экземпляров связи. Бывает *полной* (обязательное участие, минимум 1, обозначается двойной линией) и *частичной* (необязательное участие, минимум 0, обозначается одинарной линией).
* **Пояснение**:
  Структурные ограничения связей задаются парой чисел (min, max), указывающих минимальное и максимальное количество связей, в которых может участвовать экземпляр сущности.
  - Пара (0, 1) означает необязательную связь 1:1 (частичное участие).
  - Пара (1, *) означает обязательную связь 1:M (полное участие).
  *Пример*: Связь «Руководит» между `Сотрудником` и `Отделом`. Участие сотрудника частичное (не каждый сотрудник — начальник, min=0, max=1). Участие отдела полное (у каждого отдела должен быть ровно один начальник, min=1, max=1).

### 37. Нотация «Вороньи лапки». Нотация П.Чена. #er-model #notations
- [x] В нотации П. Чена сущности обозначаются прямоугольниками, связи — ромбами, а атрибуты — овалами, присоединенными к сущностям.
- [x] Нотация «Вороньи лапки» (Crow’s Foot) представляет сущности в виде таблиц, а связи — в виде линий со специальными графическими символами на концах.
- [ ] Нотация «Вороньи лапки» является текстовым файлом конфигурации, описывающим правила доступа пользователей к таблицам.
- [ ] В нотации Чена связи отображаются исключительно в виде пунктирных стрелок с указанием физических типов данных.
- [ ] Нотация «Вороньи лапки» используется только при разработке физических схем баз данных Oracle.
- [ ] Символ "воронья лапка" на конце линии связи в ER-диаграмме указывает на ограничение кардинальности "ровно один".
- [ ] Нотация П. Чена полностью запрещает использование связей типа "многие-ко-многим" на концептуальном уровне.
- [ ] Атрибуты сущностей в нотации «Вороньи лапки» обозначаются в виде ромбов, расположенных внутри структуры таблицы.
- [ ] Нотации Чена и «Вороньих лапок» требуют написания SQL-запросов перед отрисовкой графических элементов.
- [ ] Линия связи в нотации «Вороньи лапки» не может содержать информацию об обязательности участия сущности в связи.

* **Краткий ответ**: Это две графические нотации для отображения ER-диаграмм:
  - **Нотация П. Чена**: Сущности — прямоугольники, связи — ромбы, атрибуты — овалы. Кардинальность связей подписывается символами (1, M, N) над линиями связи.
  - **Нотация «Вороньи лапки» (Crow's Foot / IE)**: Сущности — прямоугольники с разделами для имени и атрибутов (похожи на таблицы). Связи — линии. Кратность и обязательность связи отображаются значками на концах линий: круг (0), черточка (1), «трезубец» или лапка (много).
* **Пояснение**:
  - В *нотации Чена* удобно показывать сложные n-арные связи и атрибуты связей (они рисуются как овалы, прикрепленные к ромбу связи). Однако для больших баз данных диаграмма Чена становится перегруженной и нечитаемой.
  - В *нотации «Вороньи лапки»* атрибуты перечислены внутри прямоугольника сущности. Ключевые атрибуты помечаются сверху. Значки на концах связи показывают min и max кардинальность:
    * `o|` — ноль или один (необязательная связь 1:1).
    * `||` — ровно один (обязательная связь 1:1).
    * `o<` — ноль или много (необязательная связь 1:M).
    * `|<` — один или много (обязательная связь 1:M).

    * `JSONB` преобразует данные в двоичный формат, поддерживает индексацию (GIN-индексы) и обеспечивает очень быстрый поиск и манипуляцию элементами (быстрое чтение, чуть более медленная вставка).

## 24. Проектирование баз данных: методология (продолжение)

### 39. Охарактеризуйте метод проектирования «снизу-вверх». Укажите, в каких ситуациях этот метод является предпочтительным. #db-design #bottom-up
- [x] Метод «снизу-вверх» (bottom-up) начинается с анализа существующих элементарных данных, отчетов и бумажных документов с последующим их объединением.
- [x] Этот метод является предпочтительным при реинжиниринге существующих информационных систем или при объединении нескольких разрозненных БД в одну.
- [ ] Проектирование «снизу-вверх» начинается с построения глобальной концептуальной ER-диаграммы предметной области.
- [ ] Данный подход предпочтителен при разработке сложных и крупных баз данных с нуля (Greenfield).
- [ ] Метод «снизу-вверх» требует обязательного удаления всех внешних ключей из реляционных таблиц на этапе проектирования.
- [ ] При этом методе проектирования сначала создаются роли пользователей СУБД, а затем на их основе пишутся SQL-запросы.
- [ ] Подход «снизу-вверх» исключает этап нормализации таблиц, так как все данные изначально считаются нормализованными.
- [ ] Метод является предпочтительным при жестких ограничениях на производительность дисковой подсистемы сервера.
- [ ] Проектирование «снизу-вверх» выполняется исключительно силами сторонних ИТ-консультантов без участия сотрудников компании.
- [ ] Этот метод основывается на автоматической генерации баз данных с помощью трансляторов языков C++ и Java.

* **Краткий ответ**: Метод «снизу-вверх» (bottom-up) — это индуктивный метод проектирования, идущий от частного к общему. Он начинается с анализа конкретных исходных данных (форм ввода, отчетов, таблиц Excel), которые группируются в атрибуты, нормализуются и объединяются в сущности и таблицы. Предпочтителен при реинжиниринге (модернизации) существующих ИС или миграции данных из старых плоских файлов и электронных таблиц в СУБД.
* **Пояснение**:
  В отличие от дедуктивного метода «сверху-вниз», где проектирование идет от абстрактных концепций бизнеса, метод «снизу-вверх» работает с уже овеществленными данными:
  1. Собираются все бумажные документы, отчеты и файлы Excel, циркулирующие в компании.
  2. Составляется полный список полей (атрибутов), встречающихся в этих документах (например, `Имя_Клиента`, `Артикул`, `Дата_Заказа`).
  3. С помощью правил нормализации эти поля распределяются по таблицам так, чтобы исключить дублирование.
  4. Формируются связи между полученными таблицами.
  
  **Когда предпочтителен**:
  - *Реинжиниринг*: Когда нужно перенести старую систему на новую платформу, и существующие базы данных служат главным источником правды.
  - *Простые, небольшие БД*: Где сущностей мало, а структура документов полностью описывает всю логику работы.
  *Недостаток*: Трудно применить на крупных проектах, так как за обилием мелких таблиц и документов легко потерять общую архитектурную концепцию и упустить новые бизнес-требования.

## 25. Качество схемы БД: избыточность и аномалии (продолжение)

### 40. Аномалии вставки. Аномалии модификации. Покажите и поясните на примерах. #anomalies #redundancy #normalization
- [x] Аномалия вставки (Insertion Anomaly) — это невозможность добавить новые данные в таблицу без привязки к несуществующим на данный момент объектам.
- [x] Аномалия модификации (обновления) приводит к логической несогласованности базы данных при изменении дублирующейся информации не во всех записях.
- [ ] Аномалия вставки возникает только тогда, когда в таблице превышено максимальное количество строк, поддерживаемое СУБД.
- [ ] Пример аномалии вставки: невозможность вставить запись из-за отсутствия свободного места на жестком диске сервера.
- [ ] Аномалия модификации описывает ситуацию, когда СУБД автоматически шифрует данные при изменении пароля пользователя.
- [ ] Для устранения аномалий вставки необходимо полностью отключить первичные и внешние ключи во всех таблицах схемы.
- [ ] Аномалия вставки возникает исключительно при использовании СУБД PostgreSQL и отсутствует в базах данных MySQL.
- [ ] Пример аномалии модификации: сбой сетевого соединения при отправке SQL-запроса на обновление записей.
- [ ] Аномалии модификации полностью устраняются путем увеличения размера буферного кэша СУБД в оперативной памяти.
- [ ] Данные аномалии возникают только в базах данных, находящихся в третьей нормальной форме (3NF).

* **Краткий ответ**:
  - **Аномалия вставки (Insertion Anomaly)**: Ситуация, когда мы не можем вставить строку в таблицу из-за того, что часть данных для нее еще не существует. *Пример*: В таблице `Сотрудники_Отделы (Таб_Номер, ФИО, Номер_Отдела, Название_Отдела)` мы не можем добавить новый созданный отдел, пока в него не устроен хотя бы один сотрудник (так как `Таб_Номер` — часть первичного ключа и не может быть NULL).
  - **Аномалия модификации / обновления (Modification/Update Anomaly)**: Возникает из-за избыточности, когда при изменении значения атрибута приходится менять его во многих строках, что грозит рассинхронизацией. *Пример*: При смене названия отдела его нужно обновить у всех сотрудников отдела. Если обновить не у всех — отдел получит разные названия в разных строках.
* **Пояснение**:
  Эти аномалии указывают на плохую структуру базы данных и лечатся нормализацией.
  Рассмотрим еще **Аномалию удаления (Deletion Anomaly)**: удаление одних данных приводит к нежелательному уничтожению других. *Пример*: Если увольняется единственный сотрудник отдела из таблицы `Сотрудники_Отделы`, мы удаляем его строку. Но вместе с ней безвозвратно стирается информация о существовании самого отдела (его номер и название), так как они хранились в этой же строке.
  *Решение*: Разделение таблицы на две независимые таблицы `Сотрудники (id, name, dep_id)` и `Отделы (id, name)`.

### 41. Функциональные зависимости. Полная функциональная зависимость. #normalization #functional-dependencies
- [x] Функциональная зависимость $X \to Y$ означает, что для любых двух кортежей отношения при равенстве значений атрибута $X$ обязательно совпадают значения $Y$.
- [x] Полная функциональная зависимость $X \to Y$ имеет место, если $Y$ функционально зависит от составного ключа $X$, но не зависит от любого его подмножества.
- [ ] Функциональная зависимость описывает исключительно математическое деление числовых колонок друг на друга в SQL.
- [ ] Частичная функциональная зависимость не влияет на приведение отношений ко второй нормальной форме (2NF).
- [ ] В функциональной зависимости $X \to Y$ множество атрибутов $Y$ называют детерминантом, а $X$ — зависимой частью.
- [ ] Полная функциональная зависимость может существовать только для таблиц, имеющих простой (одноатрибутный) первичный ключ.
- [ ] Зависимость называют транзитивной, если из $X \to Y$ и $Y \to Z$ не следует функциональная зависимость $X \to Z$.
- [ ] Функциональные зависимости определяются динамически СУБД на основе анализа частоты выполнения SQL-запросов.
- [ ] Полная функциональная зависимость гарантирует, что значения зависимого атрибута всегда равны NULL.
- [ ] Функциональные зависимости используются исключительно для настройки резервного копирования базы данных.

* **Краткий ответ**:
  - **Фуциональная зависимость (ФЗ, $X \to Y$)**: Отношение между атрибутами, при котором значение атрибута $X$ однозначно определяет значение атрибута $Y$.
  - **Полная функциональная зависимость**: Зависимость $X \to Y$, при которой атрибут $Y$ зависит от всего составного детерминанта $X$, но не зависит ни от какой его части (подмножества $X'$).
* **Пояснение**:
  - *Детерминант* — левая часть ФЗ ($X$).
  - *Зависимая часть* — правая часть ФЗ ($Y$).
  *Пример*: Отношение `Успеваемость (ID_Студента, ID_Предмета, Оценка, ФИО_Студента)`.
  Первичный ключ здесь составной: `(ID_Студента, ID_Предмета)`.
  - Зависимость `(ID_Студента, ID_Предмета) -> Оценка` является **полной**, так как оценка определяется только студентом и предметом совместно. Она не зависит отдельно от студента или отдельно от предмета.
  - Зависимость `(ID_Студента, ID_Предмета) -> ФИО_Студента` является **частичной** (неполной), так как `ФИО_Студента` функционально зависит от части составного ключа — `ID_Студента` (т.е. `ID_Студента -> ФИО_Студента`). Наличие частичных зависимостей неключевых полей от составного ключа нарушает правила 2NF.

### 42. Ненормализованная форма. Показать (нарисовать) пример. Почему денормализуют базу, если нормализация считается хорошей практикой? #normalization #denormalization
- [x] Ненормализованная форма допускает наличие неатомарных значений (массивов, списков или вложенных таблиц) в ячейках отношений.
- [x] Денормализацию (намеренное введение избыточности) проводят для повышения производительности чтения данных, чтобы избежать тяжелых операций соединения (JOIN) в высоконагруженных системах (OLAP/высокий Read Load).
- [ ] Ненормализованная форма всегда обеспечивает максимальную скорость выполнения транзакций записи данных (INSERT/UPDATE).
- [ ] Денормализация базы данных полностью исключает риск появления аномалий модификации и удаления строк.
- [ ] Денормализация выполняется разработчиком до начала этапа концептуального проектирования ER-модели.
- [ ] В ненормализованной форме таблицы отсутствуют любые типы функциональных зависимостей между столбцами.
- [ ] Пример денормализации — создание уникального индекса B-Tree для ускорения поиска по первичному ключу.
- [ ] Денормализованная база данных требует меньшего дискового пространства для хранения таблиц по сравнению с нормализованной.
- [ ] Ненормализованная таблица обязательно должна содержать только внешние ключи без содержательных полей.
- [ ] Денормализация таблиц поддерживается только в нереляционных СУБД типа Redis.

* **Краткий ответ**:
  - **Ненормализованная форма (UNF)**: Таблица, содержащая неатомарные (многозначные) атрибуты или повторяющиеся группы полей.
    *Пример (схематично)*:
    `[Студент: Иванов] | [Группа: ИТ-1] | [Телефоны: 8-900..., 8-911...] | [Предметы: Базы Данных, Программирование]`
  - **Причины денормализации**: Приведение к 3NF разбивает одну таблицу на множество мелких. При чтении приходится делать много соединений (`JOIN`), что сильно нагружает процессор и диск. Для ускорения тяжелых аналитических выборок (OLAP) или в высоконагруженных веб-проектах базу денормализуют — объединяют таблицы обратно, мирясь с избыточностью ради скорости чтения.
* **Пояснение**:
  *Плата за денормализацию*: Ускоряя чтение, мы замедляем запись (приходится обновлять данные в нескольких местах) и берем на себя ответственность за предотвращение аномалий обновления на уровне прикладного кода или триггеров.

### 43. Нормальная форма Бойса-Кодда (НФБД). Приведите пример. #normalization #bcnf
- [x] Отношение находится в НФБК (BCNF), если оно находится в 3NF и каждый его детерминант (атрибут или группа атрибутов, от которых зависит другой атрибут) является потенциальным ключом.
- [x] НФБК является более строгим вариантом 3NF и применяется к отношениям, имеющим несколько перекрывающихся составных потенциальных ключей.
- [ ] Отношение находится в НФБК, если в нем полностью отсутствуют любые потенциальные и первичные ключи.
- [ ] Пример нарушения НФБК: таблица содержит транзитивную зависимость неключевого поля от первичного ключа.
- [ ] Приведение к НФБК всегда выполняется путем добавления новых столбцов в существующие реляционные таблицы.
- [ ] В НФБК допускается существование неатомарных значений в ячейках столбцов отношений.
- [ ] НФБК требует, чтобы все потенциальные ключи таблицы состояли строго из текстовых типов данных.
- [ ] Любое отношение, находящееся в 3NF, автоматически и гарантированно находится в НФБК.
- [ ] Приведение к НФБК полностью исключает необходимость выполнения нормализации до уровня 1NF и 2NF.
- [ ] НФБК используется исключительно при физическом проектировании табличных пространств в СУБД Oracle.

* **Краткий ответ**: Отношение находится в Нормальной форме Бойса-Кодда (НФБК / BCNF), если оно находится в 3NF и любой его детерминант является потенциальным ключом.
  *Пример*: Таблица `Консультации (ID_Студента, Предмет, Преподаватель)`.
  *Условия*: Преподаватель ведет только один предмет. Студент по каждому предмету ходит к одному преподавателю.
  *Потенциальные ключи*: `(ID_Студента, Предмет)` и `(ID_Студента, Преподаватель)`.
  *Зависимости*:
  1) `(ID_Студента, Предмет) -> Преподаватель` (так как для студента и предмета преподаватель уникален).
  2) `Преподаватель -> Предмет` (так как преподаватель ведет ровно один предмет).
  Детерминант `Преподаватель` во второй зависимости **не является потенциальным ключом** (один преподаватель консультирует многих студентов). Это нарушает НФБК, хотя таблица находится в 3NF (все поля ключевые, транзитивных зависимостей нет).
  *Решение*: Декомпозиция на две таблицы: `Студент_Преподаватель (ID_Студента, Преподаватель)` и `Преподаватель_Предмет (Преподаватель, Предмет)`.
* **Пояснение**:
  Разница между 3NF и BCNF возникает только тогда, когда в отношении есть несколько составных перекрывающихся потенциальных ключей (имеющих общие атрибуты). Если потенциальный ключ один или они не перекрываются, то 3NF и BCNF совпадают.

## 26. Оптимизация баз данных

### 44. Оптимизация баз данных. Перечислите и опишите способы оптимизации БД. #scalability #performance #optimization
- [x] Способы оптимизации включают создание индексов (B-Tree, Hash), денормализацию данных, партиционирование таблиц и кэширование запросов.
- [x] Оптимизация включает тюнинг конфигурации СУБД (настройка буферов памяти), оптимизацию SQL-запросов (анализ EXPLAIN) и репликацию (Read/Write Splitting).
- [ ] Оптимизация баз данных выполняется путем обязательного удаления всех ограничений внешних ключей и триггеров.
- [ ] Самым эффективным способом оптимизации является регулярное форматирование дисков с данными.
- [ ] Способ оптимизации "денормализация" требует приведения всех таблиц базы данных к пятой нормальной форме (5NF).
- [ ] Создание индексов B-Tree замедляет выполнение запросов выборки SELECT, но значительно ускоряет операции INSERT.
- [ ] Оптимизация структуры данных исключает использование числовых типов данных, заменяя их строками TEXT.
- [ ] Анализ плана выполнения запроса EXPLAIN выполняется операционной системой автоматически при нехватке памяти.
- [ ] Репликация баз данных — это процесс автоматического удаления устаревших строк из таблиц истории.
- [ ] Способы оптимизации включают физическое уменьшение тактовой частоты процессора сервера СУБД.

* **Краткий ответ**: Способы оптимизации БД:
  1. *Индексирование*: Создание индексов по часто используемым в WHERE и JOIN полям.
  2. *Оптимизация SQL-запросов*: Анализ планов выполнения (`EXPLAIN`), устранение лишних JOIN, замена подзапросов на JOIN, использование пагинации.
  3. *Проектирование*: Нормализация (для OLTP) или денормализация (для OLAP).
  4. *Партиционирование*: Секционирование огромных таблиц.
  5. *Кэширование*: Использование Redis/Memcached перед БД.
  6. *Тюнинг СУБД*: Настройка выделения ОЗУ (в Postgres: `shared_buffers`, `work_mem`, `maintenance_work_mem`).
  7. *Масштабирование*: Репликация (мастер-слейв для чтения) и шардирование (для записи).
* **Пояснение**:
  Оптимизация должна начинаться с профилирования нагрузки.
  - Первым шагом является выявление медленных запросов (Slow Query Log) и анализ их планов через команду `EXPLAIN ANALYZE`. Это показывает, делает ли СУБД сканирование всей таблицы (Sequential Scan - медленно) или использует индекс (Index Scan - быстро).
  - Индексы ускоряют чтение, но замедляют запись (INSERT/UPDATE/DELETE), так как СУБД должна обновлять индексные деревья на диске. Поэтому индексы нужно создавать обдуманно.

## 27. Отображение ER-модели в реляционную

### 45. Правила преобразования ER-диаграммы в схему БД. Связи между сущностями. #er-model #db-design #mapping
- [x] Сильные сущности преобразуются в независимые таблицы, а связи 1:M реализуются путем переноса первичного ключа стороны "1" во внешний ключ стороны "M".
- [x] Связи M:N всегда преобразуются в отдельные промежуточные (ассоциативные) таблицы, первичный ключ которых является составным и содержит внешние ключи обеих сущностей.
- [ ] Связи 1:1 всегда требуют создания промежуточной таблицы, содержащей только первичные ключи обеих связанных сущностей.
- [ ] Слабая сущность преобразуется в таблицу, первичный ключ которой состоит исключительно из ее собственных атрибутов.
- [ ] При преобразовании связи M:N атрибуты этой связи отбрасываются и не сохраняются в реляционной схеме.
- [ ] Любая связь 1:M при преобразовании в схему реляционной БД требует обязательного создания двух новых промежуточных таблиц.
- [ ] Преобразование ER-диаграммы выполняется СУБД автоматически на физическом уровне хранения без участия разработчика.
- [ ] Многозначные атрибуты сущности преобразуются в отдельные столбцы той же самой таблицы с типом данных TEXT.
- [ ] Связи между сущностями отображаются в реляционной схеме исключительно с помощью системных триггеров.
- [ ] Составные атрибуты переносятся в реляционную таблицу в виде неизменных JSON-строк без разбиения на простые столбцы.

* **Краткий ответ**: Правила отображения:
  1. *Сильные сущности* становятся таблицами.
  2. *Слабые сущности* становятся таблицами, первичным ключом которых является комбинация ключа родителя и частичного ключа слабой сущности.
  3. *Связь 1:1*: Первичный ключ одной таблицы копируется как внешний ключ во вторую.
  4. *Связь 1:M*: Первичный ключ таблицы стороны «1» копируется как внешний ключ в таблицу стороны «M».
  5. *Связь M:N*: Создается новая промежуточная таблица с внешними ключами на обе исходные таблицы.
  6. *Многозначные атрибуты* выносятся в отдельную связанную таблицу.
* **Пояснение**:
  - *Для связи 1:M*: Ссылка идет от многих к одному. Например, у сущностей `Отдел` (1) и `Сотрудник` (M) связь реализуется добавлением поля `dep_id` в таблицу `Сотрудник`.
  - *Для связи M:N*: Например, `Студент` (M) и `Курс` (N). Мы не можем поместить внешний ключ ни в одну из таблиц без нарушения 1NF. Создается таблица `Студент_Курс (student_id, course_id)`. Оба поля являются внешними ключами, а их пара формирует составной первичный ключ промежуточной таблицы.

## 28. Архитектура СУБД: физическая и корпоративная

### 46. Типы архитектуры СУБД: 1-уровневая архитектура, 2-уровневая архитектура, 3-уровневая архитектура. Преимущества и недостатки каждого типа архитектуры СУБД. #dbms-architecture #types
- [x] Двухуровневая архитектура (клиент-сервер) разделяет логику интерфейса на клиенте и обработку данных на сервере, но страдает от сетевых перегрузок при масштабировании.
- [x] Трехуровневая архитектура включает сервер приложений (бизнес-логику), что повышает безопасность, упрощает масштабирование и снижает нагрузку на клиентские ПК.
- [ ] Одноуровневая архитектура предполагает размещение СУБД, бизнес-логики и интерфейса пользователя на трех географически удаленных серверах.
- [ ] Преимуществом одноуровневой архитектуры является максимальная защищенность данных от несанкционированного доступа по сети.
- [ ] Трехуровневая архитектура СУБД полностью исключает использование языка SQL для обмена данными между уровнями.
- [ ] Недостатком трехуровневой архитектуры является крайняя сложность резервного копирования физических файлов базы данных.
- [ ] В двухуровневой архитектуре тонкий клиент выполняет все расчеты и бизнес-логику информационной системы.
- [ ] Одноуровневая архитектура обеспечивает одновременный совместный доступ миллионов пользователей через Интернет.
- [ ] В трехзвенной архитектуре база данных размещается непосредственно в веб-браузере на компьютере конечного пользователя.
- [ ] Двухуровневая архитектура требует обязательного наличия серверов-посредников для балансировки нагрузки транзакций.

* **Краткий ответ**:
  - **1-уровневая (Mainframe/Local)**: БД, бизнес-логика и интерфейс на одном компьютере. Плюсы: Простота, скорость. Минусы: Нет сети, не масштабируется.
  - **2-уровневая (Клиент-Сервер)**: Клиент (толстый) содержит интерфейс и бизнес-логику; сервер содержит СУБД. Плюсы: Разделение ролей, общая база. Минусы: Тяжелый трафик, бизнес-логика на клиенте небезопасна.
  - **3-уровневая (Three-Tier)**: Клиент (тонкий, браузер) -> Сервер приложений (бизнес-логика) -> Сервер БД (СУБД). Плюсы: Масштабируемость, безопасность, тонкие клиенты. Минусы: Сложность разработки и администрирования.
* **Пояснение**:
  Эволюция архитектур шла по пути снижения требований к рабочим станциям пользователей (переход к тонкому клиенту) и централизации контроля. В трехзвенной архитектуре клиентский ПК вообще не подключается к базе данных напрямую. Все запросы идут к серверу приложений, который проверяет права, выполняет бизнес-правила и делает точечные запросы к СУБД, что резко повышает общую безопасность системы.

### 47. Клиент-серверная модель архитектуры. Нарисовать схему и описать. #dbms-architecture #client-server
- [x] В клиент-серверной модели клиент отправляет SQL-запросы по сети, а сервер СУБД обрабатывает их, выполняет транзакции и возвращает только результирующее отношение.
- [x] Данная модель позволяет централизованно управлять транзакциями, безопасностью и резервным копированием данных на выделенном хосте.
- [ ] Схема клиент-серверной модели требует обязательного физического объединения монитора клиента и системного блока сервера.
- [ ] Клиент в этой модели напрямую управляет чтением и записью физических секторов на жестком диске сервера.
- [ ] Недостатком модели является необходимость полной остановки сервера СУБД при отправке запроса SELECT любым клиентом.
- [ ] Клиент-серверная модель не поддерживает одновременное подключение к СУБД более одного клиентского приложения.
- [ ] В данной архитектуре сервер занимается исключительно визуализацией данных и отрисовкой окон интерфейса для пользователя.
- [ ] Сетевой обмен в клиент-серверной модели осуществляется путем пересылки файлов всей базы данных при каждом запросе.
- [ ] Модель полностью исключает возможность разграничения прав доступа к таблицам для различных сетевых клиентов.
- [ ] Схема модели описывает процесс взаимодействия исключительно внутри процессора без использования сетевых протоколов.

* **Краткий ответ**: Клиент-серверная модель (2-tier) состоит из двух активных компонентов: Клиента (Client) и Сервера (Server), связанных локальной или глобальной сетью.
  Схема: `[Клиент (Интерфейс + Логика)] <--- (SQL-запрос / Результат по TCP/IP) ---> [Сервер СУБД (Обработка, хранение, ACID)]`.
  Клиент формирует запрос и шлет его на сервер. Сервер СУБД обрабатывает запрос, считывает блоки данных с диска, фильтрует их и отправляет клиенту только отобранные строки (а не всю таблицу).
* **Пояснение**:
  Эта модель заменила архитектуру с общими файлами (File-Server), где клиент сам считывал файлы базы по сети и искал в них записи, перегружая сеть гигабайтами трафика.
  **Функции компонентов**:
  - *Сервер СУБД*: Обеспечение ACID, проверка ограничений целостности, оптимизация и выполнение запросов, управление блокировками, индексирование.
  - *Клиент*: Графический интерфейс (UI), формирование SQL-запроса, валидация ввода пользователя перед отправкой.

### 48. Корпоративная архитектура СУБД. Место БД в архитектуре ИС и приложений. #enterprise #architecture #db-role
- [x] База данных является центральным ядром корпоративной информационной системы, обеспечивающим хранение и сохранность всех критически важных бизнес-данных.
- [x] В современной архитектуре БД отделена от уровня представления и уровня бизнес-логики (хранится на выделенном сервере БД) для повышения безопасности и производительности.
- [ ] База данных размещается непосредственно внутри кода клиентского приложения на мобильном устройстве пользователя.
- [ ] Корпоративная архитектура требует, чтобы каждый отдел компании имел свою изолированную базу данных без связи с остальными.
- [ ] База данных в архитектуре ИС играет роль вспомогательного временного буфера, очищаемого при каждом закрытии приложений.
- [ ] СУБД корпоративного уровня должна работать исключительно в рамках однопользовательского локального режима.
- [ ] Место базы данных в ИС определяется как промежуточное звено между веб-браузером и операционной системой клиента.
- [ ] Корпоративные базы данных не поддерживают интеграцию с внешними веб-сервисами и API-интерфейсами приложений.
- [ ] Использование централизованной БД увеличивает затраты на дублирование информации внутри каждого подразделения.
- [ ] База данных в корпоративной архитектуре подчиняется исключительно файловым методам доступа операционной системы.

* **Краткий ответ**: В корпоративных информационных системах (КИС) база данных занимает центральное место на уровне хранения данных (Data Layer). Она функционирует как общее защищенное хранилище "золотого источника" информации для всех прикладных систем предприятия (CRM, ERP, бухгалтерия). В архитектуре приложений база данных отделяется от бизнес-логики (сервера приложений) и интерфейса (UI/представления) по паттерну MVC или Clean Architecture.
* **Пояснение**:
  В корпоративной среде базы данных группируются по типу нагрузки:
  - **OLTP (Online Transaction Processing)**: Базы данных, обслуживающие операционную деятельность (продажи, транзакции, склад). Характеризуются тысячами быстрых и коротких транзакций записи и чтения.
  - **OLAP / DWH (Data Warehouse)**: Аналитические хранилища данных. Сюда стекаются исторические данные из всех OLTP-баз для построения сложных отчетов и бизнес-аналитики (BI).
  Разделение операционных и аналитических БД предотвращает падение скорости работы КИС, когда тяжелый аналитический запрос блокирует запись новых заказов на кассах.

### 49. Опишите роль и основные задачи администратора баз данных. #dba #maintenance #security
- [x] Администратор баз данных (DBA) отвечает за установку, настройку, мониторинг производительности и резервное копирование СУБД.
- [x] К задачам DBA относятся обеспечение безопасности данных, разграничение прав доступа, оптимизация производительности дисков и ОЗУ, а также восстановление БД после сбоев.
- [ ] DBA занимается разработкой дизайна пользовательского интерфейса (UI/UX) и версткой веб-страниц приложений.
- [ ] Основной задачей DBA является написание текстов должностных инструкций для всех сотрудников ИТ-отдела компании.
- [ ] Администратор баз данных осуществляет физический ремонт клавиатур, мышей и мониторов пользователей.
- [ ] DBA должен ежедневно вручную переписывать все новые кортежи таблиц на бумажные носители информации.
- [ ] Роль DBA исключает участие в процессе планирования масштабирования и переноса баз данных на новые сервера.
- [ ] Администратор баз данных отвечает за компиляцию кода операционной системы при установке обновлений.
- [ ] DBA занимается исключительно продажей коммерческих лицензий СУБД сторонним организациям.
- [ ] Задачей DBA является удаление всех первичных ключей таблиц для ускорения выполнения запросов SELECT.

* **Краткий ответ**: Администратор БД (DBA) — специалист, обеспечивающий надежное, безопасное и эффективное функционирование систем баз данных.
  **Задачи**:
  1. Установка и обновление СУБД.
  2. Проектирование физической структуры (диски, таблицы, партиции).
  3. Обеспечение безопасности (создание пользователей, ролей, аудит доступа).
  4. Создание стратегии резервного копирования (Backup) и плановое восстановление (Recovery).
  5. Мониторинг и оптимизация производительности (настройка ОЗУ, индексов, сбор статистики).
  6. Обеспечение высокой доступности (настройка репликации, кластеров HA).
* **Пояснение**:
  DBA — критически важная роль в компании. В отличие от разработчиков баз данных (которые пишут SQL-код и проектируют логические схемы), DBA отвечает за работоспособность и сохранность данных в рабочей среде. Ошибка DBA при настройке резервного копирования может привести к потере ценных данных предприятия.

### 50. Принцип независимости внешних представлений данных от концептуального их представления. Независимость концептуального представления структур данных от физической реализации этих структур. #ansi-sparc #data-independence
- [x] Независимость внешнего уровня от концептуального (логическая независимость) позволяет менять схему таблиц без изменения пользовательских представлений.
- [x] Независимость концептуального уровня от физического (физическая независимость) позволяет менять структуры файлов и индексов без изменения схем таблиц.
- [ ] Данный принцип означает, что прикладные программы могут работать вообще без создания концептуальной схемы базы данных.
- [ ] Физическая независимость гарантирует, что база данных сможет функционировать при полном отсутствии жестких дисков.
- [ ] Логическая независимость данных требует, чтобы все пользователи имели абсолютно одинаковые права доступа к таблицам.
- [ ] Принцип запрещает использование представлений (VIEW) для скрытия конфиденциальных данных от определенных пользователей.
- [ ] Независимость концептуального уровня означает, что логическая схема базы данных пишется на языке программирования C++.
- [ ] Принцип требует обязательной перекомпиляции всех клиентских приложений при создании любого нового индекса в СУБД.
- [ ] Физическая независимость данных гарантирует автоматическую конвертацию таблиц в текстовые файлы формата CSV.
- [ ] Данный принцип применяется только к нереляционным базам данных распределенного типа.

* **Краткий ответ**: Это формулировка двух уровней независимости данных в архитектуре ANSI/SPARC:
  1. *Логическая независимость данных*: Изменение концептуальной схемы (структуры таблиц) не затрагивает внешние схемы (пользовательские представления). Например, декомпозиция таблицы компенсируется созданием `VIEW`.
  2. *Физическая независимость данных*: Изменение внутренней схемы (физического хранения, индексов, организации файлов) не требует изменения концептуальной схемы. СУБД скрывает физические детали от логического уровня.
* **Пояснение**:
  - **Пример логической независимости**: В БД была таблица `Employee (id, name, salary, address)`. Мы решили вынести `address` в отдельную таблицу `Address (id, employee_id, text)`. Это изменило концептуальный уровень. Но у нас есть старое приложение, которое ищет адрес по SELECT из `Employee`. Мы удаляем старую таблицу `Employee` и создаем `VIEW` с именем `Employee`, которая делает `SELECT id, name, salary, text AS address FROM NewEmployee JOIN Address ...`. Старая программа продолжает работать без изменения ее кода.
  - **Пример физической независимости**: Мы заметили, что поиск по `name` работает медленно. Мы пишем `CREATE INDEX ON Employee(name);`. Физическая структура изменилась (добавилось индексное дерево B-Tree), но SQL-запрос `SELECT * FROM Employee WHERE name = 'Иван'` остался тем же. СУБД сама подхватит индекс.

## 29. Транзакции и распределенные системы

### 51. Дайте определение понятию «транзакция» в контексте СУБД. Приведите пример транзакции. #transactions #acid
- [x] Транзакция — это неделимая (атомарная) последовательность операций манипулирования данными, которая либо выполняется полностью, либо не выполняется вообще.
- [x] Примером транзакции является перевод денег с одного банковского счета на другой, требующий уменьшения баланса отправителя и увеличения баланса получателя.
- [ ] Транзакция — это процесс физического копирования файлов базы данных на внешний резервный накопитель.
- [ ] Примером транзакции является отправка электронного письма пользователю при регистрации на сайте.
- [ ] Транзакция может быть частично зафиксирована в базе данных при возникновении аппаратной ошибки жесткого диска.
- [ ] Каждая отдельная строка в реляционной таблице называется транзакцией в математической теории множеств.
- [ ] Транзакции используются исключительно для шифрования трафика между клиентом и сервером СУБД.
- [ ] Пример транзакции: создание индекса B-Tree для ускорения выполнения сложного запроса выборки SELECT.
- [ ] Транзакция автоматически откатывается СУБД, если время ее выполнения превысило одну секунду.
- [ ] Понятие транзакции исключает необходимость контроля параллельного доступа пользователей к базе данных.

* **Краткий ответ**: Транзакция — это логическая единица работы с базой данных, состоящая из одного или нескольких SQL-запросов, которая переводит базу данных из одного согласованного состояния в другое и выполняется по принципу «все или ничего» (атомарность).
  *Пример*: Перевод 1000 рублей со счета А на счет Б.
  Шаг 1: `UPDATE Accounts SET balance = balance - 1000 WHERE id = 'A';`
  Шаг 2: `UPDATE Accounts SET balance = balance + 1000 WHERE id = 'B';`
  Оба шага должны выполниться вместе. Если после Шага 1 произойдет сбой, СУБД откатит списание (`ROLLBACK`), иначе деньги пропадут из системы.
* **Пояснение**:
  Транзакции подчиняются правилам **ACID**:
  - **Atomicity (Атомарность)**: Транзакция не может быть выполнена частично. При сбое все изменения откатываются.
  - **Consistency (Согласованность)**: Транзакция оставляет базу в согласованном состоянии (все правила и внешние ключи соблюдены).
  - **Isolation (Изолированность)**: Параллельные транзакции не должны влиять друг на друга.
  - **Durability (Долговечность)**: Если транзакция зафиксирована (`COMMIT`), изменения не будут потеряны даже при сбое питания сервера на следующем шаге.

### 52. Общее назначение реляционной алгебры. Ассоциативность и коммутативность. #relational-algebra #properties
- [x] Общим назначением реляционной алгебры является предоставление теоретической основы для декларативного извлечения данных из таблиц.
- [x] Свойства ассоциативности и коммутативности операций (например, соединения $\bowtie$) позволяют оптимизатору СУБД менять порядок выполнения операций для ускорения запросов.
- [ ] Реляционная алгебра предназначена исключительно для шифрования конфиденциальных данных в системном каталоге.
- [ ] Свойство коммутативности означает, что результат операции выборки зависит от физического расположения строк на диске.
- [ ] Ассоциативность реляционных операций полностью отменяет необходимость выполнения нормализации таблиц.
- [ ] Реляционная алгебра используется для написания скриптов графического интерфейса пользователя.
- [ ] Коммутативность операции разности отношений гарантирует, что $A - B = B - A$ для любых таблиц.
- [ ] Назначением алгебры является автоматическая генерация аппаратных прерываний процессора сервера СУБД.
- [ ] Ассоциативность операций требует, чтобы первичный ключ дочерней таблицы всегда был составным.
- [ ] Реляционная алгебра применяется только в базах данных, не поддерживающих использование языка запросов SQL.

* **Краткий ответ**: Реляционная алгебра — теоретический процедурный язык запросов, служащий математическим фундаментом для реляционных СУБД. Свойства ассоциативности и коммутативности реляционных операций (особенно соединения $\bowtie$ и пересечения $\cap$) позволяют оптимизатору запросов перестраивать дерево выполнения запроса (например, менять порядок соединения таблиц $A \bowtie (B \bowtie C) \equiv (A \bowtie B) \bowtie C$), выбирая наиболее быстрый план без изменения финального результата.
* **Пояснение**:
  - *Коммутативность*: Позволяет менять операнды местами. Например, естественное соединение коммутативно: $A \bowtie B \equiv B \bowtie A$. Объединение коммутативно: $A \cup B \equiv B \cup A$. Но разность отношений *не коммутативна*: $A - B \neq B - A$.
  - *Ассоциативность*: Позволяет группировать операнды. Соединение ассоциативно: $(A \bowtie B) \bowtie C \equiv A \bowtie (B \bowtie C)$.
  Оптимизатор СУБД использует эти математические законы. Если таблица $A$ содержит 10 строк, $B$ — 10 строк, а $C$ — 10 000 000 строк, то оптимизатор предпочтет сначала соединить мелкие таблицы $A \bowtie B$, а затем результат соединить с $C$.

### 53. Опишите ключевые проблемы, возникающие в распределенных системах баз данных. #scalability #distributed #cap-theorem
- [x] Проблемы включают сложность обеспечения консистентности данных (теорема CAP), задержки сети при синхронизации и сбои сетевых каналов (network partitions).
- [x] В распределенных системах усложняется выполнение распределенных транзакций, требующих тяжелых протоколов консенсуса вроде двухфазной фиксации (2PC).
- [ ] Распределенные базы данных полностью исключают возможность использования индексов для поиска строк на удаленных узлах.
- [ ] Ключевой проблемой является невозможность хранения текстовых данных на серверах, находящихся в разных часовых поясах.
- [ ] Распределенные системы баз данных работают только при физическом объединении всех процессоров на одной материнской плате.
- [ ] CAP-теорема доказывает, что распределенная база данных может одновременно обеспечить абсолютную консистентность, доступность и устойчивость к разделению.
- [ ] При сбое сети в распределенной БД автоматически удаляются все данные со всех жестких дисков системы.
- [ ] Проблема репликации заключается в том, что резервные копии данных должны храниться в зашифрованных ZIP-архивах.
- [ ] В распределенных СУБД запрещено использование внешних ключей для связывания таблиц в рамках одного сервера.
- [ ] Ключевой проблемой распределенных БД является полное отсутствие поддержки протокола передачи данных TCP/IP.

* **Краткий ответ**: Ключевые проблемы распределенных БД:
  1. *Обеспечение согласованности (Consistency)*: Синхронизация изменений между узлами (репликами).
  2. *Теорема CAP*: Распределенная система может обеспечить одновременно только два из трех свойств: согласованность (C), доступность (A) и устойчивость к разделению сети (P). При сетевом сбое приходится выбирать между C и A.
  3. *Сетевые задержки (Latency)*: Передача данных по сети на порядки медленнее обращения к локальному диску.
  4. *Распределенные транзакции*: Требуют сложных протоколов координации (например, 2PC — Two-Phase Commit, или Raft) для гарантии атомарности изменений на разных серверах.
  5. *Отказоустойчивость*: Повышается вероятность выхода из строя отдельных узлов.
* **Пояснение**:
  В распределенных системах сложно поддерживать ACID в классическом виде. Поэтому часто переходят на модель **BASE** (Basically Available, Soft state, Eventual consistency — базовая доступность, неустойчивое состояние, согласованность в конечном счете), где данные на репликах могут обновляться с задержкой, но система гарантирует, что через некоторое время все реплики примут одинаковый вид.

Шаги включают: замену логических типов физическими типами данных СУБД, физическую реализацию ключей и ограничений, создание индексов, настройку партиционирования таблиц и выбор параметров хранения на уровне дисков (табличные пространства).

  ```sql
      id INT GENERATED ALWAYS AS IDENTITY PRIMARY KEY,
      name VARCHAR(100) NOT NULL,
      birth_date DATE CHECK (birth_date > '1900-01-01')
  );
  ```

  - Спроектировать ли секционирование (партиционирование) для таблицы логов, чтобы удалять старые записи целыми секциями.

### 55. Перечислите свойства отношений и объясните, как эти свойства отличают реляционную таблицу от простого плоского файла (например, электронной таблицы Excel). #relational-model #flat-files #excel

### 56. Почему отношение "многие-ко-многим" реализуется через промежуточную таблицу? #er-model #db-design #relationships
- [x] Реляционная модель данных требует атомарности ячеек (1NF), что делает невозможным хранение списка идентификаторов в одной ячейке таблицы.
- [x] Промежуточная (связующая) таблица превращает связь M:N в две связи типа 1:M, позволяя хранить дополнительные атрибуты связи и использовать индексы.
- [ ] Реализация связи M:N без промежуточной таблицы приводит к аппаратному сбою процессора сервера баз данных.
- [ ] Промежуточная таблица необходима для автоматического шифрования персональных данных пользователей.
- [ ] Без связующей таблицы невозможно создать индексы B-Tree для ускорения выполнения запросов выборки SELECT.
- [ ] Реляционная модель данных полностью запрещает использование внешних ключей в таблицах со связью 1:M.
- [ ] Промежуточная таблица создается только для временного хранения данных при миграции на другую СУБД.
- [ ] Хранение связи M:N в одной таблице требует обязательного использования операционной системы Windows Server.
- [ ] Промежуточная таблица позволяет исключить необходимость проверки ссылочной целостности в базе данных.
- [ ] Создание промежуточной таблицы требуется только в нереляционных СУБД для эмуляции работы реляционного SQL.

* **Краткий ответ**: В реляционной модели ячейки должны быть атомарными (требование 1-й нормальной формы).
  Если у нас есть сущности `Студент` и `Курс` со связью M:N (студент учит много курсов, на курсе учится много студентов), мы не можем добавить поле `Курсы` в таблицу `Студент`, так как в одну ячейку пришлось бы записать список `[1, 5, 8]`, что нарушит атомарность. Мы также не можем записать внешний ключ `student_id` в таблицу `Курсы` по той же причине.
  *Решение*: Связь M:N разбивается на две связи 1:M с помощью промежуточной таблицы `Студент_Курс (student_id, course_id)`. Каждая строка связующей таблицы фиксирует ровно один факт связи между конкретным студентом и конкретным курсом.
* **Пояснение**:
  Кроме соблюдения 1NF, промежуточная таблица позволяет:
  - Легко масштабировать связи без изменения структуры исходных таблиц.
  - Хранить *атрибуты связи*. Например, дату записи студента на курс или его итоговую оценку по этому курсу (`student_id`, `course_id`, `grade`, `enroll_date`). Хранить оценку в таблице `Студент` или `Курс` логически невозможно.

## 31. Реляционная модель: теория и сравнение с файлами (продолжение)

### 57. Определите и сравните понятия потенциальный, первичный и внешний ключ. #relational-model #keys
- [x] Потенциальный ключ уникально идентифицирует кортеж и является минимальным суперключом, а первичный ключ — это выбранный из них основной ключ.
- [x] Внешний ключ — это атрибут одной таблицы, который ссылается на первичный (или потенциальный) ключ другой таблицы для обеспечения связи между ними.
- [ ] Потенциальный ключ может содержать дублирующиеся значения, если в таблице не созданы внешние ключи.
- [ ] Внешний ключ обязательно должен иметь тип данных, отличный от типа данных связанного первичного ключа.
- [ ] Первичный ключ таблицы может принимать значения NULL, если потенциальный ключ состоит более чем из трех атрибутов.
- [ ] Потенциальные ключи создаются СУБД автоматически на основе анализа частоты выполнения запросов SELECT.
- [ ] Внешний ключ предназначен исключительно для физического сжатия данных на жестком диске.
- [ ] Первичный ключ таблицы создается только при отсутствии в ней ограничений уникальности UNIQUE.
- [ ] Внешний ключ дочерней таблицы всегда является частью составного первичного ключа родительской таблицы.
- [ ] Количество внешних ключей в одной реляционной таблице строго ограничено и не может превышать единицу.

* **Краткий ответ**:
  - **Потенциальный ключ (Candidate Key)**: Минимальный набор атрибутов, уникально идентифицирующий каждый кортеж отношения.
  - **Первичный ключ (Primary Key)**: Один из потенциальных ключей, выбранный разработчиком для уникальной идентификации записей. Он уникален и не может содержать NULL.
  - **Внешний ключ (Foreign Key)**: Столбец дочерней таблицы, хранящий значения первичного ключа родительской таблицы для связывания строк. Может содержать NULL и дубликаты.
* **Пояснение**:
  *Сравнение*:
  - Потенциальные ключи — это "кандидаты" на роль главного ключа. Например, в таблице `Сотрудник` потенциальными ключами являются: `Табельный_Номер`, `ИНН`, `Номер_Паспорта`.
  - Архитектор выбирает один из них (например, `Табельный_Номер`) в качестве *первичного ключа*. Остальные становятся альтернативными ключами (на них вешается ограничение `UNIQUE`).
  - *Внешний ключ* служит для реализации связей. В отличие от первичного ключа, он может дублироваться (много сотрудников работают в одном отделе, поэтому `dep_id` в таблице `Сотрудники` будет повторяться).

## 30. Реляционная алгебра: свойства и операции

### 58. Опишите операции реляционной алгебры: объединение, пересечение, разность и декартово произведение отношений. Приведите примеры. #relational-algebra #set-operations
- [x] Объединение ($\cup$), пересечение ($\cap$) и разность ($-$) требуют совместимости схем отношений по количеству и типам данных атрибутов.
- [x] Декартово произведение ($R \times S$) объединяет каждый кортеж отношения $R$ с каждым кортежем отношения $S$, формируя схему из суммы их атрибутов.
- [ ] Отрация пересечения отношений возвращает только те кортежи, которые отсутствуют в обоих исходных отношениях.
- [ ] Декартово произведение возможно только для таблиц, имеющих одинаковые первичные ключи.
- [ ] Разность отношений является коммутативной операцией, то есть результат $A - B$ равен результату $B - A$.
- [ ] Объединение отношений автоматически удаляет все столбцы, содержащие пустые значения NULL.
- [ ] Примером пересечения отношений является перемножение числовых значений колонок двух таблиц в SQL.
- [ ] Декартово произведение возвращает количество строк, равное сумме строк двух исходных отношений.
- [ ] Операция разности отношений обозначается в реляционной алгебре символом деления ($\div$).
- [ ] Данные операции реляционной алгебры изменяют структуры таблиц непосредственно на жестком диске.

* **Краткий ответ**:
  - **Объединение ($R \cup S$)**: Возвращает отношение, содержащее кортежи, которые есть в $R$, в $S$ или в обоих.
  - **Пересечение ($R \cap S$)**: Возвращает только те кортежи, которые есть одновременно и в $R$, и в $S$.
  - **Разность ($R - S$)**: Возвращает кортежи, которые есть в $R$, но которых нет в $S$.
  - **Декартово произведение ($R \times S$)**: Возвращает отношение, содержащее все возможные комбинации строк из $R$ и $S$.
* **Пояснение**:
  *Примеры в SQL*:
  - Объединение: `SELECT * FROM T1 UNION SELECT * FROM T2;` (автоматически удаляет дубликаты).
  - Пересечение: `SELECT * FROM T1 INTERSECT SELECT * FROM T2;`
  - Разность: `SELECT * FROM T1 EXCEPT SELECT * FROM T2;`
  - Декартово произведение: `SELECT * FROM T1 CROSS JOIN T2;`.

### 59. Опишите операции реляционной алгебры: выборка, проекция, соединение и деление отношений. Приведите примеры. #relational-algebra #special-operations
- [x] Выборка ($\sigma$) выполняет горизонтальное подмножество строк по предикату, а проекция ($\pi$) — вертикальное подмножество столбцов с исключением дубликатов.
- [x] Соединение ($\bowtie$) сопоставляет кортежи по общему условию, а деление ($\div$) находит кортежи делимого, связанные со всеми кортежами делителя.
- [ ] Операция деления реляционной алгебры используется для деления числовых полей одной таблицы на поля другой.
- [ ] Проекция отношений всегда возвращает количество строк, строго большее, чем в исходном отношении.
- [ ] Естественное соединение выполняется по условию неравенства первичных ключей связанных таблиц.
- [ ] Операция выборки обозначается символом $\pi$ и возвращает только уникальные имена столбцов.
- [ ] Деление отношений является базовой операцией и не может быть выражено через разность и проекцию.
- [ ] Выборка позволяет объединять строки из двух разных таблиц, имеющих одинаковые типы первичных ключей.
- [ ] Результатом операции проекции является отношение, схема которого совпадает со схемой декартова произведения.
- [ ] Операции реляционной алгебры всегда физически удаляют данные с диска при выполнении фильтрации.

* **Краткий ответ**:
  - **Выборка ($\sigma_{cond}(R)$)**: Выделяет строки, подходящие под условие. В SQL: `WHERE`.
  - **Проекция ($\pi_{cols}(R)$)**: Выделяет указанные столбцы и удаляет дубликаты строк. В SQL: `SELECT DISTINCT`.
  - **Соединение ($R \bowtie S$)**: Связывание строк по общему атрибуту. В SQL: `JOIN`.
  - **Деление ($R \div S$)**: Нахождение записей в $R$, связанных со ВСЕМИ записями в $S$. В SQL реализуется через `NOT EXISTS`.
* **Пояснение**:
  *Пример деления*:
  Таблица $R$ — `Студент_Курс (Студент, Курс)`. Таблица $S$ — `Обязательные_Курсы (Курс)`.
  Результат деления $R \div S$ вернет список `Студентов`, которые записаны на ВСЕ обязательные курсы из таблицы $S$. Если в $S$ лежат курсы «БД» и «СИ», а студент Иванов записан на «БД», «СИ» и «Физика», он попадет в результат. Студент Петров, записанный только на «БД», в результат не попадет.

## 32. Процесс нормализации: глубокий разбор (продолжение)

### 60. Опишите процесс приведения отношений БД к 3НФ и НФБК. #normalization #3nf #bcnf
- [x] Приведение к 3NF требует устранения транзитивных зависимостей неключевых атрибутов путем их вынесения в отдельные таблицы вместе с их детерминантами.
- [x] Приведение к BCNF (НФБК) требует, чтобы все детерминанты функциональных зависимостей стали потенциальными ключами отношений.
- [ ] Приведение к 3NF выполняется путем объединения всех связанных таблиц в одну большую денормализованную таблицу.
- [ ] Для приведения отношения к BCNF необходимо полностью удалить все первичные и внешние ключи.
- [ ] Третья нормальная форма допускает наличие частичных зависимостей неключевых полей от составного первичного ключа.
- [ ] Процесс нормализации до уровня BCNF всегда увеличивает скорость выполнения запросов вставки INSERT.
- [ ] Отношение находится в BCNF, если все потенциальные ключи состоят исключительно из одного числового столбца.
- [ ] Приведение к 3NF гарантирует автоматическое отсутствие аномалий многозначных зависимостей (4NF).
- [ ] Нормализация до BCNF выполняется только для таблиц, не содержащих внешних ключей.
- [ ] Процесс приведения к 3NF требует удаления всех индексов и триггеров для упрощения схемы.

* **Краткий ответ**:
  - **К 3NF**: 1) Привести к 2NF; 2) Найти транзитивные зависимости вида $A \to B \to C$ (где $A$ — первичный ключ, $B$ и $C$ — неключевые поля); 3) Вынести зависимую часть в новую таблицу, где детерминант $B$ станет первичным ключом: $R_1(A, B)$ и $R_2(B, C)$.
  - **К BCNF**: 1) Привести к 3NF; 2) Найти функциональные зависимости $X \to Y$, где детерминант $X$ не является потенциальным ключом; 3) Декомпозировать отношение на $R_1(X, Y)$ и $R_2(все\_остальное, X)$, сделав $X$ первичным ключом в $R_1$.
* **Пояснение**:
  *Разница*: 3NF допускает зависимость ключевого атрибута (входящего в составной потенциальный ключ) от неключевого детерминанта. BCNF жестко запрещает это: любой детерминант обязан быть суперключом.

### 61. Почему в реляционных базах данных не принято хранить несколько значений в одной ячейке? #normalization #1nf #database-design
- [x] Хранение нескольких значений в одной ячейке нарушает первую нормальную форму (1NF) и принцип атомарности данных.
- [x] Это усложняет выполнение запросов поиска, фильтрации и сортировки по этим значениям, а также делает невозможным использование стандартных индексов.
- [ ] Хранение списков в ячейке приводит к физическому сгоранию микросхем оперативной памяти сервера СУБД.
- [ ] Реляционные базы данных вообще не способны сохранить строку, содержащую символы запятой или точки с запятой.
- [ ] Хранение нескольких значений в одной ячейке требует обязательного использования платной лицензии СУБД.
- [ ] Данный запрет введен исключительно для экономии дискового пространства на магнитных накопителях.
- [ ] Неатомарные значения ячеек полностью исключают возможность создания внешних ключей для связывания таблиц.
- [ ] СУБД автоматически удаляет строки, в которых длина текстового поля превышает 10 символов.
- [ ] Хранение списков в ячейках поддерживается только в СУБД PostgresPRO при работе в однопользовательском режиме.
- [ ] Это ограничение позволяет программистам писать SQL-запросы без использования ключевого слова SELECT.

* **Краткий ответ**: Хранение нескольких значений в одной ячейке нарушает свойство атомарности атрибутов (1NF). Это усложняет поиск/фильтрацию (приходится использовать медленный `LIKE`), делает невозможной эффективную индексацию (B-Tree не работает на подстроках), усложняет обновление элементов и контроль целостности типов.
* **Пояснение**:
  *Исключение*: В современных СУБД появились типы данных `JSONB` и массивы, позволяющие хранить коллекции в ячейках. Это объектно-реляционное расширение, используемое для слабоструктурированных данных, когда гибкость важнее строгой нормализации.

### 62. Опишите процесс приведения отношений БД к 1НФ. #normalization #1nf
- [x] Приведение к 1NF требует обеспечения атомарности всех значений атрибутов и устранения повторяющихся групп полей (многозначных атрибутов).
- [x] Каждый многозначный атрибут выносится в отдельную таблицу, связываемую с исходной через отношение "один-ко-многим".
- [ ] Приведение к 1NF заключается в обязательном шифровании всех текстовых столбцов таблицы.
- [ ] Первая нормальная форма требует удаления из таблицы всех столбцов, не входящих в первичный ключ.
- [ ] Для приведения к 1NF необходимо разделить четные и нечетные строки таблицы по разным физическим файлам.
- [ ] Отношение находится в 1NF, если в нем настроена репликация данных на три резервных сервера.
- [ ] Процесс приведения к 1NF полностью запрещает использование числовых типов данных INTEGER.
- [ ] Приведение к 1NF требует обязательного объединения всех таблиц базы данных в одну.
- [ ] Первая нормальная форма допускает существование строк-дубликатов, если у них разные даты вставки.
- [ ] Процесс приведения к 1NF выполняется только для баз данных, не использующих язык SQL.

* **Краткий ответ**: Для приведения отношения к 1NF необходимо:
  1. Обеспечить атомарность всех значений на пересечении строк и столбцов (нет массивов, списков, вложенных таблиц).
  2. Устранить повторяющиеся группы однотипных столбцов (например, `Phone1`, `Phone2`).
  3. Выбрать первичный ключ для уникальной идентификации строк.
  *Способ*: Вынос многозначных полей в отдельную дочернюю таблицу.
* **Пояснение**:
  Таблица `Сотрудники (id, name, phones)` со значением `phones = '123, 456'` не находится в 1NF.
  После приведения к 1NF мы имеем две связанные таблицы: `Сотрудники (id, name)` и `Сотрудники_Телефоны (employee_id, phone)`. Теперь у каждого сотрудника может быть любое количество телефонов при сохранении атомарности ячеек.

### 63. Опишите процесс приведения отношений БД к 2НФ. #normalization #2nf
- [x] Приведение к 2NF требует, чтобы отношение находилось в 1NF и каждый неключевой атрибут функционально полно зависел от первичного ключа.
- [x] Частичные зависимости неключевых полей от частей составного первичного ключа устраняются путем их вынесения в отдельные родительские таблицы.
- [ ] Вторая нормальная форма требует обязательного удаления всех ограничений внешних ключей из реляционной схемы.
- [ ] Приведение к 2NF актуально только для таблиц, имеющих простой (состоящий из одного атрибута) первичный ключ.
- [ ] Вторая нормальная форма допускает наличие неатомарных значений в ячейках столбцов отношений.
- [ ] Процесс приведения к 2NF заключается в создании индексов B-Tree по всем числовым полям таблицы.
- [ ] Отношение находится в 2NF, если неключевые поля транзитивно зависят от первичного ключа таблицы.
- [ ] Вторая нормальная форма запрещает использование связей типа "один-ко-многим" в схеме базы данных.
- [ ] Приведение ко 2NF выполняется на физическом уровне хранения силами операционной системы.
- [ ] Процесс приведения ко 2NF требует обязательного объединения всех таблиц базы данных в одну схему.

* **Краткий ответ**: Для приведения отношения к 2NF необходимо:
  1. Убедиться, что таблица находится в 1NF.
  2. Найти частичные функциональные зависимости от частей составного первичного ключа.
  3. Вынести эти атрибуты и соответствующую часть ключа в отдельную таблицу.
  *Примечание*: Если первичный ключ простой (одно поле, например, `id`), таблица в 1NF автоматически находится во 2NF.
* **Пояснение**:
  *Пример*: Таблица `Заказы_Товары (Код_Заказа, Код_Товара, Количество, Название_Товара)`.
  Первичный ключ: `(Код_Заказа, Код_Товара)`.
  - `Количество` зависит от всего ключа (полная зависимость).
  - `Название_Товара` зависит только от `Код_Товара` (части ключа). Это нарушение 2NF.
  *Декомпозиция*:
  1. `Заказы_Детали (Код_Заказа, Код_Товара, Количество)`
  2. `Товары (Код_Товара, Название_Товара)`

## 33. Правила Эдгара Кодда

### 64. Перечислите и охарактеризуйте правила Кодда, определяющие фундаментальные основы и структуру реляционной модели. #codd #relational-model #rules
- [x] Информационное правило требует представления всей информации в базе данных исключительно в табличном виде на логическом уровне.

## 32. Процесс нормализации: глубокий разбор (продолжение)

### 66. Опишите правила приведения ненормализованной таблицы к Нормальной форме Бойса-Кодда. #normalization #bcnf #db-design
- [x] Процесс включает последовательную нормализацию таблицы: сначала приведение к 1NF (атомарность ячеек), затем к 2NF (устранение частичных зависимостей) и к 3NF (устранение транзитивных зависимостей).
- [x] На финальном этапе выявляются функциональные зависимости $X \to Y$, где детерминант $X$ не является потенциальным ключом, и выполняется декомпозиция отношения на два новых.
- [ ] Приведение к BCNF требует немедленного удаления всех потенциальных и первичных ключей из структуры отношений.
- [ ] Правила приведения к BCNF требуют обязательного слияния всех связанных таблиц в одну большую схему.
- [ ] Отношение приводится к BCNF путем автоматического преобразования всех текстовых типов данных в числовые.
- [ ] Для перехода к BCNF необходимо отключить все ограничения целостности UNIQUE и CHECK на сервере СУБД.
- [ ] Процесс приведения к BCNF исключает необходимость проведения нормализации до уровня первой нормальной формы.
- [ ] Приведение к BCNF гарантирует, что количество строк в результирующих таблицах уменьшится в несколько раз.
- [ ] Правила приведения запрещают создание промежуточных таблиц при разделении связей "многие-ко-многим".
- [ ] Приведение к BCNF выполняется только для баз данных, не поддерживающих работу с транзакциями.

* **Краткий ответ**: Алгоритм приведения ненормализованной таблицы к BCNF (НФБК):
  1. *Привести к 1NF*: Разбить сложные/многозначные ячейки на атомарные.
  2. *Привести к 2NF*: Убрать частичные зависимости неключевых полей от составного ключа.
  3. *Привести к 3NF*: Убрать транзитивные зависимости неключевых полей.
  4. *Привести к BCNF*: Найти все функциональные зависимости $X \to Y$. Если детерминант $X$ не является потенциальным ключом, разделить таблицу так, чтобы $X$ стал первичным ключом в новом отношении: $R_1(X, Y)$ и $R_2(все\_остальное, X)$.
* **Пояснение**:
  Нормализация до BCNF — это вершина классической нормализации на основе функциональных зависимостей. Процесс нормализации всегда строится на принципе *декомпозиции без потерь (Lossless-Join Decomposition)*: при разбиении таблицы $R$ на $R_1$ и $R_2$ операция их естественного соединения $R_1 \bowtie R_2$ должна вернуть в точности исходную таблицу $R$ (без потери информации и без появления фиктивных строк).
