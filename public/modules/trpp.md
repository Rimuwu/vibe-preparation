# Вопросы к вебинару по ТРПП

## 1. Системы контроля версий (Git)

### 1. Что такое система контроля версий (СКВ) и какие примеры систем существуют? #git #mainquestion
- [x] Специализированное ПО для фиксации истории изменений в коде и координации параллельной работы разработчиков; примеры: Git, SVN, Mercurial, CVS, Perforce.
- [x] Инструмент автоматического отслеживания состояний файлов проекта с возможностью отката и совместного доступа; примеры: Git, Subversion, Mercurial, CVS.
- [ ] Серверная операционная система для автоматического развертывания веб-приложений; примеры: Linux, Ubuntu, CentOS, Debian, RedHat, Fedora.
- [ ] Реляционная база данных для резервного копирования исходного текста конфигураций; примеры: PostgreSQL, MySQL, Oracle DB, MS SQL, SQLite.
- [ ] Интегрированная среда разработки для компиляции, статического анализа и отладки программ; примеры: VS Code, IntelliJ IDEA, Eclipse, NetBeans.
- [ ] Локальный файловый менеджер для архивации рабочих директорий по расписанию; примеры: WinRAR, 7-Zip, WinZip, Total Commander, Far Manager.
- [ ] Облачный хостинг для размещения статических веб-страниц и документации; примеры: Netlify, Vercel, Firebase Hosting, GitHub Pages, Heroku.
- [ ] Распределенная поисковая система для индексации текстовых документов в сети; примеры: Elasticsearch, Apache Solr, OpenSearch, Sphinx.
- [ ] Программный брокер сообщений для обмена данными между микросервисами; примеры: RabbitMQ, Apache Kafka, ActiveMQ, NSQ, NATS.
- [ ] Мониторинговая утилита для сбора метрик производительности серверов сборки; примеры: Prometheus, Grafana, Zabbix, Datadog, Nagios.

* **Краткий ответ**: СКВ — это система для фиксации и отслеживания изменений в файлах проекта. Примеры: Git, SVN, CVS, Mercurial, Perforce.
* **Пояснение**: Системы контроля версий позволяют сохранять состояния проекта, возвращаться к предыдущим версиям, сравнивать изменения и работать над кодом в команде без риска перезаписать чужую работу.

### 2. В чем разница между полным (snapshot) и инкрементальным (delta-based) копированием в СКВ? #git #mainquestion
- [x] Snapshot-подход сохраняет файлы целиком при наличии изменений, а delta-based хранит лишь разницу (дельты) между текущей и предыдущей версиями.
- [x] Полное копирование фиксирует слепки измененных файлов базы данных, а инкрементальное записывает только цепочки дельт относительно базового коммита.
- [ ] Полное копирование выполняется вручную разработчиком перед релизом, а инкрементальное происходит автоматически на сервере при пуше в репозиторий.
- [ ] Полное копирование дублирует файлы на внешний жесткий диск, а инкрементальное переносит измененные строки во временный оперативный кэш.
- [ ] Полное копирование поддерживается локальными СКВ, а инкрементальное хранение дельт реализовано только в распределенных архитектурах.
- [ ] Snapshot-подход требует обязательного шифрования данных, в то время как delta-based системы работают исключительно с открытым исходным кодом.
- [ ] Полное копирование оптимизирует сетевой трафик при слиянии веток, а инкрементальное замедляет выполнение команд индексации файлов.
- [ ] Snapshot-копирование применяется для текстовых документов, а delta-based хранение разработано специально для тяжелых бинарных архивов.
- [ ] Полное копирование блокирует файлы для редактирования другими пользователями, а инкрементальное позволяет вести параллельную правку строк.
- [ ] Snapshot-подход используется только на центральном сервере, а инкрементальные дельты формируются на компьютерах локальных разработчиков.

* **Краткий ответ**: Полное копирование сохраняет снимки (snapshots) файлов целиком, а инкрементальное хранит только разницу (дельты/изменения) между версиями.
* **Пояснение**: Системы вроде SVN хранят базовую версию файла и цепочку последующих изменений (diff-ов). Git же хранит слепки файлов (snapshots): если файл изменился, он сохраняется полностью (со сжатием), а если не изменился — создается ссылка на предыдущую версию.

### 3. Какие основные возможности предоставляют современные системы управления версиями? #git #mainquestion
- [x] Ведение сквозной истории изменений, изоляция работы в ветках, слияние кодовой базы, откат состояний и аудит авторства строк через blame.
- [x] Хранение логов модификации файлов, организация совместной работы команды, разрешение конфликтов правки и восстановление старых ревизий.
- [ ] Автоматический запуск интеграционных тестов, сборка исполняемых файлов, статический анализ синтаксиса и деплой на целевые сервера.
- [ ] Автоматическое исправление логических ошибок в функциях, рефакторинг архитектуры по расписанию и оптимизация алгоритмов сортировки.
- [ ] Хранение резервных копий операционной системы хоста и мгновенное восстановление работоспособности СУБД при аппаратных сбоях дисков.
- [ ] Шифрование сетевого трафика приложений, генерация SSL-сертификатов, блокировка вредоносных запросов и фильтрация спама в почте.
- [ ] Визуальное проектирование интерфейсов, генерация макетов страниц, прототипирование пользовательских сценариев и экспорт изображений.
- [ ] Управление задачами сотрудников, построение диаграмм Ганта, расчет трудозатрат на спринт и автоматическое начисление премий.
- [ ] Трансляция исходного кода на другие языки программирования, минификация скриптов и сборка клиентских бандлов для фронтенда.
- [ ] Мониторинг сетевых соединений, оркестрация контейнеров кластера, распределение оперативной памяти и лимитирование ресурсов CPU.

* **Краткий ответ**: Ведение истории изменений, откат к старым версиям, ветвление и слияние, совместная работа, аудит изменений (кто и когда изменил строку).
* **Пояснение**: Современные СКВ позволяют изолированно разрабатывать фичи в ветках, находить авторов строк (команда git blame), разрешать конфликты при слиянии и восстанавливать работоспособность проекта в случае ошибок.

### 4. Каковы особенности и главный недостаток локальных СКВ на примере RCS (Revision Control System)? #git #mainquestion
- [x] Хранят ревизии файлов на локальном компьютере в виде дельта-патчей; главный недостаток — отсутствие инструментов для командной работы.
- [x] Размещают базу данных версий на одном диске пользователя; главный недостаток — высокий риск потери всей истории при поломке ПК.
- [ ] Требуют постоянного подключения к глобальной сети; главный недостаток — крайне медленная скорость выполнения локальных команд.
- [ ] Используют сложную структуру реляционных таблиц СУБД; главный недостаток — полное отсутствие поддержки версионирования текстов.
- [ ] Функционируют исключительно в Unix-системах старых версий; главный недостаток — высокая стоимость лицензий для каждого инженера.
- [ ] Автоматически блокируют операционную систему при коммите; главный недостаток — несовместимость с современными процессорами x86.
- [ ] Занимают весь объем оперативной памяти при индексации; главный недостаток — невозможность хранения файлов размером более одного мегабайта.
- [ ] Переписывают историю коммитов без ведома пользователя; главный недостаток — частые конфликты при сохранении простых текстовых строк.
- [ ] Требуют ручного вычисления контрольных сумм файлов; главный недостаток — необходимость дублирования папок перед каждой фиксацией.
- [ ] Ориентированы на работу с бинарными библиотеками сборки; главный недостаток — невозможность отслеживания изменений в скриптах.

* **Краткий ответ**: Локальные СКВ хранят базу данных версий прямо на локальном диске. Главный недостаток — отсутствие средств совместной работы и высокий риск потери данных при сбое одного ПК.
* **Пояснение**: RCS хранит патчи (разницу между версиями) в специальных файлах на том же компьютере, где работает программист. Если диск выйдет из строя или файл повредится, вся история версий будет потеряна.

### 5. Каковы ключевые особенности и недостатки централизованных систем контроля версий (ЦСКВ) вроде SVN, CVS и Perforce? #git #mainquestion
- [x] Хранят историю на одном выделенном сервере; недостатки — невозможность локальных коммитов без сети и уязвимость перед сбоем сервера.
- [x] Базируются на едином центральном хранилище ревизий; недостатки — блокировка работы всей команды при отказе связи или поломке сервера.
- [ ] Размещают полные копии репозиториев на машинах клиентов; недостатки — избыточное разрастание локального диска из-за тяжелых файлов.
- [ ] Используют сложный синтаксис команд без графического интерфейса; недостатки — невозможность коммитить текстовые изменения в базу.
- [ ] Сохраняют только одну последнюю версию проекта; недостатки — полное отсутствие инструментов возврата кода к предыдущему состоянию.
- [ ] Требуют ручной синхронизации файлов через FTP-протокол; недостатки — постоянное затирание изменений других участников разработки.
- [ ] Ограничивают количество веток в репозитории до трех штук; недостатки — медленная скорость компиляции проекта на стороне сервера.
- [ ] Работают исключительно в рамках локальных вычислительных сетей; недостатки — невозможность подключения удаленных разработчиков.
- [ ] Заставляют шифровать каждый коммит уникальным ключом; недостатки — высокие требования к вычислительной мощности процессора хоста.
- [ ] Автоматически удаляют старую историю через тридцать дней; недостатки — необходимость ежедневного создания архивов репозитория.

* **Краткий ответ**: В ЦСКВ есть один центральный server с историей. Недостатки: без сети нельзя коммитить и смотреть историю; сбой сервера парализует работу.
* **Пояснение**: Разработчик выкачивает только рабочую копию (один снимок). Для любого коммита, просмотра логов или создания ветки нужен доступ к центральному серверу. Если сервер ломается, никто не может сохранить историю своей работы.

### 6. Какие особенности и потенциальные недостатки имеют распределенные системы контроля версий (РСКВ) на примере Git? #git #mainquestion
- [x] Предоставляют локальный клон всего репозитория с историей каждому участнику; недостатки — высокий порог входа и рост папки `.git`.
- [x] Хранят полную копию базы данных объектов на каждом ПК; недостатки — сложность освоения команд и разрастание репозитория от блобов.
- [ ] Требуют обязательного онлайн-подключения для любого действия; недостатки — медленный просмотр логов изменений без связи с сервером.
- [ ] Используют только инкрементальное сохранение цепочек дельт; недостатки — регулярное повреждение файлов базы данных при коммитах.
- [ ] Исключают возможность параллельной разработки в ветках; недостатки — частая блокировка файлов проекта для сторонних участников.
- [ ] Зависят от центрального сервера при создании локальных веток; недостатки — невозможность просмотра истории изменений без интернета.
- [ ] Работают исключительно с текстовыми конфигурациями приложений; недостатки — полный отказ при попытке добавить исходный код.
- [ ] Требуют регулярной ручной очистки индекса staging area; недостатки — автоматическое удаление незакоммиченных файлов из памяти.
- [ ] Ограничивают размер рабочей директории проекта на диске; недостатки — необходимость деления кода на мелкие изолированные папки.
- [ ] Привязаны к конкретной интегрированной среде разработки; недостатки — невозможность работы через стандартную консоль системы.

* **Краткий ответ**: В РСКВ каждый клиент имеет полную копию репозитория (включая всю историю). Недостатки: высокий порог вхождения и разрастание папки `.git` при наличии тяжелых файлов.
* **Пояснение**: Большинство операций (коммиты, логи, ветвление) выполняются мгновенно и локально, без сети. Центральный сервер нужен только для синхронизации (push/pull). Но из-за хранения всей истории локальный клон может занимать много места.

### 7. Что такое «репозиторий» в контексте СКВ и в чем различие между локальным и удаленным репозиториями? #git #mainquestion
- [x] Это структурированное хранилище файлов проекта и истории их изменений. Локальный находится на ПК инженера, удаленный — на сервере.
- [x] Специализированная база данных с полной историей ревизий. Локальный размещен на диске разработчика, удаленный — на хостинг-платформе.
- [ ] Это папка временного кэша компилятора исходного кода. Локальный хранит бинарные файлы, удаленный — исходные тексты скриптов.
- [ ] Это база учетных записей участников распределенной системы. Локальный содержит права доступа, удаленный — пароли шифрования.
- [ ] Это автоматизированный процесс сборки и тестирования. Локальный запускается на компьютере, удаленный — в облачном CI-конвейере.
- [ ] Это конфигурационный файл параметров среды окружения. Локальный задает пути к библиотекам, удаленный — сетевые адреса серверов.
- [ ] Это виртуальный контейнер для изоляции запущенных процессов. Локальный работает на хосте, удаленный — в кластере оркестрации.
- [ ] Это аналитический лог фиксации ошибок компиляции кода. Локальный собирает предупреждения IDE, удаленный — метрики продакшена.
- [ ] Это зашифрованный архив для отправки релиза конечному клиенту. Локальный содержит документацию, удаленный — исполняемый файл.
- [ ] Это инструмент визуализации графа вызова функций приложения. Локальный строит UML-схемы, удаленный — диаграммы развертывания.

* **Краткий ответ**: Репозиторий — структура данных, содержащая файлы проекта и историю их изменений. Локальный находится на ПК разработчика, удаленный — на сервере хостинга.
* **Пояснение**: Разработчик коммитит изменения в свой локальный репозиторий, а затем отправляет (push) их в удаленный репозиторий, чтобы другие члены команды могли забрать (pull) их себе.

### 8. Как Git хранит данные и с помощью какого механизма обеспечивается целостность данных в репозитории? #git #mainquestion
- [x] Хранит данные в виде снимков (snapshots) объектов, адресуемых по SHA-1 хэшу содержимого, что исключает незаметную порчу файлов.
- [x] Организует данные как константные объекты базы key-value, проверяя их целостность криптографическим хэшированием SHA-1.
- [ ] Записывает изменения в реляционную базу данных SQLite, контролируя целостность с помощью механизма внешних ключей (FK).
- [ ] Упаковывает версии в стандартные архивы формата ZIP, отслеживая контрольные суммы CRC32 для каждого файла индивидуально.
- [ ] Сохраняет дельта-патчи в структурированных XML-документах, заверяя целостность цифровой подписью центрального сервера.
- [ ] Формирует линейный текстовый лог изменений в JSON-файлах, проверяя валидность кодировки при каждом перезапуске консоли.
- [ ] Передает файлы в неизменном виде на сетевой диск, доверяя проверку ошибок аппаратным контроллерам RAID-массива сервера.
- [ ] Создает скрытые резервные копии в оперативной памяти, вычисляя хэш-таблицы распределенных адресов виртуальных потоков.
- [ ] Архивирует рабочую директорию в бинарные образы, шифруя блоки данных алгоритмами AES-256 с динамическим мастер-ключом.
- [ ] Генерирует цепочки связных списков в реестре операционной системы, сверяя метаданные через системные вызовы ядра хоста.

* **Краткий ответ**: Git хранит снимки файлов как объекты в базе данных key-value. Целостность отслеживается с помощью криптографического хэширования SHA-1.
* **Пояснение**: Каждый объект в Git (содержимое файла — blob, каталог — tree, коммит — commit) получает уникальный 40-символьный хэш SHA-1, вычисляемый от его содержимого. Если файл изменится хотя бы на один бит, изменится его хэш.

### 9. Какие основные состояния могут принимать файлы в системе контроля версий Git? #git #mainquestion
- [x] Отслеживаемые (Tracked), включающие состояния Unmodified, Modified, Staged; а также полностью неотслеживаемые файлы (Untracked).
- [x] Файлы разделяются на Untracked (новые файлы) и Tracked, которые могут быть неизмененными, измененными или подготовленными к коммиту.
- [ ] Скомпилированные (Compiled), исходные (Source), временные (Temporary), системные (System) и удаленные из репозитория (Deleted).
- [ ] Локальные (Local), удаленные (Remote), заблокированные (Locked), архивные (Archived) и свободные для редактирования (Free).
- [ ] Закоммиченные (Committed), отправленные (Pushed), обновленные (Pulled), слитые (Merged) и конфликтующие при слиянии (Conflicted).
- [ ] Индексированные (Indexed), зашифрованные (Encrypted), упакованные (Packed), текстовые (Text) и бинарные библиотеки (Binary).
- [ ] Статические (Static), динамические (Dynamic), виртуальные (Virtual), исполняемые (Executable) и конфигурационные (Config).
- [ ] Проверенные (Verified), валидированные (Validated), протестированные (Tested), дефектные (Baffled) и стабильные (Stable).
- [ ] Публичные (Public), приватные (Private), защищенные (Protected), общие (Shared) и изолированные внутри контейнера (Isolated).

* **Пояснение**: При инициализации репозитория (`git init`) создается скрытая папка `.git`. Именно в ней Git хранит всю историю, метаданные и настройки, тогда как рабочая папка содержит текущую рабочую копию файлов.

* **Пояснение**: Поскольку ветка — это всего лишь указатель на коммит, создание ветки (`git branch name`) происходит мгновенно и не расходует память. При создании коммита указатель текущей ветки автоматически сдвигается вперед.

- [ ] Долгоживущие ветки содержат бинарные файлы сборки; тематические включают текстовые скрипты; удаленные ветки хранят документацию.

* **Краткий ответ**: Долгоживущие ветки существуют на протяжении всего проекта (`main`, `dev`). Тематические (feature/bugfix) создаются для одной задачи и удаляются. Удаленные ветки (например, `origin/main`) — это локальные закладки, показывающие состояние веток на удаленном сервере на момент последней связи с ним.

### 13. Как устроен процесс слияния веток с помощью команды `git merge`? #git #mainquestion

* **Краткий ответ**: `git merge` соединяет две ветки. Если ветка-источник является прямым предком целевой, происходит `fast-forward` (простая перемотка указателя). Если история разошлась, Git находит общего предка, делает трехстороннее слияние и создает коммит слияния (merge commit).
* **Пояснение**: Во время трехстороннего слияния Git пытается автоматически объединить изменения. Если в одной и той же строке одного и того же файла сделаны разные изменения, возникает конфликт слияния, который разработчик должен разрешить вручную.

## 3. Жизненный цикл ПО и управление знаниями

### 24. Что такое жизненный цикл программного обеспечения (ЖЦПО / SDLC) и его модели? #sdlc #mainquestion

* **Краткий ответ**: ЖЦПО (SDLC) — период от идеи создания ПО до его вывода из эксплуатации. Модели ЖЦПО — методологические подходы, организующие этапы разработки.
* **Пояснение**: Модель ЖЦПО определяет, как именно команда движется по этапам разработки: последовательно (каскадная модель), итерациями (инкрементная, спиральная) или с фокусом на тестирование (V-модель).

### 25. В чем суть каскадной модели («Водопад» / Waterfall), каковы ее плюсы, минусы и область применения? #sdlc #mainquestion
- [x] Последовательное прохождение этапов без возврата назад; плюсы: простота управления, четкие сроки и бюджет; минусы: высокая стоимость ошибок на поздних этапах, клиент видит результат только в конце; применяется в проектах с жесткими, неизменными требованиями (космос, медицина).
- [ ] Итеративная разработка короткими спринтами; плюсы: гибкость; минусы: нет документации; применяется в стартапах.
- [ ] Постоянное создание прототипов; плюсы: высокая вовлеченность клиента; минусы: плохой код; применяется в веб-дизайне.
- [ ] Разработка с акцентом на автоматическое тестирование; плюсы: минимум багов; минусы: долго; применяется в финтехе.

* **Краткий ответ**: Waterfall — линейная модель разработки, где каждый этап начинается только после завершения предыдущего. Плюсы: планируемость. Минусы: негибкость. Применение: проекты с фиксированными требованиями.
* **Пояснение**: В Waterfall требования собираются один раз в начале. Если на этапе тестирования выяснится ошибка в анализе требований, возврат на первый этап обойдется очень дорого.

### 26. Каковы особенности, достоинства, недостатки и область применения инкрементной модели ЖЦПО? #sdlc #mainquestion
- [x] Разработка разбивается на части (инкременты), каждая из которых поставляет рабочий кусок системы; плюсы: ранний выпуск базовой версии, гибкость к требованиям; минусы: сложность проектирования архитектуры; применяется для больших систем, которые можно вводить в строй по частям.

- [ ] Постоянное изменение требований клиентом на ходу; плюсы: нет ТЗ; минусы: хаос в разработке; применяется в заказном ПО.
- [ ] Разработка ПО на основе готовых шаблонов; плюсы: скорость; минусы: шаблонный дизайн; применяется в e-commerce.

* **Краткий ответ**: Инкрементная модель предполагает разделение разработки на выпуски готовых частей (инкрементов) продукта. Плюсы: быстрый старт MVP. Минусы: риск плохой интеграции модулей. Применение: крупные коммерческие системы.
* **Пояснение**: Продукт разбивается на функциональные блоки. Сначала разрабатывают и выпускают первый блок (например, авторизация), затем второй (каталог) и т.д. Это снижает риски по сравнению с Waterfall.

- [ ] Модель, где разработка ведется только в виртуальных средах; плюсы: независимость от железа; минусы: цена лицензий; применяется в IoT.
- [ ] Модель с автоматическим слиянием кода роботами; плюсы: экономия времени; минусы: конфликты в Git; применяется в облачных сервисах.

* **Краткий ответ**: V-модель сопоставляет каждый этап разработки (требования, архитектура, кодирование) с соответствующим этапом тестирования (системное, интеграционное, модульное). Плюсы: раннее обнаружение дефектов. Применение: критически важное ПО.
* **Пояснение**: На этапе сбора требований сразу пишется план приемочных тестов, на этапе проектирования архитектуры — план интеграционных тестов. Это позволяет найти логические несостыковки еще до написания кода.

### 28. Каковы особенности прототипирования (Prototyping Model) как модели ЖЦПО, ее плюсы, минусы и область применения? #sdlc #mainquestion
- [x] Создание быстрого макета (прототипа) для демонстрации заказчику и уточнения требований; плюсы: высокая вовлеченность клиента, точное понимание ТЗ; минусы: трата ресурсов на создание временного кода, риск того, что клиент примет прототип за готовый продукт; применяется в проектах с высокой неопределенностью UI/UX.
- [ ] Разработка ПО на основе готовых библиотек-прототипов (шаблонов); плюсы: дешевизна; минусы: отсутствие уникальности; применяется для лендингов.
- [ ] Тестирование ПО силами пользователей (бета-тест); плюсы: бесплатные тестировщики; минусы: утечки кода; применяется для мобильных игр.
- [ ] Разработка объектно-ориентированных программ с прототипным наследованием; плюсы: скорость работы; минусы: утечки памяти; применяется в JavaScript.

* **Краткий ответ**: Модель на основе прототипа предполагает создание работающей упрощенной версии системы (прототипа) для согласования требований с заказчиком. Плюсы: снижение риска недопонимания. Минусы: затраты на временный код.
* **Пояснение**: Прототипирование помогает клиенту визуализировать систему. После утверждения прототипа он либо выбрасывается (система пишется начисто), либо постепенно дорабатывается (эволюционное прототипирование).

### 29. Как устроена спиральная модель (Spiral Model) Боэма, каковы ее плюсы, минусы и область применения? #sdlc #mainquestion
- [x] Итеративная модель с акцентом на анализ рисков на каждом витке спирали; плюсы: гибкость, детальное управление рисками; минусы: высокая стоимость анализа рисков, зависимость от экспертов; применяется в уникальных, дорогих и сложных проектах оборонного и исследовательского секторов.
- [ ] Модель автоматической непрерывной сборки с интеграцией тестов; плюсы: скорость; минусы: частые релизы пугают пользователей; применяется в SaaS.
- [ ] Линейная модель, в которой код пишется по спирали от простых функций к сложным; плюсы: простота кода; минусы: плохая масштабируемость; применяется в вузах.
- [ ] Разработка программ с использованием архитектуры спирального кэширования; плюсы: производительность; минусы: сложность настройки; применяется в базах данных.

* **Краткий ответ**: Спиральная модель делает упор на оценку рисков. Разработка идет витками, каждый из которых включает планирование, анализ рисков, конструирование (создание прототипа) и оценку заказчиком.
* **Пояснение**: Спиральная модель идеально подходит для проектов, где провал из-за неверных технологических решений или изменения рынка может обернуться большими финансовыми потерями.

### 30. Какую роль играет документирование знаний в разработке ПО и чем отличаются системы Confluence и MediaWiki? #docs #mainquestion

* **Пояснение**: Хорошая документация сокращает время адаптации новых разработчиков (onboarding) и снижает зависимость проекта от ключевых сотрудников (bus factor).

## 4. Непрерывная интеграция и доставка (CI/CD, Окружение)

### 31. Что такое концепция CI/CD в современной разработке программного обеспечения? #ci-cd #mainquestion
- [x] Набор практик автоматизации процессов сборки, тестирования (Continuous Integration) и развертывания (Continuous Delivery/Deployment) кода в рабочую среду при каждом изменении в репозитории.
- [ ] Технология контейнеризации приложений на уровне ядра операционной системы для повышения производительности диска.
- [ ] Процесс непрерывного общения с клиентом и сбора требований (Continuous Interview / Client Discussion).
- [ ] Методология парного программирования, где один разработчик пишет код, а второй непрерывно его комментирует.

* **Краткий ответ**: CI/CD — автоматизированный конвейер (pipeline) сборки, тестирования и доставки изменений от разработчика на продакшен-сервер.
* **Пояснение**: Внедрение CI/CD позволяет сократить Time-to-Market (время выхода фичи на рынок), повысить качество ПО за счет частых автоматических тестов и минимизировать человеческий фактор при деплое.

### 32. Каковы ключевые принципы успешного построения процессов CI/CD? #ci-cd #mainquestion
- [x] Единый репозиторий кода, автоматизация всех этапов (сборка, тесты, деплой), быстрое получение обратной связи, воспроизводимость сборок в идентичных средах и постоянная готовность релиза к развертыванию.
- [ ] Проведение ручного тестирования перед каждым коммитом, хранение кода на локальных флешках и развертывание по FTP вручную.
- [ ] Использование только одного языка программирования во всем проекте и отказ от использования сторонних библиотек.
- [ ] Компиляция кода на сервере клиента в реальном времени и шифрование исходников перед слиянием веток.

* **Краткий ответ**: Принципы: автоматизация всего процесса, версионирование конфигураций (IaC), быстрая обратная связь, идентичность окружений разработки и продакшена, частые небольшие релизы.
* **Пояснение**: Окружения (dev, staging, prod) должны быть максимально похожи, чтобы избежать ошибок вида "на моем компьютере все работает". Автоматические тесты должны выполняться быстро для эффективной работы команды.

### 33. Как организуется управление требованиями в контексте CI/CD? #ci-cd #mainquestion
- [x] Требования декомпозируются на мелкие задачи (User Stories/Tasks) в трекере (Jira, GitLab Issues), связываются с ветками Git и автоматически валидируются приемочными тестами в CI-конвейере.
- [ ] Требования пишутся в виде одного большого документа MS Word, который разработчики читают перед запуском CI-сервера.
- [ ] База требований хранится непосредственно в базе данных приложения и обновляется SQL-скриптами на этапе сборки.
- [ ] Требования формируются автоматически искусственным интеллектом на основе логов ошибок в рабочей среде.

* **Краткий ответ**: Управление требованиями в CI/CD строится на их декомпозиции, связывании задач с ветками в Git и автоматической проверке функционала через тесты в пайплайне.
* **Пояснение**: Связь "Задача -> Ветка в Git -> Тесты в CI/CD -> Релиз" обеспечивает полную прослеживаемость (traceability). Мы всегда знаем, какое требование реализовано в текущей сборке.

### 34. Что такое конвейер развертывания (Deployment Pipeline) и из каких типовых этапов он состоит? #ci-cd #mainquestion
- [x] Автоматизированное прохождение кода от коммита до продакшена; этапы: сборка (Build) -> модульное тестирование (Unit Test) -> интеграционное тестирование (Integration Test) -> деплой на тестовую среду (Staging) -> деплой на бой (Production).
- [ ] Логистическая цепочка поставки серверов и оборудования в дата-центр компании.
- [ ] Последовательность ручных проверок кода тимлидом, системным аналитиком и конечным заказчиком.
- [ ] Схема движения задач по колонкам Kanban-доски: Backlog -> In Progress -> Code Review -> Done.

* **Краткий ответ**: Конвейер развертывания — это автоматизированный процесс доставки изменений. Основные этапы: Build (Сборка), Test (Тестирование), Deploy (Развертывание), Release (Выпуск в эксплуатацию).
* **Пояснение**: Каждый этап является фильтром: если код падает на модульных тестах, сборка прерывается и переход на этап интеграционных тестов или деплоя не происходит. Это гарантирует, что сломанный код не попадет к пользователям.

### 35. Что такое «среда окружения» (Environment) в разработке ПО и какие объекты в нее входят? #bash #mainquestion
- [x] Выделенная инфраструктура, где запускается приложение; объекты включают операционную систему, сетевую конфигурацию, переменные окружения, базы данных и установленные runtime-библиотеки.

- [ ] Процесс общения разработчиков в мессенджерах и на планерках (командная среда).

* **Краткий ответ**: Среда окружения — это комплекс аппаратных, программных и сетевых средств, в котором функционирует ПО. Объекты окружения: ОС, СУБД, веб-серверы, системные библиотеки, конфигурационные файлы и переменные среды.
* **Пояснение**: Типичный проект имеет несколько сред: Development (локальный ПК), Testing/Staging (среда тестирования), Production (боевой сервер).

### 36. Назовите основные переменные среды окружения в ОС Linux и их назначение. #bash #mainquestion

### 37. Что такое «ярус» (tier) или уровень системного окружения в многоуровневой архитектуре приложений? #bash #mainquestion
- [x] Логическое разделение компонентов системы по их функциональным обязанностям (например, трехзвенная архитектура: клиентский ярус представления, ярус бизнес-логики и ярус данных).

* **Краткий ответ**: Системный ярус (Tier) — логически и физически обособленный слой архитектуры приложения (например, Presentation Tier — фронтенд, Application Tier — бэкенд, Data Tier — база данных).
* **Пояснение**: Разделение по ярусам позволяет масштабировать каждый слой независимо. Например, при высокой нагрузке на чтение базы данных можно добавить реплики на ярусе данных, не меняя код на бэкенде.

### 38. Какие основные компоненты входят в архитектуру современных сред окружения для веб-приложений? #bash #mainquestion
- [x] Балансировщик нагрузки (Load Balancer), веб-сервер (Nginx/Apache), сервер приложений (Node.js/Gunicorn), база данных (PostgreSQL/MongoDB) и система кэширования (Redis/Memcached).

* **Пояснение**: Архитектура обеспечивает отказоустойчивость и масштабируемость. Балансировщик (например, Nginx) распределяет запросы пользователей по нескольким серверам приложений, снижая нагрузку на каждый из них.

## 5. Виртуализация, контейнеризация и Docker

### 39. Что такое виртуализация и какие основные задачи она решает? #virtualization #mainquestion
- [x] Технология предоставления изолированных вычислительных ресурсов на базе одного физического компьютера; задачи: консолидация серверов, изоляция приложений, эффективное использование железа.
- [ ] Процесс создания 3D-моделей физических объектов для компьютерных игр и тренажеров.
- [ ] Разработка программного интерфейса (API) для удаленного управления робототехникой.
- [ ] Оптимизация исходного кода путем удаления неиспользуемых функций и сжатия бинарных файлов.

* **Краткий ответ**: Виртуализация — создание виртуального представления физических ресурсов с помощью специального ПО (гипервизора). Решает задачи оптимизации утилизации железа, изоляции и быстрого развертывания сред.
* **Пояснение**: Виртуализация позволяет запустить на одном мощном сервере десятки виртуальных машин с разными ОС, загружая физический процессор и память на максимум, вместо простоя оборудования.

### 40. Каковы главные преимущества внедрения технологий виртуализации в ИТ-инфраструктуру компании? #virtualization #mainquestion
- [x] Снижение затрат на покупку физического оборудования, быстрое создание новых серверов, высокая отказоустойчивость (миграция ВМ на другой хост) и упрощение резервного копирования.
- [ ] Увеличение физической скорости передачи данных по медным кабелям сети Ethernet.
- [ ] Полное отсутствие необходимости в написании тестов и администрировании серверов.
- [ ] Автоматическое ускорение работы любого однопоточного программного кода в несколько раз.

* **Краткий ответ**: Преимущества: экономия на оборудовании/электричестве, гибкость (создание ВМ за минуты), изоляция безопасности, удобство резервного копирования (снимки/snapshots) и легкое масштабирование.
* **Пояснение**: Благодаря виртуализации компания может арендовать ресурсы в облаке (IaaS) вместо покупки дорогостоящих серверов в собственный дата-центр.

### 41. С какими недостатками и ограничениями связано использование классической виртуализации? #virtualization #mainquestion
- [x] Потери производительности на оверхед (эмуляцию железа гипервизором), высокий расход оперативной памяти и диска (каждая ВМ требует полноценную гостевую ОС) и сложность администрирования.
- [ ] Полное отсутствие защиты от сетевых атак из внешней сети Интернет.
- [ ] Несовместимость с любыми базами данных и языками программирования, кроме C++.
- [ ] Ограничение на максимальное количество файлов в проекте (не более 1000 файлов).

* **Краткий ответ**: Недостатки: накладные расходы на работу гипервизора (оверхед), большой размер дисковых образов и потребление ОЗУ из-за необходимости запускать гостевую ОС для каждой машины.
* **Пояснение**: Для каждой виртуальной машины гипервизор выделяет фиксированные ресурсы и запускает отдельное ядро ОС. Это создает избыточность, которую решает более легковесная контейнеризация.

### 42. Что такое виртуализация на уровне операционной системы (контейнеризация), каковы ее задачи и применение? #virtualization #mainquestion
- [x] Изоляция процессов в рамках одной родительской ОС с использованием общего ядра (например, Docker); задачи: изоляция окружений приложений с минимальными накладными расходами; применяется для микросервисов и CI/CD.
- [ ] Установка нескольких физических жестких дисков на один сервер с целью ускорения файловых операций ОС.
- [ ] Эмуляция работы процессоров ARM на компьютерах с архитектурой x86-64 через программу QEMU.

* **Краткий ответ**: Виртуализация на уровне ОС (контейнеризация) запускает изолированные контейнеры, использующие ядро хост-системы. Задачи: минимизация оверхеда, скорость запуска. Применяется для деплоя приложений.

### 49. Почему контейнеризация стала стандартом де-факто в современных вычислительных системах и облачных технологиях? #docker #mainquestion

* **Пояснение**: Контейнер упаковывает приложение вместе со всеми зависимостями. Такой образ можно запустить на ноутбуке разработчика, на стейджинге и в облаке без изменения кода.

### 52. Что такое образ контейнера (Container Image) и какие два основных состояния выделяют у контейнера? #docker #mainquestion
- [x] Образ — это статический, доступный только для чтения шаблон файловой системы приложения; два состояния контейнера: пассивное (образ на диске) и активное (запущенный изолированный процесс в памяти).

- [ ] Образ — это резервная копия всей операционной системы хоста; два состояния: архивированное и распакованное.
- [ ] Образ — это SQL-схема базы данных; два состояния: транзакционное и нетранзакционное.

* **Краткий ответ**: Образ (Image) — неизменяемый шаблон с кодом и библиотеками. Контейнер (Container) — это запущенный процесс на основе образа. Состояния: пассивное (образ на диске) и активное (запущенный контейнер).
* **Пояснение**: Образ состоит из слоев (layers) и доступен только для чтения. При запуске контейнера поверх слоев образа создается тонкий перезаписываемый слой (writable layer), куда контейнер пишет временные файлы.

- [x] Программное обеспечение, управляющее созданием, запуском и остановкой контейнеров на хосте (например, Docker Engine, containerd, CRI-O); задачи: скачивание образов, изоляция ресурсов, настройка сети.

* **Краткий ответ**: Контейнерный движок — системная служба, которая принимает команды от пользователя/оркестратора, скачивает образы, настраивает сетевые интерфейсы, изолирует диски и запускает процессы контейнеров.
* **Пояснение**: Движок взаимодействует с ядром ОС Linux (через системную библиотеку runc), настраивая пространства имен (namespaces) и контрольные группы (cgroups), чтобы изолировать запускаемые процессы.

### 58. Что такое пространства имен (Namespaces) в Linux/Docker и какие основные типы используются для изоляции контейнеров? #docker #mainquestion
- [x] Технология ядра Linux, обеспечивающая изоляцию системных ресурсов; основные: `pid` (процессы), `net` (сеть), `mnt` (файловая система), `ipc` (межпроцессное взаимодействие), `uts` (имя хоста), `user` (UID/GID).

- [ ] Папки внутри контейнера, разделяющие файлы по типам данных (логи, конфиги, код).

* **Краткий ответ**: Namespaces (пространства имен) изолируют ресурсы ОС так, чтобы контейнер видел только свои процессы, сети, диски. Основные: `pid`, `net`, `mnt`, `ipc`, `uts`, `user`.
* **Пояснение**: `pid` namespace гарантирует, что процессы внутри контейнера не видят процессы хоста. `net` namespace дает собственный стек сети. `mnt` namespace изолирует корневую файловую систему.

## 6. Управление конфигурациями, DevOps и микросервисы

### 59. Что такое конфигурационное управление (Configuration Management) в контексте ИТ-инфраструктуры, каковы его цели и задачи? #ansible #mainquestion
- [x] Процесс систематического поддержания параметров работы систем в заданном состоянии; цели: автоматизация настройки серверов, исключение дрейфа конфигураций (configuration drift) и обеспечение воспроизводимости инфраструктуры.
- [ ] Настройка конфигурационных параметров в файле `settings.gradle` для сборки проекта.
- [ ] Управление правами доступа пользователей к репозиториям Git на сервере GitHub Enterprise.
- [ ] Процесс распределения оперативной памяти между виртуальными машинами в гипервизоре.

* **Краткий ответ**: Конфигурационное управление автоматизирует настройку, установку ПО и администрирование множества серверов, приводя их к единому описанному состоянию.
* **Пояснение**: Вместо того чтобы вручную настраивать сервера по SSH, администратор описывает конфигурацию в виде кода (IaC) в инструментах конфигурационного управления (например, Ansible, Puppet), которые автоматически настраивают системы.

### 60. Какие основные процедуры и этапы входят в процесс конфигурационного управления ИТ-инфраструктурой? #ansible #mainquestion
- [x] Идентификация конфигураций, контроль изменений (через Git), учет статуса (мониторинг текущего состояния серверов) и аудит/проверка соответствия эталону.
- [ ] Компиляция кода, запуск тестов, упаковка в архивы и отправка артефактов в Docker Hub.
- [ ] Выбор методологии ЖЦПО, сбор требований, написание ТЗ и дизайн пользовательского интерфейса.
- [ ] Регистрация доменных имен, покупка SSL-сертификатов и настройка DNS-серверов.

* **Краткий ответ**: Процедуры включают: идентификацию элементов конфигурации, управление изменениями (IaC), учет состояния систем (сопоставление с желаемым) и аудит соответствия.
* **Пояснение**: Все изменения в инфраструктуру вносятся через изменение кода конфигурационных скриптов (плейбуков Ansible) в Git-репозитории. Инструменты конфигурационного управления накатывают эти изменения и проверяют соответствие серверов эталону.

### 61. Каковы основные положения и цели методологии DevOps? #devops #mainquestion
- [x] Культурное и техническое объединение процессов разработки (Dev) и эксплуатации (Ops) для повышения частоты релизов, улучшения качества ПО и автоматизации процессов поставки ценности пользователю.
- [ ] Назначение одного разработчика ответственным за ручную сборку и деплой кода по выходным дням.
- [ ] Отказ от использования облачных провайдеров и переход на собственное физическое серверное оборудование.
- [ ] Использование языков программирования Bash и Python для написания абсолютно всего кода проекта.

* **Краткий ответ**: DevOps — это культура совместной работы разработчиков (Development) и системных администраторов (Operations) с фокусом на автоматизацию процессов, непрерывную интеграцию (CI/CD) и быструю реакцию на обратную связь.
* **Пояснение**: DevOps устраняет "стену непонимания" между разработчиками (которые хотят чаще выпускать фичи) и сисадминами (которые хотят стабильности) через автоматические тесты, мониторинг и общую ответственность за продукт.

### 62. Что такое непрерывная доставка (Continuous Delivery) и какие основные элементы она в себя включает? #devops #mainquestion
- [x] Практика, при которой код автоматически собирается, тестируется и подготавливается к выпуску, так что любое изменение может быть развернуто в продакшен в один клик вручную.
- [ ] Процесс круглосуточной доставки пиццы и кофе в офис ИТ-компании для разработчиков.
- [ ] Постоянное скачивание обновлений операционной системы хоста на локальный диск сервера.
- [ ] Автоматическая публикация исходного кода всех проектов компании в открытый доступ на GitHub.

* **Краткий ответ**: Continuous Delivery (CD) гарантирует, что код в основной ветке Git всегда находится в работоспособном и готовом к релизу состоянии. Развертывание на продакшен выполняется по нажатию кнопки человеком.
* **Пояснение**: Ключевые элементы CD: автоматическое тестирование, сборка артефактов (Docker-образов), развертывание на тестовые среды (staging). Отличие от Continuous Deployment заключается в том, что в Deployment выкатка на продакшен происходит полностью автоматически без ручного подтверждения.

### 63. Что такое непрерывная интеграция (Continuous Integration) и каковы ее ключевые элементы? #devops #mainquestion
- [x] Практика регулярного слияния кода всех разработчиков в общую ветку (несколько раз в день) с последующим автоматическим запуском сборки и тестов для раннего выявления ошибок.
- [ ] Процесс интеграции базы данных проекта с внешними государственными реестрами через защищенные каналы связи.
- [ ] Ежедневное проведение совещаний с участием всех отделов компании для обсуждения планов разработки.
- [ ] Объединение всех сторонних библиотек в один большой zip-архив для ускорения работы IDE.

* **Краткий ответ**: Continuous Integration (CI) — практика частого объединения изменений в основной ветке репозитория и их мгновенной автоматической проверки. Элементы: общий репозиторий (Git), автоматическая сборка, автотесты.
* **Пояснение**: CI позволяет избежать "интеграционного ада" в конце разработки, когда ветки расходились месяцами, и их слияние превращалось в кошмар с сотнями конфликтов. При CI ошибки находятся и исправляются сразу.

### 64. Что такое микросервисная архитектура и каковы ключевые принципы ее проектирования? #microservices #mainquestion
- [x] Подход к созданию ПО в виде набора мелких, независимых, взаимодействующих по сети сервисов; принципы: единая ответственность (Single Responsibility), независимое развертывание, собственная база данных для каждого сервиса и децентрализованное управление.
- [ ] Разработка приложения в виде единого монолитного блока с использованием микропроцессоров малой мощности.
- [ ] Использование миниатюрных функций (не более 10 строк кода) во всем исходном коде приложения.
- [ ] Создание веб-страниц, оптимизированных для отображения на экранах смарт-часов и фитнес-браслетов.

* **Краткий ответ**: Микросервисы — архитектурный стиль, разденяющий приложение на небольшие слабосвязанные службы. Принципы: децентрализация данных (database-per-service), API-ориентированное взаимодействие (REST, gRPC, очереди), независимый деплой.
* **Пояснение**: Каждый микросервис решает одну бизнес-задачу (например, сервис оплаты, сервис доставки). Они разрабатываются независимыми командами, могут быть написаны на разных языках и общаются друг с другом по сети.

### 65. Каковы основные преимущества и недостатки микросервисной архитектуры по сравнению с монолитной? #microservices #mainquestion
- [x] Преимущества: независимое масштабирование, гибкость технологий, простота развертывания фич; недостатки: сложность распределенной системы, накладные расходы на сетевые вызовы, трудности обеспечения согласованности данных (eventual consistency).
- [ ] Преимущества: отсутствие необходимости в базах данных; недостатки: программа работает только при наличии сети Интернет.
- [ ] Преимущества: автоматическое отсутствие ошибок в коде; недостатки: высокая стоимость покупки компиляторов.
- [ ] Преимущества: очень простая отладка на одном компьютере; недостатки: невозможность совместной работы разработчиков.

* **Краткий ответ**: Плюсы: масштабируемость отдельных частей, изоляция сбоев (падение одного сервиса не ломает всю систему), свобода выбора стека технологий. Минусы: сетевой оверхед, сложность отладки и тестирования, распределенные транзакции.
* **Пояснение**: Хотя микросервисы дают гибкость крупным командам, они приносят серьезную сложность в инфраструктуру (требуются оркестраторы типа Kubernetes, распределенное логирование Jaeger/Elasticsearch, брокеры сообщений RabbitMQ/Kafka).
