# Вопросы к вебинару по ТРПП

## 1. Системы контроля версий (Git)

### 1. Что такое система контроля версий (СКВ) и какие примеры систем существуют? #git
- [x] Программное обеспечение для управления изменениями в коде, позволяющее отслеживать историю изменений и координировать совместную работу; примеры: Git, SVN, Mercurial, CVS, Perforce.
- [ ] Серверная операционная система для автоматического развертывания веб-приложений; примеры: Linux, Ubuntu, CentOS.
- [ ] Реляционная база данных для резервного копирования исходного кода; примеры: PostgreSQL, MySQL, Oracle DB.
- [ ] Интегрированная среда разработки для автоматической компиляции и отладки программ; примеры: VS Code, IntelliJ IDEA, Eclipse.

* **Краткий ответ**: СКВ — это система для фиксации и отслеживания изменений в файлах проекта. Примеры: Git, SVN, CVS, Mercurial, Perforce.
* **Пояснение**: Системы контроля версий позволяют сохранять состояния проекта, возвращаться к предыдущим версиям, сравнивать изменения и работать над кодом в команде без риска перезаписать чужую работу.

### 2. В чем разница между полным (snapshot) и инкрементальным (delta-based) копированием в СКВ? #git
- [x] Полное копирование сохраняет состояние каждого файла целиком при его изменении (как снимки в Git), а инкрементальное — только различия (дельты) относительно предыдущей версии (как в SVN).
- [ ] Полное копирование выполняется вручную разработчиком перед релизом, а инкрементальное — автоматически сервером при каждом коммите.
- [ ] Полное копирование копирует файлы на внешний жесткий диск, а инкрементальное — только во временный кэш оперативной памяти.
- [ ] Полное копирование поддерживается только локальными СКВ, а инкрементальное — исключительно распределенными системами.

* **Краткий ответ**: Полное копирование сохраняет снимки (snapshots) файлов целиком, а инкрементальное хранит только разницу (дельты/изменения) между версиями.
* **Пояснение**: Системы вроде SVN хранят базовую версию файла и цепочку последующих изменений (diff-ов). Git же хранит слепки файлов (snapshots): если файл изменился, он сохраняется полностью (со сжатием), а если не изменился — создается ссылка на предыдущую версию.

### 3. Какие основные возможности предоставляют современные системы управления версиями? #git
- [x] Хранение истории изменений, ветвление и слияние веток, совместная работа нескольких разработчиков, откат к предыдущим версиям и аудит авторства изменений.
- [ ] Автоматический запуск тестов, деплой на сервер, компиляция кода в машинные инструкции и сборка дистрибутивов.
- [ ] Автоматическое исправление синтаксических ошибок в коде, рефакторинг по расписанию и оптимизация алгоритмов.
- [ ] Хранение резервных копий всей операционной системы и восстановление работы серверов при аварийных сбоях.

* **Краткий ответ**: Ведение истории изменений, откат к старым версиям, ветвление и слияние, совместная работа, аудит изменений (кто и когда изменил строку).
* **Пояснение**: Современные СКВ позволяют изолированно разрабатывать фичи в ветках, находить авторов строк (команда git blame), разрешать конфликты при слиянии и восстанавливать работоспособность проекта в случае ошибок.

### 4. Каковы особенности и главный недостаток локальных СКВ на примере RCS (Revision Control System)? #git
- [x] Хранят версии локально на компьютере пользователя в виде дельт; главный недостаток — невозможность полноценной совместной работы команды и риск потери всей истории при сбое диска.
- [ ] Требуют постоянного интернет-соединения; главный недостаток — медленная скорость работы с локальными файлами.
- [ ] Используют сложную структуру баз данных; главный недостаток — отсутствие поддержки текстовых файлов.
- [ ] Работают только в Unix-системах; главный недостаток — высокая стоимость лицензии на каждого разработчика.

* **Краткий ответ**: Локальные СКВ хранят базу данных версий прямо на локальном диске. Главный недостаток — отсутствие средств совместной работы и высокий риск потери данных при сбое одного ПК.
* **Пояснение**: RCS хранит патчи (разницу между версиями) в специальных файлах на том же компьютере, где работает программист. Если диск выйдет из строя или файл повредится, вся история версий будет потеряна.

### 5. Каковы ключевые особенности и недостатки централизованных систем контроля версий (ЦСКВ) вроде SVN, CVS и Perforce? #git
- [x] Все версии хранятся на единственном центральном сервере; недостатки — зависимость от подключения к сети для совершения коммитов и единая точка отказа (выход сервера из строя блокирует работу команды).
- [ ] Файлы хранятся только на локальных компьютерах; недостатки — высокая нагрузка на сетевые каналы из-за постоянного обмена данными.
- [ ] Сложный синтаксис команд и отсутствие графического интерфейса; недостатки — невозможность хранения бинарных файлов.
- [ ] Хранение только одной версии проекта без истории изменений; недостатки — невозможность отката кода к предыдущему состоянию.

* **Краткий ответ**: В ЦСКВ есть один центральный сервер с историей. Недостатки: без сети нельзя коммитить и смотреть историю; сбой сервера парализует работу.
* **Пояснение**: Разработчик выкачивает только рабочую копию (один снимок). Для любого коммита, просмотра логов или создания ветки нужен доступ к центральному серверу. Если сервер ломается, никто не может сохранить историю своей работы.

### 6. Какие особенности и потенциальные недостатки имеют распределенные системы контроля версий (РСКВ) на примере Git? #git
- [x] Каждый разработчик имеет локальную копию всего репозитория с полной историей; недостатки — сложность освоения для новичков и большой размер локального репозитория при наличии крупных бинарных файлов.
- [ ] Обязательное наличие центрального сервера для любой операции; недостатки — медленная работа команд просмотра логов.
- [ ] Невозможность работы без интернета; недостатки — отсутствие поддержки веток и слияний.
- [ ] Использование только инкрементального сохранения; недостатки — высокая частота повреждения файлов базы данных.

* **Краткий ответ**: В РСКВ каждый клиент имеет полную копию репозитория (включая всю историю). Недостатки: высокий порог вхождения и разрастание папки `.git` при наличии тяжелых файлов.
* **Пояснение**: Большинство операций (коммиты, логи, ветвление) выполняются мгновенно и локально, без сети. Центральный сервер нужен только для синхронизации (push/pull). Но из-за хранения всей истории локальный клон может занимать много места.

### 7. Что такое «репозиторий» в контексте СКВ и в чем различие между локальным и удаленным репозиториями? #git
- [x] Репозиторий — это хранилище проекта, содержащее его файлы и всю историю изменений. Локальный хранится на компьютере разработчика, а удаленный — на выделенном сервере (например, GitHub, GitLab) для совместного доступа.
- [ ] Репозиторий — это папка с кэшем компилятора. Локальный содержит скомпилированный код, удаленный — исходники.
- [ ] Репозиторий — это база данных пользователей системы. Локальный хранит права доступа, удаленный — пароли.
- [ ] Репозиторий — это процесс сборки приложения. Локальный запускается на ПК, удаленный — в CI-конвейере.

* **Краткий ответ**: Репозиторий — структура данных, содержащая файлы проекта и историю их изменений. Локальный находится на ПК разработчика, удаленный — на сервере хостинга.
* **Пояснение**: Разработчик коммитит изменения в свой локальный репозиторий, а затем отправляет (push) их в удаленный репозиторий, чтобы другие члены команды могли забрать (pull) их себе.

### 8. Как Git хранит данные и с помощью какого механизма обеспечивается целостность данных в репозитории? #git
- [x] Хранит данные в виде слепков (snapshots) объектов (blobs, trees, commits), адресуемых по SHA-1 хэшу содержимого, что делает невозможным незаметное изменение файлов.
- [ ] Хранит данные в реляционной СУБД SQLite, целостность проверяется по внешним ключам (Foreign Keys).
- [ ] Хранит данные в виде архивов ZIP, целостность отслеживается через контрольные суммы CRC32 для каждого архива.
- [ ] Хранит данные в виде дельта-патчей в XML-файлах, целостность проверяется цифровой подписью сервера.

* **Краткий ответ**: Git хранит снимки файлов как объекты в базе данных key-value. Целостность отслеживается с помощью криптографического хэширования SHA-1.
* **Пояснение**: Каждый объект в Git (содержимое файла — blob, каталог — tree, коммит — commit) получает уникальный 40-символьный хэш SHA-1, вычисляемый от его содержимого. Если файл изменится хотя бы на один бит, изменится его хэш.

### 9. Какие основные состояния могут принимать файлы в системе контроля версий Git? #git
- [x] Отслеживаемые (Tracked), которые делятся на: неизмененные (Unmodified), измененные (Modified), подготовленные (Staged); и неотслеживаемые (Untracked).
- [ ] Скомпилированные (Compiled), исходные (Source), временные (Temporary) и удаленные (Deleted).
- [ ] Локальные (Local), удаленные (Remote), заблокированные (Locked) и свободные (Free).
- [ ] Закоммиченные (Committed), отправленные (Pushed), обновленные (Pulled) и конфликтующие (Conflicted).

* **Краткий ответ**: Файлы бывают Untracked (неотслеживаемые) и Tracked (отслеживаемые). Tracked делятся на Unmodified (неизмененные), Modified (измененные) и Staged (подготовленные к коммиту).
* **Пояснение**: Untracked — новые файлы, о которых Git еще не знает. Modified — измененные файлы. Staged — измененные файлы, отмеченные для включения в следующий коммит (после `git add`). Unmodified — файлы, совпадающие с последним коммитом.

### 10. Какова структура служебного каталога `.git` в репозитории Git и за что отвечают основные папки? #git
- [x] `objects` хранит все содержимое базы данных; `refs` хранит указатели на коммиты (ветки и теги); `HEAD` указывает на текущую активную ветку; `index` — это область подготовленных файлов (staging area).
- [ ] `src` хранит исходный код; `bin` хранит бинарные файлы сборки; `config` содержит настройки компиляции.
- [ ] `commits` содержит лог в формате JSON; `branches` хранит исходные файлы веток; `remote` хранит URL серверов.
- [ ] `db` хранит SQL-файлы истории; `cache` хранит кэш коммитов; `auth` содержит SSH-ключи.

* **Краткий ответ**: Папка `.git` содержит: `objects` (база объектов), `refs` (ссылки на ветки/теги), `HEAD` (ссылка на текущую ветку/коммит), `index` (индекс/staging area), `config` (настройки репозитория).
* **Пояснение**: При инициализации репозитория (`git init`) создается скрытая папка `.git`. Именно в ней Git хранит всю историю, метаданные и настройки, тогда как рабочая папка содержит текущую рабочую копию файлов.

### 11. Как технически реализовано ветвление в Git, что представляет собой ветка master (main) и указатель HEAD? #git
- [x] Ветка — это простой перемещаемый указатель на конкретный коммит. `master` — ветка по умолчанию. `HEAD` — указатель на текущую активную ветку или коммит, с которым работает пользователь.
- [ ] Ветка — это полная копия всех файлов проекта в отдельной папке. `master` — главная папка на сервере. `HEAD` — первый коммит репозитория.
- [ ] Ветка — это строка в таблице базы данных коммитов. `master` — заблокированная ветка. `HEAD` — указатель на последний слияние.
- [ ] Ветка — это массив изменений в оперативной памяти. `master` — ветка, содержащая только релизы. `HEAD` — указатель на удаленный сервер.

* **Краткий ответ**: Ветка в Git — это просто указатель (текстовый файл с SHA-1 хэшем) на коммит. `master`/`main` — стандартное имя основной ветки. `HEAD` — ссылка на текущий снимок/ветку в рабочей директории.
* **Пояснение**: Поскольку ветка — это всего лишь указатель на коммит, создание ветки (`git branch name`) происходит мгновенно и не расходует память. При создании коммита указатель текущей ветки автоматически сдвигается вперед.

### 12. В чем разница между долгоживущими (long-running) и тематическими (feature) ветками, и что такое удаленные ветки? #git
- [x] Долгоживущие ветки (например, `main`, `develop`) отражают стадии жизненного цикла ПО; тематические создаются под конкретные фичи и удаляются после слияния; удаленные ветки (`remote-tracking`) отражают состояние веток на сервере.
- [ ] Долгоживущие ветки хранят старый код, тематические — новый, а удаленные ветки находятся на компьютерах других разработчиков.
- [ ] Долгоживущие ветки создаются только администратором, тематические — разработчиками, а удаленные ветки — роботами CI.
- [ ] Долгоживущие ветки нельзя удалять физически, тематические хранятся в кэше, а удаленные ветки удалены из истории Git.

* **Краткий ответ**: Долгоживущие ветки существуют на протяжении всего проекта (`main`, `dev`). Тематические (feature/bugfix) создаются для одной задачи и удаляются. Удаленные ветки (например, `origin/main`) — это локальные закладки, показывающие состояние веток на удаленном сервере на момент последней связи с ним.
* **Пояснение**: Использование тематических веток позволяет разработчикам вести параллельную работу над разными фичами в изоляции, не засоряя основную ветку нестабильным кодом. Удаленные ветки обновляются при командах `git fetch` или `git pull`.

### 13. Как устроен процесс слияния веток с помощью команды `git merge`? #git
- [x] Объединяет ветки путем поиска общего предка и создания специального коммита слияния (merge commit) с двумя родителями, либо делает перемотку (fast-forward), если история не расходилась.
- [ ] Переносит коммиты одной ветки и вставляет их поверх другой, переписывая историю коммитов.
- [ ] Копирует файлы из одной ветки в другую, удаляя историю исходной ветки.
- [ ] Отправляет все локальные ветки на удаленный сервер без создания локальных записей.

* **Краткий ответ**: `git merge` соединяет две ветки. Если ветка-источник является прямым предком целевой, происходит `fast-forward` (простая перемотка указателя). Если история разошлась, Git находит общего предка, делает трехстороннее слияние и создает коммит слияния (merge commit).
* **Пояснение**: Во время трехстороннего слияния Git пытается автоматически объединить изменения. Если в одной и той же строке одного и того же файла сделаны разные изменения, возникает конфликт слияния, который разработчик должен разрешить вручную.

### 14. Как работает перенос коммитов с помощью команды `git rebase` и чем он отличается от `git merge`? #git
- [x] Берет коммиты из текущей ветки и временно их сохраняет, переносит базу ветки на последний коммит целевой ветки, а затем применяет сохраненные коммиты по очереди, создавая линейную историю без коммитов слияния.
- [ ] Объединяет две ветки в одну на удаленном сервере, сохраняя все коммиты слияния в неизменном виде.
- [ ] Удаляет все локальные коммиты и заменяет их коммитами из удаленного репозитория.
- [ ] Создает новый коммит, объединяющий все файлы в один большой архив, и переименовывает ветку.

* **Краткий ответ**: `git rebase` переносит основание текущей ветки на другой коммит. Она переписывает историю, временно откладывая коммиты ветки, обновляя её базу до целевой ветки и накатывая отложенные коммиты поверх. Это дает красивую линейную историю.
* **Пояснение**: Главное отличие от `git merge` — `rebase` переписывает хэши коммитов (создает их заново с новыми родителями), поэтому золотое правило Git гласит: никогда не делать `rebase` для веток, которые уже отправлены в удаленный общий репозиторий (публичные ветки).

## 2. Сборка ПО и автоматизация (Make, Gradle)

### 15. В чем разница между компилируемыми и интерпретируемыми языками программирования, и что такое компиляция интерпретирующего типа? #build
- [x] Компилируемые языки переводят весь исходный код в машинный код до запуска программы (C, C++), а интерпретируемые выполняют код строка за строкой с помощью интерпретатора (Python, JS); компиляция интерпретирующего типа (JIT) компилирует код в машинный прямо во время исполнения.
- [ ] Компилируемые языки работают только на Windows, интерпретируемые — на Linux; компиляция интерпретирующего типа компилирует код на удаленном сервере.
- [ ] Компилируемые языки не имеют типов данных, а интерпретируемые строго типизированы; компиляция интерпретирующего типа преобразует машинный код обратно в исходный.
- [ ] Компилируемые языки преобразуют код в байт-код перед компиляцией в машинный код; интерпретирующие языки требуют сборки проекта.

* **Краткий ответ**: Компилируемые языки транслируются в машинный код до запуска. Интерпретируемые выполняются на лету интерпретатором. JIT (Just-In-Time) компиляция переводит части интерпретируемого кода в машинные инструкции прямо в процессе выполнения для ускорения.
* **Пояснение**: С++ — типичный пример компилируемого языка. JavaScript и Python — интерпретируемые. Современные движки (например, V8 для JS) используют JIT-компиляцию, сочетая скорость компилируемых языков и гибкость интерпретируемых.

### 25. В чем суть каскадной модели («Водопад» / Waterfall), каковы ее плюсы, минусы и область применения? #sdlc
- [x] Последовательное прохождение этапов без возврата назад; плюсы: простота управления, четкие сроки и бюджет; минусы: высокая стоимость ошибок на поздних этапах, клиент видит результат только в конце; применяется в проектах с жесткими, неизменными требованиями (космос, медицина).

- [ ] Постоянное создание прототипов; плюсы: высокая вовлеченность клиента; минусы: плохой код; применяется в веб-дизайне.

* **Краткий ответ**: Waterfall — линейная модель разработки, где каждый этап начинается только после завершения предыдущего. Плюсы: планируемость. Минусы: негибкость. Применение: проекты с фиксированными требованиями.
* **Пояснение**: В Waterfall требования собираются один раз в начале. Если на этапе тестирования выяснится ошибка в анализе требований, возврат на первый этап обойдется очень дорого.

* **Пояснение**: Спиральная модель идеально подходит для проектов, где провал из-за неверных технологических решений или изменения рынка может обернуться большими финансовыми потерями.

### 30. Какую роль играет документирование знаний в разработке ПО и чем отличаются системы Confluence и MediaWiki? #docs
- [x] Позволяет сохранить экспертизу команды, описать архитектуру и регламенты. Confluence — коммерческая корпоративная wiki-система от Atlassian с глубокой интеграцией с Jira и WYSIWYG-редактором; MediaWiki — бесплатный движок с открытым кодом, использующий вики-разметку.
- [ ] Используется для автоматической генерации JSDoc-комментариев. Confluence генерирует документацию на Java, а MediaWiki — на PHP.
- [ ] Позволяет вести версионирование бинарных файлов проекта. Confluence хранит файлы на AWS, а MediaWiki — на локальных серверах Apache.
- [ ] Предназначено для создания резервных копий исходного кода. Confluence работает через Git, а MediaWiki — через SVN.

* **Краткий ответ**: Документирование фиксирует архитектуру, требования и процессы проекта. Confluence — платное enterprise-решение с тесной интеграцией с Jira. MediaWiki — открытый бесплатный движок с акцентом на вики-текст.
* **Пояснение**: Хорошая документация сокращает время адаптации новых разработчиков (onboarding) и снижает зависимость проекта от ключевых сотрудников (bus factor).

## 4. Непрерывная интеграция и доставка (CI/CD, Окружение)

### 31. Что такое концепция CI/CD в современной разработке программного обеспечения? #ci-cd
- [x] Набор практик автоматизации процессов сборки, тестирования (Continuous Integration) и развертывания (Continuous Delivery/Deployment) кода в рабочую среду при каждом изменении в репозитории.
- [ ] Технология контейнеризации приложений на уровне ядра операционной системы для повышения производительности диска.
- [ ] Процесс непрерывного общения с клиентом и сбора требований (Continuous Interview / Client Discussion).
- [ ] Методология парного программирования, где один разработчик пишет код, а второй непрерывно его комментирует.

* **Краткий ответ**: CI/CD — автоматизированный конвейер (pipeline) сборки, тестирования и доставки изменений от разработчика на продакшен-сервер.
* **Пояснение**: Внедрение CI/CD позволяет сократить Time-to-Market (время выхода фичи на рынок), повысить качество ПО за счет частых автоматических тестов и минимизировать человеческий фактор при деплое.

### 32. Каковы ключевые принципы успешного построения процессов CI/CD? #ci-cd
- [x] Единый репозиторий кода, автоматизация всех этапов (сборка, тесты, деплой), быстрое получение обратной связи, воспроизводимость сборок в идентичных средах и постоянная готовность релиза к развертыванию.
- [ ] Проведение ручного тестирования перед каждым коммитом, хранение кода на локальных флешках и развертывание по FTP вручную.
- [ ] Использование только одного языка программирования во всем проекте и отказ от использования сторонних библиотек.
- [ ] Компиляция кода на сервере клиента в реальном времени и шифрование исходников перед слиянием веток.

* **Краткий ответ**: Принципы: автоматизация всего процесса, версионирование конфигураций (IaC), быстрая обратная связь, идентичность окружений разработки и продакшена, частые небольшие релизы.
* **Пояснение**: Окружения (dev, staging, prod) должны быть максимально похожи, чтобы избежать ошибок вида "на моем компьютере все работает". Автоматические тесты должны выполняться быстро для эффективной работы команды.

### 33. Как организуется управление требованиями в контексте CI/CD? #ci-cd
- [x] Требования декомпозируются на мелкие задачи (User Stories/Tasks) в трекере (Jira, GitLab Issues), связываются с ветками Git и автоматически валидируются приемочными тестами в CI-конвейере.
- [ ] Требования пишутся в виде одного большого документа MS Word, который разработчики читают перед запуском CI-сервера.
- [ ] База требований хранится непосредственно в базе данных приложения и обновляется SQL-скриптами на этапе сборки.
- [ ] Требования формируются автоматически искусственным интеллектом на основе логов ошибок в рабочей среде.

* **Краткий ответ**: Управление требованиями в CI/CD строится на их декомпозиции, связывании задач с ветками в Git и автоматической проверке функционала через тесты в пайплайне.
* **Пояснение**: Связь "Задача -> Ветка в Git -> Тесты в CI/CD -> Релиз" обеспечивает полную прослеживаемость (traceability). Мы всегда знаем, какое требование реализовано в текущей сборке.

### 34. Что такое конвейер развертывания (Deployment Pipeline) и из каких типовых этапов он состоит? #ci-cd
- [x] Автоматизированное прохождение кода от коммита до продакшена; этапы: сборка (Build) -> модульное тестирование (Unit Test) -> интеграционное тестирование (Integration Test) -> деплой на тестовую среду (Staging) -> деплой на бой (Production).
- [ ] Логистическая цепочка поставки серверов и оборудования в дата-центр компании.
- [ ] Последовательность ручных проверок кода тимлидом, системным аналитиком и конечным заказчиком.
- [ ] Схема движения задач по колонкам Kanban-доски: Backlog -> In Progress -> Code Review -> Done.

* **Краткий ответ**: Конвейер развертывания — это автоматизированный процесс доставки изменений. Основные этапы: Build (Сборка), Test (Тестирование), Deploy (Развертывание), Release (Выпуск в эксплуатацию).
* **Пояснение**: Каждый этап является фильтром: если код падает на модульных тестах, сборка прерывается и переход на этап интеграционных тестов или деплоя не происходит. Это гарантирует, что сломанный код не попадет к пользователям.

### 35. Что такое «среда окружения» (Environment) в разработке ПО и какие объекты в нее входят? #bash
- [x] Выделенная инфраструктура, где запускается приложение; объекты включают операционную систему, сетевую конфигурацию, переменные окружения, базы данных и установленные runtime-библиотеки.
- [ ] Рабочее место разработчика, включая стол, стул, монитор и освещение в офисе.
- [ ] Совокупность файлов исходного кода, папки `.git` и IDE разработчика.
- [ ] Процесс общения разработчиков в мессенджерах и на планерках (командная среда).

* **Краткий ответ**: Среда окружения — это комплекс аппаратных, программных и сетевых средств, в котором функционирует ПО. Объекты окружения: ОС, СУБД, веб-серверы, системные библиотеки, конфигурационные файлы и переменные среды.
* **Пояснение**: Типичный проект имеет несколько сред: Development (локальный ПК), Testing/Staging (среда тестирования), Production (боевой сервер).

### 36. Назовите основные переменные среды окружения в ОС Linux и их назначение. #bash
- [x] `PATH` (пути поиска исполняемых файлов), `HOME` (путь к домашней директории), `USER` (имя текущего пользователя), `LANG` (языковые настройки), `SHELL` (путь к интерпретатору командной строки).
- [ ] `GET`, `POST`, `PUT`, `DELETE` (HTTP-методы для обращения к системным API).
- [ ] `IP`, `PORT`, `DNS`, `GATEWAY` (настройки сетевых интерфейсов ядра Linux).
- [ ] `CPU`, `RAM`, `DISK`, `GPU` (характеристики аппаратного обеспечения сервера).

* **Краткий ответ**: Переменные окружения Linux: `PATH` (где искать исполняемые файлы), `HOME` (домашний каталог), `USER` (имя пользователя), `SHELL` (текущая командная оболочка), `LANG` (локализация).
* **Пояснение**: Переменные среды — это пары ключ-значение, доступные процессам в ОС. Проверить их список в терминале Bash можно командой `env` или `printenv`, а вывести конкретную переменную — `echo $PATH`.

### 37. Что такое «ярус» (tier) или уровень системного окружения в многоуровневой архитектуре приложений? #bash
- [x] Логическое разделение компонентов системы по их функциональным обязанностям (например, трехзвенная архитектура: клиентский ярус представления, ярус бизнес-логики и ярус данных).
- [ ] Физическая полка в серверной стойке дата-центра, на которой располагаются блейд-серверы.
- [ ] Версия операционной системы Linux (ярусы обозначают степень стабильности дистрибутива).
- [ ] Очередь задач в CI-сервере Jenkins, разделенная по приоритетам выполнения сборок.

* **Краткий ответ**: Системный ярус (Tier) — логически и физически обособленный слой архитектуры приложения (например, Presentation Tier — фронтенд, Application Tier — бэкенд, Data Tier — база данных).
* **Пояснение**: Разделение по ярусам позволяет масштабировать каждый слой независимо. Например, при высокой нагрузке на чтение базы данных можно добавить реплики на ярусе данных, не меняя код на бэкенде.

### 38. Какие основные компоненты входят в архитектуру современных сред окружения для веб-приложений? #bash
- [x] Балансировщик нагрузки (Load Balancer), веб-сервер (Nginx/Apache), сервер приложений (Node.js/Gunicorn), база данных (PostgreSQL/MongoDB) и система кэширования (Redis/Memcached).

- [ ] Клавиатура, мышь, монитор, сетевая карта и материнская плата сервера.

* **Краткий ответ**: Основные компоненты среды: балансировщики нагрузки, веб-серверы (обратные прокси), серверы приложений (выполняющие код), базы данных (хранилища) и сервисы кэширования/очередей.
* **Пояснение**: Архитектура обеспечивает отказоустойчивость и масштабируемость. Балансировщик (например, Nginx) распределяет запросы пользователей по нескольким серверам приложений, снижая нагрузку на каждый из них.

## 5. Виртуализация, контейнеризация и Docker

### 39. Что такое виртуализация и какие основные задачи она решает? #virtualization
- [x] Технология предоставления изолированных вычислительных ресурсов на базе одного физического компьютера; задачи: консолидация серверов, изоляция приложений, эффективное использование железа.
- [ ] Процесс создания 3D-моделей физических объектов для компьютерных игр и тренажеров.
- [ ] Разработка программного интерфейса (API) для удаленного управления робототехникой.
- [ ] Оптимизация исходного кода путем удаления неиспользуемых функций и сжатия бинарных файлов.

* **Краткий ответ**: Виртуализация — создание виртуального представления физических ресурсов с помощью специального ПО (гипервизора). Решает задачи оптимизации утилизации железа, изоляции и быстрого развертывания сред.
* **Пояснение**: Виртуализация позволяет запустить на одном мощном сервере десятки виртуальных машин с разными ОС, загружая физический процессор и память на максимум, вместо простоя оборудования.

### 40. Каковы главные преимущества внедрения технологий виртуализации в ИТ-инфраструктуру компании? #virtualization
- [x] Снижение затрат на покупку физического оборудования, быстрое создание новых серверов, высокая отказоустойчивость (миграция ВМ на другой хост) и упрощение резервного копирования.
- [ ] Увеличение физической скорости передачи данных по медным кабелям сети Ethernet.
- [ ] Полное отсутствие необходимости в написании тестов и администрировании серверов.
- [ ] Автоматическое ускорение работы любого однопоточного программного кода в несколько раз.

* **Краткий ответ**: Преимущества: экономия на оборудовании/электричестве, гибкость (создание ВМ за минуты), изоляция безопасности, удобство резервного копирования (снимки/snapshots) и легкое масштабирование.
* **Пояснение**: Благодаря виртуализации компания может арендовать ресурсы в облаке (IaaS) вместо покупки дорогостоящих серверов в собственный дата-центр.

### 41. С какими недостатками и ограничениями связано использование классической виртуализации? #virtualization
- [x] Потери производительности на оверхед (эмуляцию железа гипервизором), высокий расход оперативной памяти и диска (каждая ВМ требует полноценную гостевую ОС) и сложность администрирования.
- [ ] Полное отсутствие защиты от сетевых атак из внешней сети Интернет.
- [ ] Несовместимость с любыми базами данных и языками программирования, кроме C++.
- [ ] Ограничение на максимальное количество файлов в проекте (не более 1000 файлов).

* **Краткий ответ**: Недостатки: накладные расходы на работу гипервизора (оверхед), большой размер дисковых образов и потребление ОЗУ из-за необходимости запускать гостевую ОС для каждой машины.
* **Пояснение**: Для каждой виртуальной машины гипервизор выделяет фиксированные ресурсы и запускает отдельное ядро ОС. Это создает избыточность, которую решает более легковесная контейнеризация.

### 42. Что такое виртуализация на уровне операционной системы (контейнеризация), каковы ее задачи и применение? #virtualization
- [x] Изоляция процессов в рамках одной родительской ОС с использованием общего ядра (например, Docker); задачи: изоляция окружений приложений с минимальными накладными расходами; применяется для микросервисов и CI/CD.
- [ ] Установка нескольких физических жестких дисков на один сервер с целью ускорения файловых операций ОС.
- [ ] Эмуляция работы процессоров ARM на компьютерах с архитектурой x86-64 через программу QEMU.
- [ ] Процесс обновления операционной системы Linux до более свежей версии без перезагрузки компьютера.

* **Краткий ответ**: Виртуализация на уровне ОС (контейнеризация) запускает изолированные контейнеры, использующие ядро хост-системы. Задачи: минимизация оверхеда, скорость запуска. Применяется для деплоя приложений.
* **Пояснение**: В отличие от аппаратной виртуализации (где для каждой ВМ запускается своя гостевая ОС), контейнеризация использует общие системные вызовы ядра хоста, разделяя ресурсы с помощью механизмов Linux (`namespaces`, `cgroups`).

### 43. В чем суть виртуализации серверов приложений, ее задачи и область применения? #virtualization
- [x] Создание абстрактного слоя между кодом приложения и физическим сервером (виртуальные машины Java JVM, .NET CLR); задачи: кроссплатформенность и безопасность выполнения; применяется для запуска корпоративного ПО.
- [ ] Настройка балансировщика Nginx на перенаправление трафика между виртуальными хостами (Virtual Hosts).
- [ ] Запуск веб-серверов в изолированных виртуальных локальных сетях (VLAN) без доступа к сети Интернет.
- [ ] Замена физической оперативной памяти сервера файлом подкачки (swap) на жестком диске.

* **Краткий ответ**: Виртуализация серверов приложений (на примере JVM, BEAM, CLR) транслирует байт-код в машинные инструкции во время работы. Задачи: кроссплатформенность и управление памятью.
* **Пояснение**: Среда выполнения (Runtime) берет на себя управление памятью (Garbage Collection), безопасность (песочница) и платформозависимые вызовы, изолируя приложение от деталей конкретной ОС.

### 44. Что такое виртуализация сети (Network Virtualization), ее задачи и где она применяется? #virtualization
- [x] Объединение физических сетевых ресурсов в единую логическую сеть или разделение одной сети на независимые виртуальные (VLAN, VPN, SDN); задачи: безопасность, гибкость настроек; применяется в облачных дата-центрах.
- [ ] Эмуляция работы Wi-Fi роутеров на домашних персональных компьютерах через USB-модемы.
- [ ] Процесс шифрования трафика браузера с помощью протокола HTTPS / TLS.
- [ ] Использование виртуальных IP-адресов, которые генерируются случайным образом при каждом запросе.

* **Краткий ответ**: Виртуализация сети отделяет логическое представление сети от физического оборудования (коммутаторов, кабелей). Задачи: создание изолированных подсетей, динамическое управление трафиком. Применяется в облаках и SDN.
* **Пояснение**: Программное управление сетями (SDN) позволяет создавать виртуальные коммутаторы, маршрутизаторы и брандмауэры программным путем за доли секунды, что критично для облачных провайдеров.

### 45. Что такое аппаратная виртуализация (Hardware Virtualization), ее задачи и применение? #virtualization
- [x] Создание виртуальных машин с помощью гипервизора, полностью эмулирующего физическое железо, на которых запускаются гостевые ОС; задачи: запуск разных ОС на одном ПК; применяется в облачных хостингах.
- [ ] Создание виртуального жесткого диска в оперативной памяти компьютера (RAM-диск) для ускорения чтения.
- [ ] Использование технологий 3D-печати для создания копий компьютерных комплектующих.
- [ ] Программная эмуляция кнопок клавиатуры и мыши для автоматизации действий в играх.

* **Краткий ответ**: Аппаратная виртуализация эмулирует аппаратную платформу с помощью гипервизора (ESXi, Hyper-V, VirtualBox). Задачи: запуск разнородных ОС в изоляции на одном физическом сервере.
* **Пояснение**: Гостевая ОС работает так, словно имеет доступ к реальному физическому железу, благодаря тому, что гипервизор перехватывает ее инструкции и передает их физическому процессору.

### 46. В чем заключается виртуализация систем хранения данных (Storage Virtualization), ее задачи и применение? #virtualization
- [x] Объединение физических накопителей из разных устройств в единый пул памяти, управляемый централизованно (SAN, NAS, SDS); задачи: отказоустойчивость, абстракция от дисков; применяется в дата-центрах.
- [ ] Запись файлов на CD и DVD-диски вместо использования жестких дисков и SSD.
- [ ] Архивация файлов в форматы ZIP и RAR для экономии свободного пространства на диске компьютера.
- [ ] Использование облачных хранилищ вроде Яндекс.Диск для хранения личных фотографий.

* **Краткий ответ**: Виртуализация СХД объединяет физические диски в логический пул, скрывая сложность аппаратного уровня. Задачи: непрерывная доступность данных, простота расширения емкости, RAID-массивы.
* **Пояснение**: Программный слой распределяет данные, делает зеркалирование (снижая риск потери при выходе диска из строя) и предоставляет единый логический диск для серверов.

### 47. Что такое виртуализация сервисов (Service Virtualization), ее задачи и применение? #virtualization
- [x] Имитация (заглушка) поведения внешних зависимых API и сервисов (баз данных, сторонних систем), которые недоступны или нестабильны во время тестирования приложения; задачи: непрерывное тестирование; применяется в QA.
- [ ] Перенос веб-сервисов из локального офиса компании на облачные серверы Amazon AWS.
- [ ] Преобразование монолитного приложения в микросервисы с использованием Docker Compose.
- [ ] Настройка автозапуска системных служб в Linux при старте операционной системы.

* **Краткий ответ**: Виртуализация сервисов — создание заглушек (mocks/stubs), симулирующих ответы реальных систем (например, платежного шлюза) для проведения тестирования.
* **Пояснение**: Если реальный платежный шлюз берет комиссию за транзакцию или нестабилен в тестовой среде, разработчики создают виртуальный сервис, который имитирует успешную оплату или ошибку.

### 48. В чем состоят ключевые особенности контейнерной виртуализации, ее задачи и применение? #docker
- [x] Метод виртуализации, при котором ядро ОС изолирует несколько независимых пространств пользователя (контейнеров) без эмуляции железа; задачи: легковесность, скорость развертывания; применяется в CI/CD и микросервисах.
- [ ] Упаковка исходного кода в зашифрованные zip-архивы для отправки клиентам по электронной почте.
- [ ] Использование физических морских контейнеров для транспортировки серверов дата-центра.
- [ ] Создание виртуальных машин с установкой Windows поверх Linux через эмулятор Wine.

* **Краткий ответ**: Контейнерная виртуализация изолирует приложения на уровне операционной системы, делясь ядром хоста. Задачи: быстрый запуск (миллисекунды), минимальный оверхед, переносимость.
* **Пояснение**: Контейнеры запускаются как обычные изолированные процессы хост-системы. Они не требуют оверхеда на работу гипервизора и запуск ядра гостевой ОС.

### 49. Почему контейнеризация стала стандартом де-факто в современных вычислительных системах и облачных технологиях? #docker
- [x] Обеспечивает концепцию "упаковано один раз — запускается везде", гарантирует идентичность сред, экономит ресурсы серверов и отлично масштабируется системами оркестрации (Kubernetes).
- [ ] Позволяет полностью отказаться от использования языков программирования и писать программы визуально.
- [ ] Гарантирует 100% защиту от любых физических повреждений серверов и перебоев с питанием.
- [ ] Является единственным способом запуска приложений на мобильных устройствах Android и iOS.

* **Краткий ответ**: Контейнеризация решает проблему несоответствия окружений (устраняет ошибку "на моем ПК работает, а на сервере нет"), снижает расходы на инфраструктуру за счет плотного размещения приложений.
* **Пояснение**: Контейнер упаковывает приложение вместе со всеми зависимостями. Такой образ можно запустить на ноутбуке разработчика, на стейджинге и в облаке без изменения кода.

### 50. Что представляет собой контейнер приложения (Application Container)? #docker
- [x] Изолированная среда выполнения, содержащая только само приложение и его непосредственные системные зависимости, разделяющая ядро операционной системы с другими контейнерами.
- [ ] Физический корпус системного блока, защищенный от пыли, влаги и перепадов температур.
- [ ] Папка `src` в корне проекта, куда разработчики складывают файлы исходного кода.
- [ ] База данных, в которой хранятся метаданные о конфигурации всех виртуальных машин.

* **Краткий ответ**: Контейнер приложения — это запущенный экземпляр образа, представляющий собой изолированную группу процессов, имеющих собственную файловую систему, сеть и лимиты ресурсов.
* **Пояснение**: В отличие от системных контейнеров, контейнеры приложений обычно запускают один конкретный процесс (например, веб-сервер Nginx или бэкенд Node.js) и завершают работу при его остановке.

### 51. Что такое оркестрация контейнеров и какие основные задачи она выполняет? #docker
- [x] Автоматизированное управление жизненным циклом множества контейнеров в кластере; задачи: балансировка нагрузки, масштабирование, самовосстановление (перезапуск упавших) и обновление без простоя.
- [ ] Процесс написания фоновой музыки для демонстрационных видеороликов программных продуктов.
- [ ] Ручной запуск докер-контейнеров разработчиками на своих локальных компьютерах через консоль.
- [ ] Слияние нескольких репозиториев Git в один большой монорепозиторий перед сборкой.

* **Краткий ответ**: Оркестрация — автоматизация развертывания, масштабирования, сетевой связности и мониторинга контейнеров на множестве серверов. Главный инструмент — Kubernetes.
* **Пояснение**: Когда у вас сотни контейнеров на десятках серверов, оркестратор следит за тем, чтобы работало нужное число копий приложения, автоматически переносит контейнеры с вышедших из строя серверов и направляет на них трафик.

### 52. Что такое образ контейнера (Container Image) и какие два основных состояния выделяют у контейнера? #docker
- [x] Образ — это статический, доступный только для чтения шаблон файловой системы приложения; два состояния контейнера: пассивное (образ на диске) и активное (запущенный изолированный процесс в памяти).
- [ ] Образ — это графический скриншот интерфейса запущенной программы; два состояния: свернутое и развернутое.
- [ ] Образ — это резервная копия всей операционной системы хоста; два состояния: архивированное и распакованное.
- [ ] Образ — это SQL-схема базы данных; два состояния: транзакционное и нетранзакционное.

* **Краткий ответ**: Образ (Image) — неизменяемый шаблон с кодом и библиотеками. Контейнер (Container) — это запущенный процесс на основе образа. Состояния: пассивное (образ на диске) и активное (запущенный контейнер).
* **Пояснение**: Образ состоит из слоев (layers) и доступен только для чтения. При запуске контейнера поверх слоев образа создается тонкий перезаписываемый слой (writable layer), куда контейнер пишет временные файлы.

### 53. Что такое контейнерный движок (Container Engine) и какие задачи он выполняет? #docker
- [x] Программное обеспечение, управляющее созданием, запуском и остановкой контейнеров на хосте (например, Docker Engine, containerd, CRI-O); задачи: скачивание образов, изоляция ресурсов, настройка сети.
- [ ] Процессорная плата, ускоряющая компиляцию программного обеспечения на сервере сборки.
- [ ] База данных для хранения логов контейнеров и отправки уведомлений администраторам.
- [ ] Инструмент автоматического перевода кода приложения с одного языка на другой.

* **Краткий ответ**: Контейнерный движок — системная служба, которая принимает команды от пользователя/оркестратора, скачивает образы, настраивает сетевые интерфейсы, изолирует диски и запускает процессы контейнеров.
* **Пояснение**: Движок взаимодействует с ядром ОС Linux (через системную библиотеку runc), настраивая пространства имен (namespaces) и контрольные группы (cgroups), чтобы изолировать запускаемые процессы.

### 54. Что такое контейнерный хост (Container Host) и каковы его задачи? #docker
- [x] Физический компьютер или виртуальная машина, на которой установлена операционная система и контейнерный движок; задачи: предоставление аппаратных ресурсов (CPU, RAM, диск, сеть) для работы контейнеров.
- [ ] Веб-сайт в Интернете, на котором разработчики ведут обсуждение архитектуры проектов.
- [ ] Удаленный сервер реестра, где хранятся скомпилированные образы докер-контейнеров.
- [ ] Главный поток выполнения программы, управляющий запуском дочерних потоков.

* **Краткий ответ**: Контейнерный хост — физический или виртуальный сервер с ОС, на котором запущен контейнерный движок и выполняются сами контейнеры.
* **Пояснение**: Хост обеспечивает выделение системных ресурсов (ядер процессора, оперативной памяти) под нужды контейнеров через cgroups и предоставляет сетевые интерфейсы для связи.

### 55. Что такое сервер реестра контейнеров (Container Registry) и какие задачи он решает? #docker
- [x] Хранилище образов контейнеров (например, Docker Hub, GitLab Registry); задачи: централизованное хранение образов, управление версиями (тегами), контроль доступа.
- [ ] Сервер базы данных, в котором хранятся системные настройки и реестр Windows.
- [ ] Компьютер разработчика, на котором ведется история коммитов в Git-репозиторий.
- [ ] Балансировщик нагрузки, распределяющий запросы пользователей по сетевым адресам.

* **Краткий ответ**: Реестр контейнеров — серверное приложение или облачный сервис для публикации, хранения, каталогизации и скачивания образов контейнеров (например, Docker Hub).
* **Пояснение**: Разработчик или CI/CD сервер собирает образ и отправляет его в реестр командой `docker push`. При развертывании на хост-сервере движок скачивает этот образ из реестра командой `docker pull`.

### 56. Что такое Docker и какова роль его компонентов Docker Daemon и Docker Client? #docker
- [x] Платформа для разработки, доставки и запуска контейнеров; Docker Client — утилита командной строки (`docker`), отправляющая команды; Docker Daemon (`dockerd`) — фоновая служба на хосте, выполняющая эти команды.
- [ ] СУБД для хранения неструктурированных данных; Daemon управляет кэшем, а Client выполняет SQL-запросы.
- [ ] Система контроля версий; Daemon хранит коммиты на сервере, а Client делает коммиты локально.
- [ ] Инструмент автоматизации сборки Java; Daemon компилирует классы, а Client запускает юнит-тесты.

* **Краткий ответ**: Docker — платформа для контейнеризации. Docker Client — CLI-клиент, с которым общается пользователь. Docker Daemon — серверный процесс, управляющий объектами Docker (образы, контейнеры, сети, тома).
* **Пояснение**: Docker использует клиент-серверную архитектуру. Клиент может находиться на вашем ноутбуке, а демон — на удаленном сервере. Они общаются по REST API через UNIX-сокет или сетевой интерфейс.

### 57. Какова взаимосвязь между Docker-образом (Image) и Docker-реестром (Registry)? #docker
- [x] Docker-образ — это неизменяемый шаблон сборки контейнера; Docker-реестр — это сервис хранения и распространения этих образов (например, Docker Hub).
- [ ] Docker-образ — это скомпилированный бинарный файл, а Docker-реестр — база данных для хранения логов работы этого файла.
- [ ] Docker-образ — это копия оперативной памяти контейнера, а Docker-реестр — жесткий диск, куда этот образ периодически сбрасывается.
- [ ] Docker-образ — это виртуальная машина в облаке, а Docker-реестр — веб-интерфейс для ее настройки.

* **Краткий ответ**: Образ — это упакованный шаблон приложения, а реестр — библиотека/облако, где эти образы хранятся для общего доступа.
* **Пояснение**: Чтобы запустить контейнер, Docker проверяет, есть ли нужный образ локально. Если его нет, он обращается к указанному реестру (по умолчанию Docker Hub), скачивает образ и затем запускает контейнер.

### 58. Что такое пространства имен (Namespaces) в Linux/Docker и какие основные типы используются для изоляции контейнеров? #docker
- [x] Технология ядра Linux, обеспечивающая изоляцию системных ресурсов; основные: `pid` (процессы), `net` (сеть), `mnt` (файловая система), `ipc` (межпроцессное взаимодействие), `uts` (имя хоста), `user` (UID/GID).
- [ ] Директивы в коде C++, предотвращающие конфликты имен классов в разных библиотеках.
- [ ] Записи в DNS-сервере, связывающие IP-адрес хоста с доменными именами контейнеров.
- [ ] Папки внутри контейнера, разделяющие файлы по типам данных (логи, конфиги, код).

* **Краткий ответ**: Namespaces (пространства имен) изолируют ресурсы ОС так, чтобы контейнер видел только свои процессы, сети, диски. Основные: `pid`, `net`, `mnt`, `ipc`, `uts`, `user`.
* **Пояснение**: `pid` namespace гарантирует, что процессы внутри контейнера не видят процессы хоста. `net` namespace дает собственный стек сети. `mnt` namespace изолирует корневую файловую систему.

## 6. Управление конфигурациями, DevOps и микросервисы

### 59. Что такое конфигурационное управление (Configuration Management) в контексте ИТ-инфраструктуры, каковы его цели и задачи? #ansible
- [x] Процесс систематического поддержания параметров работы систем в заданном состоянии; цели: автоматизация настройки серверов, исключение дрейфа конфигураций (configuration drift) и обеспечение воспроизводимости инфраструктуры.
- [ ] Настройка конфигурационных параметров в файле `settings.gradle` для сборки проекта.
- [ ] Управление правами доступа пользователей к репозиториям Git на сервере GitHub Enterprise.
- [ ] Процесс распределения оперативной памяти между виртуальными машинами в гипервизоре.

* **Краткий ответ**: Конфигурационное управление автоматизирует настройку, установку ПО и администрирование множества серверов, приводя их к единому описанному состоянию.
* **Пояснение**: Вместо того чтобы вручную настраивать сервера по SSH, администратор описывает конфигурацию в виде кода (IaC) в инструментах конфигурационного управления (например, Ansible, Puppet), которые автоматически настраивают системы.

### 60. Какие основные процедуры и этапы входят в процесс конфигурационного управления ИТ-инфраструктурой? #ansible
- [x] Идентификация конфигураций, контроль изменений (через Git), учет статуса (мониторинг текущего состояния серверов) и аудит/проверка соответствия эталону.
- [ ] Компиляция кода, запуск тестов, упаковка в архивы и отправка артефактов в Docker Hub.
- [ ] Выбор методологии ЖЦПО, сбор требований, написание ТЗ и дизайн пользовательского интерфейса.
- [ ] Регистрация доменных имен, покупка SSL-сертификатов и настройка DNS-серверов.

* **Краткий ответ**: Процедуры включают: идентификацию элементов конфигурации, управление изменениями (IaC), учет состояния систем (сопоставление с желаемым) и аудит соответствия.
* **Пояснение**: Все изменения в инфраструктуру вносятся через изменение кода конфигурационных скриптов (плейбуков Ansible) в Git-репозитории. Инструменты конфигурационного управления накатывают эти изменения и проверяют соответствие серверов эталону.

### 61. Каковы основные положения и цели методологии DevOps? #devops
- [x] Культурное и техническое объединение процессов разработки (Dev) и эксплуатации (Ops) для повышения частоты релизов, улучшения качества ПО и автоматизации процессов поставки ценности пользователю.
- [ ] Назначение одного разработчика ответственным за ручную сборку и деплой кода по выходным дням.
- [ ] Отказ от использования облачных провайдеров и переход на собственное физическое серверное оборудование.
- [ ] Использование языков программирования Bash и Python для написания абсолютно всего кода проекта.

* **Краткий ответ**: DevOps — это культура совместной работы разработчиков (Development) и системных администраторов (Operations) с фокусом на автоматизацию процессов, непрерывную интеграцию (CI/CD) и быструю реакцию на обратную связь.
* **Пояснение**: DevOps устраняет "стену непонимания" между разработчиками (которые хотят чаще выпускать фичи) и сисадминами (которые хотят стабильности) через автоматические тесты, мониторинг и общую ответственность за продукт.

### 62. Что такое непрерывная доставка (Continuous Delivery) и какие основные элементы она в себя включает? #devops
- [x] Практика, при которой код автоматически собирается, тестируется и подготавливается к выпуску, так что любое изменение может быть развернуто в продакшен в один клик вручную.
- [ ] Процесс круглосуточной доставки пиццы и кофе в офис ИТ-компании для разработчиков.
- [ ] Постоянное скачивание обновлений операционной системы хоста на локальный диск сервера.
- [ ] Автоматическая публикация исходного кода всех проектов компании в открытый доступ на GitHub.

* **Краткий ответ**: Continuous Delivery (CD) гарантирует, что код в основной ветке Git всегда находится в работоспособном и готовом к релизу состоянии. Развертывание на продакшен выполняется по нажатию кнопки человеком.
* **Пояснение**: Ключевые элементы CD: автоматическое тестирование, сборка артефактов (Docker-образов), развертывание на тестовые среды (staging). Отличие от Continuous Deployment заключается в том, что в Deployment выкатка на продакшен происходит полностью автоматически без ручного подтверждения.

### 63. Что такое непрерывная интеграция (Continuous Integration) и каковы ее ключевые элементы? #devops
- [x] Практика регулярного слияния кода всех разработчиков в общую ветку (несколько раз в день) с последующим автоматическим запуском сборки и тестов для раннего выявления ошибок.
- [ ] Процесс интеграции базы данных проекта с внешними государственными реестрами через защищенные каналы связи.
- [ ] Ежедневное проведение совещаний с участием всех отделов компании для обсуждения планов разработки.
- [ ] Объединение всех сторонних библиотек в один большой zip-архив для ускорения работы IDE.

* **Краткий ответ**: Continuous Integration (CI) — практика частого объединения изменений в основной ветке репозитория и их мгновенной автоматической проверки. Элементы: общий репозиторий (Git), автоматическая сборка, автотесты.
* **Пояснение**: CI позволяет избежать "интеграционного ада" в конце разработки, когда ветки расходились месяцами, и их слияние превращалось в кошмар с сотнями конфликтов. При CI ошибки находятся и исправляются сразу.

### 64. Что такое микросервисная архитектура и каковы ключевые принципы ее проектирования? #microservices
- [x] Подход к созданию ПО в виде набора мелких, независимых, взаимодействующих по сети сервисов; принципы: единая ответственность (Single Responsibility), независимое развертывание, собственная база данных для каждого сервиса и децентрализованное управление.
- [ ] Разработка приложения в виде единого монолитного блока с использованием микропроцессоров малой мощности.
- [ ] Использование миниатюрных функций (не более 10 строк кода) во всем исходном коде приложения.
- [ ] Создание веб-страниц, оптимизированных для отображения на экранах смарт-часов и фитнес-браслетов.

* **Краткий ответ**: Микросервисы — архитектурный стиль, разденяющий приложение на небольшие слабосвязанные службы. Принципы: децентрализация данных (database-per-service), API-ориентированное взаимодействие (REST, gRPC, очереди), независимый деплой.
* **Пояснение**: Каждый микросервис решает одну бизнес-задачу (например, сервис оплаты, сервис доставки). Они разрабатываются независимыми командами, могут быть написаны на разных языках и общаются друг с другом по сети.

### 65. Каковы основные преимущества и недостатки микросервисной архитектуры по сравнению с монолитной? #microservices
- [x] Преимущества: независимое масштабирование, гибкость технологий, простота развертывания фич; недостатки: сложность распределенной системы, накладные расходы на сетевые вызовы, трудности обеспечения согласованности данных (eventual consistency).
- [ ] Преимущества: отсутствие необходимости в базах данных; недостатки: программа работает только при наличии сети Интернет.
- [ ] Преимущества: автоматическое отсутствие ошибок в коде; недостатки: высокая стоимость покупки компиляторов.
- [ ] Преимущества: очень простая отладка на одном компьютере; недостатки: невозможность совместной работы разработчиков.

* **Краткий ответ**: Плюсы: масштабируемость отдельных частей, изоляция сбоев (падение одного сервиса не ломает всю систему), свобода выбора стека технологий. Минусы: сетевой оверхед, сложность отладки и тестирования, распределенные транзакции.
* **Пояснение**: Хотя микросервисы дают гибкость крупным командам, они приносят серьезную сложность в инфраструктуру (требуются оркестраторы типа Kubernetes, распределенное логирование Jaeger/Elasticsearch, брокеры сообщений RabbitMQ/Kafka).
